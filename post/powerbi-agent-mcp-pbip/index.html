AI Agent 操作 Power BI 的方式全解析：从 MCP 到 PBIP

2026-06-21 15:15 25 min read 人阅读

一份梳理「AI Agent 如何连接、修改 Power BI」的完整笔记，讲清每种方式的本质、优缺点与选型逻辑（截至 2026 年中）

一、先纠正一个常见的认知偏差

很多人最初接触这个话题时，会把 Agent 操作 Power BI 的方式理解成两个并列的选项：

  • 一种是「通过 MCP 连接 PBIX」；
  • 一种是「直接修改本地的 PBIP」。

这个划分能用，但不够准确，容易让你在后面被各种说法绕进去。更本质的划分维度是这一句话：

Agent 到底是在跟「正在运行的模型引擎」对话，还是在「编辑磁盘上的文本文件」？

想清楚这条主线，后面所有方式都能顺理成章地归位——包括 MCP、PBIP、XMLA、Tabular Editor、REST API。因为它们要么属于「跟引擎对话」，要么属于「改文本」，要么属于「管理运维」这三大类。

下面先把这两种最常被提到的方式讲透，再扩展到其他方式。

二、方式 A：通过 MCP 连接（跟引擎对话）

这里指的是微软官方的 Power BI Modeling MCP Server（npm 包名 @microsoft/powerbi-modeling-mcp，目前处于 Public Preview 阶段，接口在正式发布前可能仍有较大变化）。

它不只是「连 PBIX」

一个关键的事实：这个 MCP 本身就支持三种连接目标，远不止「PBIX」一种：

  1. 连接正在运行的 Power BI Desktop 实例（也就是你当前打开的那个 .pbix）；
  2. 连接 Fabric 工作区里已发布的语义模型；
  3. 打开本地 PBIP 文件夹里的 TMDL 定义。

对应的自然语言连接指令大致是：

Connect to '[文件名]' in Power BI Desktop
Connect to semantic model '[模型名]' in Fabric Workspace '[工作区名]'
Open semantic model from PBIP folder '[PBIP 里 definition / TMDL 文件夹路径]'

它的底层原理

MCP 底层是通过 TOM（Tabular Object Model） 去操作模型对象的。TOM 是一套 .NET 的对象模型抽象，它把数据模型表达成 Model、Table、Column、Measure 这样的类。所以 Agent 说「帮我加一个度量值」，MCP 就翻译成 TOM 的写操作，作用到引擎上。

这里要破除一个误区：「MCP」和「PBIP」不是对立关系。MCP 完全可以操作 PBIP（上面第 3 种连接方式）。所以真正对立的不是「MCP vs PBIP」，而是「连引擎 vs 改文本」。

MCP 能做什么

  • 用自然语言创建/修改表、列、度量值、关系；
  • 批量操作（支持事务，可一次性改几百个对象并带错误处理）；
  • 应用建模最佳实践（Best Practice）；
  • 执行并校验 DAX 查询，调试度量值。

三、方式 B：直接修改本地 PBIP（改文本）

PBIP（Power BI Project） 把一个原本是二进制的 .pbix，拆成了一组纯文本文件：

  • 数据模型部分 → TMDL 文件（.tmdl），描述表、度量值、关系等；
  • 报表部分 → PBIR（report 定义，早期形态是 report.json），描述图表、页面、布局。

既然是文本，Agent 就可以像改普通代码一样，用文件读写工具直接编辑这些文件。这个过程不需要 MCP、不需要 Power BI 在运行，纯粹就是改文本，改完保存。

它最大的价值在于：文本天然适合 Git 版本控制，可以 diff、可以 code review、可以多人协作合并。

四、两种方式的本质区别

把 A 和 B 放在一起对比，区别就非常清晰了：

对比项 方式 A（MCP / 引擎） 方式 B（改 PBIP 文本）
Agent 对话的对象 运行中的 AS 引擎（通过 TOM） 磁盘上的文本文件
是否需要 Power BI 运行 是（Desktop 模式）/ 否（PBIP 模式） 否
改完是否立即生效 是，实时写回引擎 否，要重新打开 / 部署才生效
是否校验语法 / DAX 会（引擎直接拒绝非法操作） 不会，写错了也照样存盘
Git 版本控制友好度 一般（Desktop 模式难 diff） 优秀（文本天然友好）

五、除了这两种，还有哪些方式

Agent 操作 Power BI 远不止上面两条路。按「面向开发者」到「面向自动化运维」排列，还有这些：

1. XMLA Endpoint + TOM（C# / .NET 编程）

直接用 Microsoft.AnalysisServices.Tabular 命名空间写代码，连接已发布模型的 XMLA 端点进行增删改。这是最底层、最强大的方式——MCP 和 Tabular Editor 本质上都是它的封装。

典型连接串形如：

string connectionString =
    "powerbi://api.powerbi.com/v1.0/myorg/{WorkspaceName};initial catalog={ModelName}";
Server server = new Server();
server.Connect(connectionString);
// 修改模型对象
server.Databases["{ModelName}"].Model.SaveChanges();

限制：XMLA 端点只支持 Fabric / Premium / Embedded / PPU 这类专属容量，共享容量用不了；服务主体（Service Principal）在 PPU 下不可用。另外 TOM 本身不能设置数据源凭据，刷新凭据要配合 Power BI REST API。微软已于 2025 年 6 月在所有 Fabric 与 Power BI 容量上默认开启 XMLA 读写。

2. Tabular Editor（GUI + CLI）

社区里采用率最高的第三方建模工具。它的 CLI（命令 te，目前 Limited Public Preview） 跨平台（Windows / macOS / Linux），支持一整套命令：

命令族 作用
te load / save / init 模型读写、转换、初始化
te set / add / rm / mv 增删改移对象属性
te ls / find / diff / deps 列举、搜索、对比、依赖分析
te validate / bpa run / format 校验、最佳实践检查、DAX 格式化
te query / script / macro 跑 DAX、C# 脚本、宏
te deploy / refresh 部署到工作区、触发刷新、增量刷新
te test run 断言测试、快照、A/B 对比

Agent 完全可以通过命令行调用它来批量改模型、接 CI/CD，这是「Agent 工程化操作 Power BI」很现实的一条路。注意 CLI 预览版有时限（预览构建在 2026-09-30 后停用），不建议直接用于生产流水线。

3. Power BI / Fabric REST API

负责的是管理运维面：发布报告、触发刷新、管理语义模型 / 网关 / 权限。它不做细粒度建模（不能直接加一个度量值），但「日更新、自动刷新、部署流水线」全靠它。

4. Semantic Link / sempy（Fabric Notebook 里的 Python）

在 Fabric 笔记本里用 Python 读写语义模型、执行 DAX。适合数据科学 / ML 场景的程序化操作。

5. 远程 MCP Server（querying 版）

微软除了上面那个本地的「Modeling MCP」，还有一个远程 MCP，定位完全不同——它是用来查数据、生成洞察（读），而不是改模型（写）。如果 Agent 的任务是「问数据」而非「改模型」，应该用这个。

六、综合优缺点对比

维度 MCP（连引擎） 直接改 PBIP 文本 XMLA + TOM 编程 Tabular Editor CLI
上手难度 低（自然语言） 低（就是改文本） 高（要写 C#） 中（学命令）
是否实时生效 ✅ 立即 ❌ 要重开 / 部署 ✅ 立即 ✅ 部署后
语法 / DAX 校验 ✅ 引擎拦截非法操作 ❌ 写错也不报错 ✅ ✅ 含 BPA
版本控制（Git） △ Desktop 模式难 diff ✅ 文本天然友好 △ ✅
批量操作 ✅ 支持事务批处理 △ 手工逐文件 ✅ 最强 ✅ 强
报表视觉层（图表） ❌ 基本只管模型 ✅ 可改 PBIR，但易出错 ❌ 只管模型 ❌ 只管模型
依赖环境 需 Node / MCP host 几乎无依赖 需 .NET + 专属容量 需 CLI
适合 CI/CD 自动化 △ ✅ ✅ ✅

MCP 方式小结

优点：自然语言驱动，最适合「边聊边改」的交互式建模；引擎实时校验，DAX 写错、关系冲突会被直接拒绝，不会留下坏模型；支持事务批处理，适合一次性改大量对象。

缺点：仍处 Preview，接口可能变；连 Desktop 实例时不利于 Git diff（改动直接进了二进制 .pbix）；强依赖 MCP host 环境；几乎只覆盖语义模型，不管报表视觉层。

直接改 PBIP 小结

优点：Git 友好是最大杀器——TMDL 是文本，能 diff、能 review、能合并；零运行时依赖；是唯一能碰报表视觉层（PBIR） 的常规方式。

缺点：没有任何校验——写错 DAX 语法、写错引用、漏个括号，文件照样存盘，要重新打开 Power BI 才发现报错，调试反馈慢；复杂改动（改关系、改分区）手工编辑文本繁琐易错；改完不会实时生效。

七、一个重点：MCP 只能改模型层，碰不到图表层

这是一个非常容易踩的认知点，单独拎出来讲。

结论先行：官方的 Modeling MCP 只能修改数据模型层，不能修改图表（报表视觉）层。

为什么

微软官方那个 MCP 的全名就叫 Power BI Modeling MCP Server——名字里的 "Modeling" 已经划定了边界。它底层走 TOM，而 TOM 这套对象模型本身只包含数据模型的东西：

  • 表（Table）、列（Column）
  • 度量值（Measure）、计算列
  • 关系（Relationship）
  • 角色与行级安全（RLS Role）
  • 计算组、分区等

这些统称语义模型 / 数据模型层。而图表、页面、布局、配色、切片器属于报表层（Report layer），在 PBIP 里存放在单独的 PBIR 文件中，完全不在 TOM 的管辖范围内。

所以 MCP 碰不到图表，不是「故意不让改」，而是它操作的引擎层级里根本没有「图表」这个概念。

那图表层怎么用 Agent 改

目前只有一条常规路径：直接编辑 PBIP 里的 PBIR 文本文件（也就是方式 B）。报表的视觉对象、页面、布局都是 JSON / 文本，Agent 用文件读写工具能改。

但有两个坑：

  1. 没有校验、容错极差。PBIR 的 JSON 结构复杂且对格式敏感，Agent 手改很容易写出打不开的报表，而且要重新打开 Power BI 才知道坏没坏，反馈慢。
  2. PBIR 仍在演进。这个增强报表格式微软还在迭代，结构可能变，自动化稳定性不如模型层的 TMDL。

一张表说清层级边界

层级 MCP（Modeling）能改吗 怎么让 Agent 改
数据模型层（表 / 度量值 / 关系 / RLS） ✅ 能，且有校验 MCP 最合适
报表图表层（视觉对象 / 页面 / 布局） ❌ 不能 只能直接改 PBIP 的 PBIR 文本，风险高

八、选型建议

结合不同用途，给一条决策路径：

  • 交互式探索、边问边改模型 → 用 MCP（Desktop 模式），反馈快、有校验。
  • 要进 Git、做 code review、团队协作、CI/CD → 用 PBIP 文本编辑；更优解是 MCP 连 PBIP，既有引擎校验、又落成可 diff 的文本，两者优点兼得。
  • 企业级批量治理、自动化部署、最佳实践检查 → Tabular Editor CLI 或 XMLA + TOM。
  • 只是发布、刷新、管权限（运维面） → REST API。
  • 只是查数、出洞察，不改模型 → 远程 MCP。
  • 要改图表 / 报表布局 → 目前只能直接改 PBIR 文本，且需谨慎校验。

九、总结

把整件事压缩成几句话：

  1. Agent 操作 Power BI 的本质分三类：跟引擎对话（MCP、XMLA+TOM、Tabular Editor）、改文本文件（PBIP 的 TMDL / PBIR）、管理运维（REST API）。
  2. MCP 胜在交互与实时校验，PBIP 胜在版本控制与覆盖报表层，而且两者可以叠加——MCP 操作 PBIP 往往是最佳实践。
  3. 再往工程化、自动化走，就上 Tabular Editor CLI 或直接 XMLA + TOM。
  4. MCP 只管模型层，不管图表层；图表目前没有像模型那样成熟、带校验的 Agent 操作方式，只能退回到「直接改 PBIR 文本」，风险较高。

这套生态变化很快（MCP 还在 Preview，PBIR 还在演进，Tabular Editor CLI 还在预览）。如果未来微软推出专门的「报表层 MCP」或 PBIR 趋于稳定，「MCP 碰不到图表」这条结论可能会被改写——但截至 2026 年中，以上就是全貌。
