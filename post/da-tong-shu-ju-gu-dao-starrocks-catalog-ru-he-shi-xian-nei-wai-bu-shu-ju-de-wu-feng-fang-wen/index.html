打通数据孤岛：StarRocks Catalog 如何实现内外部数据的无缝访问

2026-04-21 19:59 34 min read -- 人阅读

StarRocks Catalog 深度解析：统一内外部数据的“元数据中枢”

一、什么是 Catalog？StarRocks 的“元数据中枢”

在大数据领域，数据源往往是多样且分散的——企业内部可能有 MySQL 业务库、Hive 数据仓库、Iceberg 数据湖，甚至还有外部的 JDBC 数据库。如果每个数据源都需要单独建立连接、单独管理元数据，不仅运维成本高昂，数据孤岛问题也会日益严重。

StarRocks 自 2.3 版本起引入了 Catalog（数据目录） 功能，实现在一套系统内同时维护内部和外部数据，让用户可以轻松访问并查询存储在各类外部数据源中的数据。

通俗理解，Catalog 就像是 StarRocks 的“元数据中枢”——它统一管理着 StarRocks 内部创建的表和数据库，同时也作为连接外部数据源的“桥梁”，让用户无需将数据导入或迁移，就能直接查询 Hive、Iceberg、Hudi、Delta Lake、JDBC 等外部数据源中的数据。

在深入之前，先明确两个基本概念：

  • 内部数据（Internal Data）：指保存在 StarRocks 集群中的数据，通过 CREATE DATABASE 和 CREATE TABLE 语句创建，由 StarRocks 自身管理存储和元数据。
  • 外部数据（External Data）：指保存在外部数据源中的数据，如 Apache Hive、Apache Iceberg、Apache Hudi、Delta Lake 及 JDBC 数据源等。

Catalog 的核心价值在于：用户不再需要在多个系统间切换，通过统一的 SQL 入口即可实现对内外部数据的全面访问，大大降低了数据接入的成本，提升了数据分析的效率。

二、Catalog 的架构设计：FE 统一元数据管理

2.1 统一元数据管理

StarRocks 的 Catalog 系统由 FE（Frontend）节点统一管理。所有 Catalog 的元数据信息（包括 Catalog 名称、类型、连接属性、数据库列表、表结构等）都存储在 FE 的内存中，并通过 EditLog + Checkpoint 机制持久化，保证高可用。

这种设计的优势在于：

  • 集中式管理：一个 FE 集群管理所有 Catalog，元数据访问延迟极低
  • 统一查询入口：用户只需连接 StarRocks，即可查询所有已配置的数据源
  • 权限统一控制：Catalog 级别的权限可以精细控制用户对不同数据源的访问

2.2 查询执行流程

当用户通过 External Catalog 查询外部数据时，StarRocks 会用到外部数据源的两个核心组件：

  1. 元数据服务（Metadata Service）：用于将外部数据的元数据（如库、表、分区、列信息）暴露出来，供 StarRocks 的 FE 进行查询规划。FE 从 Metastore 获取表结构、分区信息后，生成最优的分布式执行计划。
  2. 存储系统（Storage System）：用于实际存储数据文件。数据文件以不同的格式（如 Parquet、ORC、Avro）存储在分布式文件系统（如 HDFS）或对象存储（如 S3、OSS）中。当 FE 将生成的查询计划分发给各个 BE（或 CN）后，各个 BE（或 CN）会并行扫描存储系统中的目标数据，执行计算并返回查询结果。

这种“元数据与数据分离”的架构，使得 StarRocks 能够直接利用外部数据源的现有存储和元数据服务，无需任何数据迁移，实现“零成本”接入。

三、Internal Catalog：内部数据的“大管家”

3.1 什么是 Internal Catalog？

Internal Catalog（内部数据目录） 用于管理 StarRocks 集群内部的所有数据。例如，执行 CREATE DATABASE 和 CREATE TABLE 语句创建的数据库和数据表，都由 Internal Catalog 统一管理。

3.2 default_catalog

每个 StarRocks 集群都有且仅有一个 Internal Catalog，名为 default_catalog。这是 StarRocks 默认生效的 Catalog，当你执行普通的 SELECT * FROM my_table 查询时，StarRocks 会自动在 default_catalog 中查找该表。

3.3 使用 Internal Catalog 查询数据

-- 直接查询内部表（默认使用 default_catalog）
SELECT * FROM my_db.my_table;

-- 显式指定 default_catalog（等效于上一语句）
SELECT * FROM default_catalog.my_db.my_table;

Internal Catalog 不需要手动创建，StarRocks 在集群初始化时就会自动生成。它承载着 StarRocks 自身存储的所有数据，是数据仓库内部数据的唯一管理入口。

四、External Catalog：打通外部数据的“桥梁”

4.1 什么是 External Catalog？

External Catalog（外部数据目录） 用于连接外部的 Metastore 服务。在 StarRocks 中，你可以通过 External Catalog 直接查询外部数据，无需进行数据导入或迁移。

这对于 Lakehouse 架构尤其关键——数据仍然保存在数据湖中（如 Hive、Iceberg 表），StarRocks 只作为计算引擎，通过 External Catalog 读取数据进行极速分析。

4.2 支持的 External Catalog 类型

StarRocks 当前支持创建以下类型的 External Catalog：

Catalog 类型 支持版本 数据源 核心特性
Hive Catalog v2.3+ Apache Hive 支持 HDFS 和对象存储，生态成熟
Iceberg Catalog v2.3+ Apache Iceberg 支持时间旅行、Schema 演化
Hudi Catalog v2.3+ Apache Hudi 支持增量查询、Upsert
Delta Lake Catalog v2.3+ Delta Lake 支持 ACID 事务
JDBC Catalog v2.3+ MySQL、PostgreSQL 等 关系型数据库集成
Paimon Catalog v3.1+ Apache Paimon 流批一体数据湖
Elasticsearch Catalog v3.1+ Elasticsearch 全文搜索集成
Unified Catalog v3.2+ 多数据源统一 统一元数据管理

4.3 创建 External Catalog 示例

以下以创建 Hive Catalog 为例，展示完整的创建语法：

-- 创建 Hive Catalog
CREATE EXTERNAL CATALOG hive_catalog
PROPERTIES (
    "type" = "hive",
    "hive.metastore.uris" = "thrift://192.168.1.100:9083"
);

创建 Iceberg Catalog（基于 Hive Metastore）：

CREATE EXTERNAL CATALOG iceberg_catalog
PROPERTIES (
    "type" = "iceberg",
    "iceberg.catalog.type" = "hive",
    "hive.metastore.uris" = "thrift://192.168.1.100:9083"
);

创建 JDBC Catalog（以 MySQL 为例）：

CREATE EXTERNAL CATALOG mysql_catalog
PROPERTIES (
    "type" = "jdbc",
    "jdbc.driver" = "com.mysql.cj.jdbc.Driver",
    "jdbc.url" = "jdbc:mysql://127.0.0.1:3306",
    "jdbc.user" = "root",
    "jdbc.password" = "123456"
);

五、Unified Catalog：多源统一的终极方案

5.1 什么是 Unified Catalog？

从 StarRocks 3.2 版本开始，引入了 Unified Catalog（统一数据目录）。它将 Apache Paimon、Apache Hive、Apache Iceberg、Apache Hudi、Apache Kudu 和 Delta Lake 视为一个融合的统一数据源，让你可以在不先导入数据的情况下，跨所有这些格式查询和管理数据。

在 Unified Catalog 诞生之前，如果你需要同时查询 Hive 表和 Iceberg 表，需要分别创建 Hive Catalog 和 Iceberg Catalog，然后通过跨 Catalog 查询来关联。Unified Catalog 则将所有表格式统一在同一个 Catalog 下管理，极大简化了多源数据的访问体验。

5.2 创建 Unified Catalog

CREATE EXTERNAL CATALOG unified_catalog
COMMENT '统一数据湖目录'
PROPERTIES (
    "type" = "unified",
    "unified.metastore.type" = "hive",
    "hive.metastore.uris" = "thrift://hive-metastore:9083"
);

5.3 支持的操作

操作 Paimon Hive Iceberg Hudi Kudu Delta Lake
查询（SELECT） ✅ ✅ ✅ ✅ ✅ ✅
导入到 StarRocks ❌ ✅ ✅ ✅ ✅ ✅
创建/删除数据库 ❌ ✅ ✅ ❌ ❌ ❌
创建/删除表 ❌ ✅ ✅ ❌ ❌ ❌
插入数据 ❌ ✅(Parquet) ✅(Parquet) ❌ ❌ ❌

六、Catalog 的日常操作与管理

6.1 查看 Catalog

-- 查看当前集群中的所有 Catalog
SHOW CATALOGS;

-- 查看某个 External Catalog 的创建语句
SHOW CREATE CATALOG hive_catalog;

6.2 切换 Catalog

-- 切换当前会话生效的 Catalog
SET CATALOG hive_catalog;

-- 切换到指定 Catalog 下的数据库
USE hive_catalog.my_hive_db;

6.3 查看 Catalog 下的数据库和表

-- 查看指定 Catalog 下的所有数据库
SHOW DATABASES FROM hive_catalog;

-- 查看表结构
DESC hive_catalog.my_hive_db.my_hive_table;

6.4 删除 Catalog

-- 删除 External Catalog
DROP CATALOG hive_catalog;

6.5 查询外部数据

切换到 Hive Catalog 后，可以直接查询其中的表：

-- 切换 Catalog
SET CATALOG hive_catalog;

-- 查询数据
SELECT * FROM my_hive_db.sales_table LIMIT 10;

七、跨 Catalog 联邦查询：打通数据孤岛

StarRocks 最强大的能力之一，是支持跨 Catalog 的联邦查询（Federated Query）。你可以在一条 SQL 中同时查询 Internal Catalog 中的内部表和多个 External Catalog 中的外部表，实现真正的“数据联邦”。

语法格式为：catalog_name.db_name.table_name

7.1 在 default_catalog 中查询外部表

-- 在内部 Catalog 中，查询 Hive Catalog 中的表
SELECT * FROM hive_catalog.hive_db.hive_table;

7.2 内部表与外部表的 JOIN

-- 将 Hive 外部表与 StarRocks 内部表进行 JOIN 分析
SELECT h.order_id, h.amount, o.user_name
FROM hive_catalog.sales_db.orders h
JOIN default_catalog.olap_db.users o
ON h.user_id = o.user_id;

7.3 多源联邦查询

-- 一条 SQL 同时查询 Hive、Iceberg 和 StarRocks 内部表
SELECT 
    h.order_id,
    i.customer_segment,
    s.total_revenue
FROM hive_catalog.sales_db.orders h
JOIN iceberg_catalog.customers_db.customers i
    ON h.customer_id = i.customer_id
JOIN default_catalog.analytics_db.sales_summary s
    ON h.order_date = s.sale_date
WHERE h.order_date >= '2024-01-01';

这种跨 Catalog 的联邦查询能力，让 StarRocks 真正成为了企业数据架构中的“统一查询网关”——用户不需要关心数据到底存在哪里，只需要关心如何分析数据。

八、Catalog 权限管理

在多用户环境下，精细化的权限控制至关重要。StarRocks 支持对 Catalog 进行细粒度的权限管理。

8.1 常用 Catalog 权限

权限名称 核心作用 示例操作
USAGE 允许访问目标 Catalog（基础权限） SET CATALOG、SHOW DATABASES
CREATE DATABASE 允许在 Catalog 下创建新数据库 CREATE DATABASE
DROP 允许删除 Catalog 下的数据库 DROP DATABASE
ALL 授予 Catalog 下所有权限 所有操作

8.2 授权示例

-- 授予用户访问 Catalog 的基础权限
GRANT USAGE ON CATALOG hive_catalog TO 'user1'@'%';

-- 授予用户在 Catalog 下创建数据库的权限
GRANT CREATE DATABASE ON CATALOG hive_catalog TO 'user1'@'%';

-- 授予角色对 Catalog 内所有表和视图的查询权限
GRANT SELECT ON ALL TABLES IN ALL DATABASES TO ROLE analyst_role;

执行授权操作的用户需要具备 ADMIN 权限（如 root 用户），或拥有被授权 Catalog 的 GRANT OPTION（授权权限）。

8.3 中心化访问控制（StarRocks 4.0+）

在 StarRocks 4.0 中，联合 Apache Iceberg 引入了基于 REST Catalog 的中心化访问控制能力，支持：

  • 权限控制（Authorization）：在数据库、表乃至列级别执行精细化访问策略
  • 凭证管理（Credential Vending）：按需签发短期、最小权限的访问令牌，确保数据安全

当身份验证、权限控制与凭证签发都交由 Catalog 统一管理后，多引擎协同访问同一数据湖的安全问题得到了彻底解决。

九、Catalog 在湖仓一体架构中的核心作用

在 Lakehouse 架构中，Catalog 扮演着“数据湖与数据仓库之间的桥梁”角色。

传统架构的痛点：在传统的架构中，往往需要将 Hive 中的数据重新导入到 StarRocks 中（或导入到其他 OLAP 引擎中）做进一步的加速处理。这种方式带来了数据冗余、同步延迟和运维复杂度。

Catalog 带来的变革：通过 External Catalog，StarRocks 可以直接分析湖上的所有数据，而无需逐个表进行 Schema 建模和导入。用户只需创建一个 Hive/Iceberg Catalog，指定一个元数据地址（如 Hive Metastore、阿里云 DLF 或 AWS Glue），StarRocks 就可以直接根据元数据将所有的库表信息同步过来。

在实际企业实践中，这种模式已被广泛采用：

  • 腾讯微信将数据写入 Iceberg 作为统一存储，借助 StarRocks 实现准实时分析，数据新鲜度大幅提升，查询性能提高 3-6 倍。
  • 携程旅行把数据统一存储在 Hive，利用 StarRocks 直接服务 BI 报表和交互式分析，并按需创建物化视图，使查询性能提升 10 倍以上。

十、最佳实践与注意事项

10.1 命名规范

  • Catalog 名称只能包含字母、数字和下划线（_），必须以字母开头
  • 建议使用有意义的名称，如 hive_prod、iceberg_warehouse 等

10.2 权限规划

  • 遵循最小权限原则，普通用户只授予 USAGE 权限
  • 创建数据库、删除数据库等高风险权限仅授予管理员
  • 建议使用角色（ROLE）管理权限，而非直接授予用户

10.3 元数据同步

  • External Catalog 的元数据会定期从外部 Metastore 同步，同步间隔可通过参数调整
  • 如果外部数据源新增了表或分区，StarRocks 会在元数据刷新周期后自动感知

10.4 网络与连通性

  • 确保 StarRocks 集群可以访问外部数据源的 Metastore 服务（如 HMS 的 9083 端口）
  • 确保 BE/CN 节点可以访问数据文件的存储系统（如 HDFS DataNode、S3 Endpoint）

10.5 性能优化建议

  • 对于高频查询的外部表，建议在 StarRocks 中创建异步物化视图进行加速
  • 合理设置分区裁剪策略，减少 BE 扫描的数据量
  • 利用 StarRocks 的 CBO 优化器和统计信息，生成最优的跨 Catalog 查询计划

十一、总结

StarRocks 的 Catalog 系统是实现“统一数据访问”的基石，它的设计哲学可以概括为：

  1. 统一管理：Internal Catalog 管理内部数据，External Catalog 连接外部数据源，Unified Catalog 统一多源管理，一个入口访问所有数据。
  2. 零数据迁移：无需将外部数据导入 StarRocks，直接通过 Catalog 进行查询，节省存储成本和同步开销。
  3. 联邦查询：支持跨 Catalog 的 JOIN 查询，一条 SQL 打通数据孤岛，内部表与外部表无缝关联分析。
  4. 生态丰富：全面支持 Hive、Iceberg、Hudi、Delta Lake、Paimon、JDBC、Elasticsearch 等主流数据源，覆盖绝大多数企业数据场景。
  5. 精细权限：Catalog 级别的权限控制，结合 StarRocks 4.0 的中心化访问控制，保障数据安全。

如果说 FE 是 StarRocks 的“智慧大脑”，BE 是“强健四肢”，那么 Catalog 就是 StarRocks 的“信息中枢”——它让 StarRocks 不再只是一个高性能分析引擎，更是一个能够连接万物的数据枢纽。理解 Catalog，就是理解了 StarRocks 如何成为 Lakehouse 时代的数据分析核心。
