StarRocks FE 元数据持久化揭秘：EditLog 与 Checkpoint 机制深度解析

2026-04-21 19:56 21 min read 人阅读

EditLog + Checkpoint：StarRocks FE 如何实现元数据的“永不丢失”与“秒级恢复”

一、元数据：StarRocks 集群的“核心资产”

在深入 EditLog 和 Checkpoint 之前，我们需要先理解它们守护的对象——元数据。

StarRocks 集群的元数据包含了所有关键信息：有哪些 Catalog、数据库、表、列、分区、分桶、物化视图、用户权限、BE 节点状态等。没有元数据，存储在各 BE 节点上的数据文件就成了一堆无法解读的二进制碎片。

FE（Frontend）节点负责管理全部元数据，并且为了极致性能，元数据主要存储在 FE 的内存中。内存存储带来了亚毫秒级的访问延迟，但同时也带来了一个致命问题：一旦 FE 进程重启或宕机，内存中的所有元数据将瞬间蒸发。

EditLog 和 Checkpoint，正是为解决这一难题而生的两大利器。

二、EditLog：元数据变更的“流水日记”

2.1 什么是 EditLog？

EditLog（编辑日志）是一份顺序记录所有元数据变更操作的日志文件。每当用户或系统执行一条会改变元数据的命令——例如 CREATE TABLE、ALTER TABLE ADD PARTITION、DROP VIEW、GRANT 等——FE 在执行操作之前，会先将这次操作记录到 EditLog 中。

可以把它理解为数据库的 “操作日记” ：不是记录“现在元数据长什么样”，而是记录“谁在什么时候做了什么改动”。只要按顺序重放这些日记，就能从零开始重建出完整的元数据状态。

2.2 EditLog 的存储：BDBJE

StarRocks FE 的 EditLog 并不是简单写在本地文件里，而是依托于 BDBJE（Berkeley DB Java Edition） 进行存储和管理。BDBJE 是一个高性能、支持事务的嵌入式键值存储引擎，天然支持数据持久化和多副本同步。

选择 BDBJE 的原因有三：

  • 事务保证：元数据变更与 EditLog 写入是原子操作，要么全部成功，要么全部失败，杜绝“半成功”状态。
  • 高可用同步：BDBJE 内置了类 Paxos 的复制协议，Leader FE 写入的 EditLog 会自动同步到所有 Follower FE 的 BDBJE 实例中。
  • 高效读写：基于 LSM 树结构，适合顺序写入和随机读取的混合负载。

2.3 EditLog 的同步机制

在 FE 集群中，只有 Leader FE 拥有写入 EditLog 的权限。同步流程如下：

  1. Leader FE 接收到元数据变更请求（如建表）。
  2. Leader 将该操作序列化为一条 EditLog 条目，提交到本地 BDBJE。
  3. BDBJE 通过内部复制协议，将这条 EditLog 同步推送给所有 Follower FE 和 Observer FE 的 BDBJE 实例。
  4. 必须等到超过半数的 Follower FE 确认同步成功后，Leader 才认为本次写入真正完成。
  5. 各 FE 节点在后台异步将 EditLog 回放（Replay） 到自己的内存元数据中。

这种机制带来了两个关键好处：

  • 数据不丢失：只要多数派节点存活，即使 Leader 宕机，新 Leader 也能从完整的 EditLog 中恢复全部元数据。
  • 读写分离：Follower 和 Observer 通过回放 EditLog 保持内存元数据与 Leader 一致，从而对外提供一致的只读查询服务。

2.4 EditLog 的格式与生命周期

每条 EditLog 由三部分组成：

  • JournalId：全局递增的唯一序号，用于确定回放顺序。
  • OperationType：操作类型编码，如 OP_CREATE_TABLE、OP_ADD_PARTITION。
  • Data：操作的具体内容，如建表的 SQL 文本或序列化后的表结构对象。

EditLog 会随时间不断累积。如果永远不清理，BDBJE 的存储空间会被耗尽，FE 重启时从头回放上百万条日志也会极其缓慢。这正是 Checkpoint 登场的时刻。

三、Checkpoint：元数据的“定期快照”

3.1 什么是 Checkpoint？

Checkpoint 是 FE 在某一时刻将内存中的完整元数据序列化并持久化到磁盘的静态快照。这个快照文件被称为 Image 文件，它记录了生成时刻元数据的全貌，而非变更过程。

如果说 EditLog 是“流水日记”，那么 Checkpoint 就是“定期拍摄的纪念相册”。有了相册，你不需要从出生第一天的日记开始读，只需要翻到最近一本相册，再读几页后续日记即可。

3.2 Checkpoint 的触发时机

StarRocks 提供了灵活的 Checkpoint 触发策略，可通过 FE 配置参数调整：

参数 默认值 说明
edit_log_roll_num 50000 EditLog 累计达到该数量后触发 Checkpoint
checkpoint_interval_second 60 后台线程检查间隔（秒）
max_edit_log_roll_num 200000 EditLog 数量强制触发阈值

通常以 edit_log_roll_num 为主要触发条件。当 Leader FE 检测到自上次 Checkpoint 以来累积的 EditLog 条数超过 50,000 条时，就会启动一次 Checkpoint 流程。

3.3 Checkpoint 的生成流程

Checkpoint 由 Leader FE 的后台线程负责执行，主要步骤：

  1. 冻结当前状态：获取一个一致性读视图，确保序列化过程中内存元数据不会被并发修改。
  2. 序列化元数据：将 GlobalStateMgr 中的全部对象（数据库、表、分区、物化视图、权限等）递归序列化为二进制流。
  3. 写入 Image 文件：将二进制流写入 FE 本地目录 fe/meta/image/，文件命名格式为 image.{JournalId}，其中 JournalId 是生成时刻已回放的最新 EditLog 序号。
  4. 清理旧文件：新 Image 生成成功后，删除比当前 Image 更旧的 Image 文件和对应的 EditLog 归档文件。

Image 文件命名示例：

image.150000   // 表示该快照包含了 JournalId ≤ 150000 的所有元数据变更

3.4 Image 文件的分发

Image 文件仅在 Leader FE 上生成，但 Follower 和 Observer 也需要定期获取最新快照，否则重启时回放的 EditLog 会过长。

分发机制有两种：

  • 被动拉取：Follower/Observer 检测到自身 EditLog 落后 Leader 过多，或本地没有最新 Image 时，会通过 HTTP 从 Leader 下载 Image 文件。
  • 主动清理触发：当 Follower 发现需要回放的 EditLog 数量超过阈值，会主动向 Leader 请求最新 Image。

这种设计避免了每个 FE 都独立执行昂贵的 Checkpoint 计算，Leader 生产一份，全集群共享。

四、FE 重启恢复：EditLog + Checkpoint 的完美协作

当任何一个 FE 节点重启（无论是计划内维护还是意外宕机），恢复流程如下：

阶段一：加载基础快照

  1. FE 扫描本地 fe/meta/image/ 目录，找到 JournalId 最大的 Image 文件。
  2. 将 Image 文件反序列化到内存，恢复出一个“基础版本”的元数据。此时元数据状态定格在生成该 Image 的时刻。

阶段二：回放增量 EditLog

  1. FE 连接 BDBJE，读取 JournalId 大于 Image 文件中记录 JournalId 的所有 EditLog。
  2. 按照 JournalId 从小到大的顺序，逐条回放（Replay）EditLog，将内存元数据更新至最新状态。

阶段三：就绪服务

  1. 回放完成后，FE 将自己的角色（Leader/Follower/Observer）与集群当前状态对齐。
  2. 如果是 Follower 或 Observer，开始监听 Leader 后续推送的新 EditLog。
  3. 对外提供服务。

为什么重启这么快？

因为恢复的核心耗时在于“回放 EditLog 的数量”。Checkpoint 机制将需要回放的 EditLog 数量控制在一个可配置的上限内（默认最多 50,000 条），而非从头回放几百万条。这使得 FE 的重启恢复时间通常在 几秒到几十秒 之间，保证了集群的高可用性。

五、日常运维与监控

理解原理后，还需要知道如何观察和维护 EditLog 与 Checkpoint 的状态。

5.1 查看 EditLog 和 Checkpoint 状态

-- 查看 FE 节点元数据状态（包括 JournalId、Checkpoint 版本等）
SHOW PROC '/frontends';

-- 查看详细的 EditLog 统计信息
SHOW PROC '/statistic';

关注关键指标：

  • ReplayedJournalId：当前 FE 已回放的最新 EditLog 序号，应与 Leader 的 MaxJournalId 尽量接近。
  • IsCheckpointing：是否正在执行 Checkpoint。
  • LastCheckpointJournalId：最近一次 Checkpoint 对应的 JournalId。

5.2 手动触发 Checkpoint

正常情况下无需手动干预，但在进行重大元数据变更后（如批量建表），可以主动触发一次 Checkpoint 以立即归档状态：

ADMIN SET FRONTEND CONFIG ("edit_log_roll_num" = "1");
-- 等待 Checkpoint 完成后恢复原值
ADMIN SET FRONTEND CONFIG ("edit_log_roll_num" = "50000");

5.3 故障恢复场景处理

场景 处理方式
Leader 宕机无法恢复 从剩余 Follower 中自动选举出新 Leader，新 Leader 拥有完整 EditLog 和 Image，数据零丢失。
所有 FE 的本地 Image 文件均损坏 从备份的 Image 文件恢复，然后从 BDBJE 回放 EditLog。BDBJE 自身有多副本，通常不会全部损坏。
EditLog 回放卡住 可能是 BDBJE 数据损坏或网络分区导致，需检查 bdbje_heartbeat_timeout_second 等网络配置，必要时从健康节点复制 BDBJE 数据目录。

六、最佳实践建议

  1. 合理设置 Checkpoint 阈值：默认 50,000 条适用于大多数场景。如果元数据变更极其频繁（如大量动态分区创建），可适当调低至 30,000 条以减少恢复时间；如果变更很少，可调高至 80,000 条以减少 Checkpoint 频率。
  2. 定期备份 Image 文件：虽然 BDBJE 有多副本，但建议将 Leader 生成的 Image 文件定期备份到外部存储（如对象存储），作为极端灾难的恢复手段。
  3. 监控 EditLog 堆积数量：设置告警，当 MaxJournalId - ReplayedJournalId 持续大于 10,000 时，检查 Follower 同步是否延迟或网络是否抖动。
  4. 避免频繁的小型元数据操作：例如在循环中逐条执行 ALTER TABLE ADD PARTITION，会快速产生大量 EditLog。尽量合并为批量操作。

七、总结

EditLog 与 Checkpoint 是 StarRocks FE 高可用架构中不可或缺的两块基石：

机制 角色 特点
EditLog 增量流水 顺序记录每次元数据变更，基于 BDBJE 实现多副本同步，保证数据不丢失
Checkpoint 全量快照 定期将内存元数据序列化为 Image 文件，控制恢复时需要回放的日志量

两者配合，实现了 “数据零丢失 + 恢复秒级” 的目标。理解这套机制，不仅有助于日常运维和故障排查，更能让你深刻体会到 StarRocks 在设计之初对稳定性和高可用性的极致追求。
