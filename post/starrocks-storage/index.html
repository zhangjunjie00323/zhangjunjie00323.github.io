StarRocks 数据存储说明

2026-05-14 21:20 58 min read 人阅读

本文用同一份业务数据贯穿说明：在 StarRocks 中从 Table → Partition → Tablet → Rowset → Segment 各层级分别存什么、怎么组织，并配有可对照的数字与行级示例，便于建立直观印象。

1. 示例业务：订单明细表

假设有一张订单明细表 dwd_order_detail，逻辑上就是一张「用户可见的表」。

1.1 示例数据（10 行）

下面这 10 行是业务语义上的原始行（你在 SELECT * 时看到的形态）：

order_id user_id city order_date amount
10001 U01 杭州 2026-01-15 120.00
10002 U02 上海 2026-01-15 88.50
10003 U01 杭州 2026-01-16 200.00
10004 U03 北京 2026-01-16 45.00
10005 U02 上海 2026-01-17 300.00
10006 U04 深圳 2026-01-17 60.00
10007 U01 杭州 2026-01-18 99.00
10008 U05 广州 2026-01-18 150.00
10009 U03 北京 2026-01-19 70.00
10010 U02 上海 2026-01-19 110.00

建表时若约定：

  • 分区键：order_date（按天 RANGE 分区，便于按日期裁剪）。
  • 分桶键：user_id（DISTRIBUTED BY HASH(user_id) BUCKETS 4，便于并行与分散热点；示例里故意用较少桶数便于数清楚）。

则这 10 行在物理上会落入「若干分区 → 若干 Tablet → 多次写入形成多个 Rowset → 每个 Rowset 再切成 Segment 文件」。

2. 层级总览：谁存「什么」

层级 对用户是否直接可见 主要存什么 典型大小/数量级（经验值，非硬性上限）
Table 是 表名、列定义、模型（Duplicate/Aggregate/Unique/Primary）、分区/分桶规则等元数据 + 全表数据的逻辑集合 行数、列数无上限（受集群与运维约束）
Partition 间接（通过分区列体现） 某个分区键取值范围内的所有行（逻辑切片） 单分区常见建议量级：数十 GB 级内更易治理（过大则运维与压缩压力上升）
Tablet 否（内部单元） 哈希分桶后落在同一分片上的行子集 + 多副本 单 Tablet 常见观察：百 MB～数 GB；桶数由 BUCKETS n 决定
Rowset 否 一次导入或一次 Compaction 产生的一批不可变数据（版本视角的一段增量） 与导入批次、Compaction 策略强相关
Segment 否 列存物理文件（Segment 文件），真正落盘 单文件常见上限量级：数百 MB 级（版本差异以官方为准）

下面按层级展开，并始终用 §1 的 10 行订单 做「同一批数据在不同层级长什么样」的对照。

3. Table（表）：逻辑全集 + 元数据

3.1 存什么

  • 逻辑上：整张表的所有行（你看到的全部业务数据）。
  • 物理上：表本身不等于「一个大文件」，而是元数据 + 指向各 Partition/Tablet 的组织关系。
  • 元数据包括（概念列举）：库名、表名、列类型、键模型、分区表达式、分桶列与桶数、副本数、索引/排序键等。

3.2 用示例数据怎么理解

对上文的 10 行来说：

  • Table 层回答的问题是：「这张表里允许出现哪些列、按什么规则分区与分桶、用什么模型存」。
  • 这 10 行在 Table 层就是「都属于 dwd_order_detail 这一逻辑集合」；至于它们最终在哪个 Tablet，由下一层决定。

3.3 元数据在哪里、行数据在哪里（概念分工）

便于和集群组件对应，可以粗略记：

  • FE（Frontend）：保存表的逻辑定义与分区/分桶/Tablet 路由信息等元数据（你 SHOW CREATE TABLE、DESC 看到的内容归属这类）。查询规划阶段需要知道「去哪些 Tablet 取数」。
  • BE（Backend）：在本地磁盘上保存 Tablet 的数据文件（Rowset、Segment 等落在这里）。同一 Tablet 的多个副本会在多个 BE 上各存一份。

因此：Table 不等于某一个 BE 上的一个大文件；Table 是逻辑对象，物理数据按 Tablet 分散在多个 BE 上。

4. Partition（分区）：按分区键切的第一刀

4.1 存什么

  • 每个分区保存：partition_key 落在该分区范围内的所有行（逻辑上的子集）。
  • 查询裁剪：WHERE order_date = '2026-01-16' 时，优化器尽量只扫 2026-01-16 对应分区，减少 IO。

4.2 用示例数据划分分区

按 order_date 每天一个分区，则 10 行会落入 5 个分区（每天若干行）：

分区（示例命名） 包含的 order_id（来自 §1） 行数
p20260115 10001, 10002 2
p20260116 10003, 10004 2
p20260117 10005, 10006 2
p20260118 10007, 10008 2
p20260119 10009, 10010 2

直观理解：分区像「按日期分抽屉」——每个抽屉里仍然可能有多个城市、多个用户，但日期范围先把数据粗分一遍。

4.3 分区与示例行的「一一对应」

把 §1 的每一行都标上逻辑分区名（便于后文引用）：

order_id order_date 逻辑分区
10001 2026-01-15 p20260115
10002 2026-01-15 p20260115
10003 2026-01-16 p20260116
10004 2026-01-16 p20260116
10005 2026-01-17 p20260117
10006 2026-01-17 p20260117
10007 2026-01-18 p20260118
10008 2026-01-18 p20260118
10009 2026-01-19 p20260119
10010 2026-01-19 p20260119

4.4 分区类型（知识点，与示例无关）

常见分区思路包括 RANGE（按连续区间，如日期）、LIST（按枚举值列表）、以及表达式分区（用表达式定义分桶规则，具体语法以版本文档为准）。示例表采用的是「按天 RANGE」这一最常见分析场景。

5. Tablet（分片 / 数据分片）：分桶后的并行与分布单元

5.1 存什么

  • 在每个分区内，再按 DISTRIBUTED BY HASH(user_id) BUCKETS 4 把行哈希到 4 个 Tablet（编号记为 T0～T3）。
  • 副本：每个 Tablet 会有多份（如 3 副本），分布在不同 BE 上，用于容错；业务行内容相同，只是副本位置不同。

5.2 用示例数据演示分桶（简化哈希示意）

真实哈希值由引擎计算；为便于阅读，这里用 hash(user_id) % 4 的假想结果 演示分布（仅教学用，不等同真实哈希函数）：

order_id user_id order_date 假想 bucket_id = hash(user_id) % 4
10001 U01 2026-01-15 1 → Tablet T1
10002 U02 2026-01-15 2 → Tablet T2
10003 U01 2026-01-16 1 → Tablet T1
10004 U03 2026-01-16 3 → Tablet T3
10005 U02 2026-01-17 2 → Tablet T2
10006 U04 2026-01-17 0 → Tablet T0
10007 U01 2026-01-18 1 → Tablet T1
10008 U05 2026-01-18 0 → Tablet T0
10009 U03 2026-01-19 3 → Tablet T3
10010 U02 2026-01-19 2 → Tablet T2

在分区 p20260115 内两行会落到不同 Tablet：

  • 10001（U01）→ T1
  • 10002（U02）→ T2

在分区 p20260116 内：

  • 10003（U01）→ T1
  • 10004（U03）→ T3

直观理解：

  • 分区决定「这行属于哪一天那个抽屉」。
  • Tablet决定「在这个抽屉里，这行按用户哈希后放到哪个格子」，格子之间可并行扫描。

5.3 倾斜示例（帮助理解为什么要选好分桶键）

若把分桶键改成 city，而数据里 80% 行 city='杭州'，则大量行会落到同一 Tablet，并行度名存实亡——这是存储层「热点」的常见来源之一。

5.4 全量映射：每一行落在「哪个分区 + 哪个 Tablet」

在 §4 按天分区、§5.2 假想 hash(user_id) % 4 的前提下，10 行订单的完整逻辑地址如下（这是理解「扫描裁剪 + 并行度」最有用的一张表）：

order_id user_id order_date 分区 Tablet（假想）
10001 U01 2026-01-15 p20260115 T1
10002 U02 2026-01-15 p20260115 T2
10003 U01 2026-01-16 p20260116 T1
10004 U03 2026-01-16 p20260116 T3
10005 U02 2026-01-17 p20260117 T2
10006 U04 2026-01-17 p20260117 T0
10007 U01 2026-01-18 p20260118 T1
10008 U05 2026-01-18 p20260118 T0
10009 U03 2026-01-19 p20260119 T3
10010 U02 2026-01-19 p20260119 T2

Tablet 总数（逻辑分片数）：每个分区都有 4 个 Tablet（T0～T3），共 5 个分区，故「分片槽位」共 5 × 4 = 20 个；其中部分槽位在本示例中只有 0 或 1 行数据，但槽位仍然存在。

5.5 副本与 BE：同一 Tablet 上的数据长什么样

设 replication_num = 3，集群有 BE-a、BE-b、BE-c 三台。则 Tablet T1 在分区 p20260116 上的那一份数据（逻辑上包含 order_id=10003 那一行），会在三个 BE 上各存 一整套 Rowset/Segment 文件（内容等价，路径不同），任一副本损坏可通过其他副本修复。

用示意（非真实路径）：

副本 所在节点 含义
T1-p20260116-replica1 BE-a T1 的第一份物理数据
T1-p20260116-replica2 BE-b T1 的第二份物理数据
T1-p20260116-replica3 BE-c T1 的第三份物理数据

查询时，FE 会为每个 Tablet 选择一个可读的副本下发扫描任务；导入时，写入需满足多数副本成功等一致性策略（细节以官方「副本写入协议」为准）。

6. Rowset：一次写入或一次 Compaction 产生的「不可变批次」

6.1 存什么

  • Rowset 不是业务表里的某一列，而是存储引擎内部的一次数据快照单元：
    • Stream Load / Broker Load / Routine Load / INSERT 等每次成功写入，通常会生成一个或多个 Rowset（与批次大小、并行度有关）。
    • Compaction：后台把多个小 Rowset 合并成更少、更大的 Rowset，以减少碎片、提升扫描效率。

6.2 用同一批示例数据模拟两次导入

假设 不是 一次性写入 10 行，而是分两次导入到同一张表、同一分区：

第一次导入（仅 2026-01-15～2026-01-16 的 4 行）：

order_id user_id city order_date amount
10001 U01 杭州 2026-01-15 120.00
10002 U02 上海 2026-01-15 88.50
10003 U01 杭州 2026-01-16 200.00
10004 U03 北京 2026-01-16 45.00

第二次导入（剩余 6 行）：

order_id user_id city order_date amount
10005 U02 上海 2026-01-17 300.00
10006 U04 深圳 2026-01-17 60.00
10007 U01 杭州 2026-01-18 99.00
10008 U05 广州 2026-01-18 150.00
10009 U03 北京 2026-01-19 70.00
10010 U02 上海 2026-01-19 110.00

在存储引擎视角（概念化）可能出现：

  • Rowset R1（由第一次导入生成）：包含多个 Tablet 上的增量文件集合。
  • Rowset R2（由第二次导入生成）：同理。

对 Tablet T1（一直承接 user_id = U01 的行）来说，时间线上可能先后承接：

  • R1 中带 10001、10003 的片段；
  • R2 中带 10007 的片段。

Compaction 之后（示意）：R1、R2 在 T1 上可能被合并成 Rowset R3，对外查询结果仍是那几行业务行，但底层文件更少、更顺。

一句话：Rowset 是「某次写入或某次合并留下的版本化数据包」；查询会按可见版本规则在多个 Rowset 上合并出最终行集。

6.3 同一 Tablet 上多个 Rowset 时，查询怎么「看成一行」

仍看 Tablet T1（§5.2 中承接 U01 的 10001、10003、10007）。若两次导入分别在 T1 上留下 Rowset R1（含 10001、10003）与 Rowset R2（含 10007），则 SELECT 扫描 T1 时，执行引擎需要：

  1. 枚举该 Tablet 上当前可见的所有 Rowset（R1、R2，以及 Compaction 之后可能替换为 R3）。
  2. 对每个 Rowset 读取其 Segment 中的列块。
  3. 在算子层（如 Scan）把多段数据按版本与模型规则拼成连续行流。

对 Duplicate Key 模型，多版本多副本语义相对简单：多行都保留，除非业务用 SQL 去重。若换成 Unique / Primary Key，同一主键上的多版本还可能涉及合并读（以最新版本为准），存储层会配合删除向量或合并策略（实现细节因版本与模型而异，此处只建立概念）。

6.4 用时间线表示 Rowset 与 Compaction（Tablet T1 示意）

时间 →

导入1 ──► T1 上出现 Rowset R1（行：10001, 10003）
导入2 ──► T1 上出现 Rowset R2（行：10007）
后台 Compaction ──► 可能合并为 Rowset R3（仍对应三行，文件更少）

要点：Compaction 不改变这三次业务上的「事实行」集合（仍是三行），改变的是底层文件布局与 Rowset 个数。

7. Segment：列存落盘文件

7.1 存什么

  • Segment 是 列存格式的物理文件（每列或列组按块存储，带编码与压缩）。
  • 一个 Rowset 可由 一个或多个 Segment 组成（与写入大小、阈值有关）。

7.2 用示例数据在「列方向」上理解

仍取 Tablet T1 上与 user_id = U01 相关的行（10001、10003、10007），在 Segment 中更接近下面这种按列存储的逻辑视图（数值为示意）：

列 order_id 的一段编码数据块（顺序与行对应关系由元数据维护）：

[10001, 10003, 10007]

列 amount 的一段编码数据块：

[120.00, 200.00, 99.00]

列 order_date 可能以字典或编码形式存放（概念上仍是三个值）：

['2026-01-15', '2026-01-16', '2026-01-18']

直观理解：

  • 你在 SQL 里写的是行；
  • 在 Segment 里更多是列块 + 编码 + 压缩 + 索引信息；
  • 分析型查询只读少数列时，可以只解码相关列，减少 IO。

7.3 Segment 内部大致有什么（概念层）

Segment 除了「列数据块」外，通常还带有辅助元数据，用于跳过无关数据与定位行：

  • 列编码与压缩：例如字典编码对低基数 city 列往往更省空间。对 §1 数据，city 取值只有 {杭州, 上海, 北京, 深圳, 广州}，字典编码后列块里可能存的是短整型编码 + 字典表，查询 WHERE city='杭州' 时可先解字典再扫块。
  • Zone Map / 列统计（概念）：记录每个数据块内某列的 min/max 等；若查询带 order_date = '2026-01-16'，可能跳过根本不包含该日期的块（具体能否跳过取决于块边界与数据排列）。
  • Ordinal / 行号映射：把列存块内的顺序与逻辑行对应起来，便于与其他列对齐组装成行。

7.4 仍用 order_id = 10003 说明「一行在 Segment 里不是连续一块」

业务行：

10003, U01, 杭州, 2026-01-16, 200.00

在 Segment 中更接近：

  • order_id 列块里某处存 10003；
  • user_id 列块里同一序位存 U01；
  • amount 列块里同一序位存 200.00；

它们通过**相同行序（ordinal）**对齐，而不是像 CSV 那样把五个字段粘在同一字节串里。

8. 把五层串成一条「单行旅程」

以 order_id = 10003 这一行为例：

  1. Table：属于 dwd_order_detail。
  2. Partition：order_date = 2026-01-16 → 分区 p20260116。
  3. Tablet：user_id = U01 → 假想到 T1。
  4. Rowset：若该行来自「第一次导入」→ 落在 Rowset R1 的某个 Segment 中；若之后又有一次导入只影响别的行，则 10003 仍可能只在 R1 可见，直到 Compaction 与版本管理将其合并进新 Rowset（对查询透明）。
  5. Segment：该行的各列值编码后存放在 某个 .dat 类列存文件 的数据块中（具体扩展名与内部格式以引擎实现为准）。

9. 小结表（对照记忆）

层级 这一层「像什么」 用 §1 示例一句话说明
Table 整张业务表 + 元数据 10 行都属于同一张逻辑表
Partition 按日期的抽屉 10 行分布在 5 个按天的分区
Tablet 抽屉里的并行格子 同一分区内按 user_id 哈希到 T0～T3
Rowset 每次导入/合并产生的一包不可变数据 分两次导入会产生 R1、R2；Compaction 可能合并为 R3
Segment 列存文件 10003 的 amount=200 与同行的其他列编码在不同列块中

10. 与 §1 对齐的建表 DDL 示例（Duplicate 明细模型）

下面给出与全文假设一致的示意建表（分区范围、桶数仅为教学用小表配置；生产请按数据量与官方建议调整）。语法以你集群版本文档为准，此处重点是「这些子句对应上文哪一层」。

CREATE TABLE dwd_order_detail (
    order_id    BIGINT,
    user_id     VARCHAR(32),
    city        VARCHAR(64),
    order_date  DATE,
    amount      DECIMAL(18,2)
)
DUPLICATE KEY(order_id)
PARTITION BY RANGE(order_date) (
    PARTITION p20260115 VALUES [('2026-01-15'), ('2026-01-16')),
    PARTITION p20260116 VALUES [('2026-01-16'), ('2026-01-17')),
    PARTITION p20260117 VALUES [('2026-01-17'), ('2026-01-18')),
    PARTITION p20260118 VALUES [('2026-01-18'), ('2026-01-19')),
    PARTITION p20260119 VALUES [('2026-01-19'), ('2026-01-20'))
)
DISTRIBUTED BY HASH(user_id) BUCKETS 4
PROPERTIES (
    "replication_num" = "3"
);

若在你使用的 StarRocks 版本上 RANGE 分区子句 与上文不完全一致，以当前版本官方建表示例为准；本文重点是各子句与 Partition / Tablet / 副本 的对应关系。

DDL 片段 对应存储层级
PARTITION BY RANGE(order_date) §4 Partition：按 order_date 落入哪个分区
DISTRIBUTED BY HASH(user_id) BUCKETS 4 §5 Tablet：每分区内 4 个桶 → 4 个 Tablet
DUPLICATE KEY(order_id) §6 读合并语义：明细保留；与 Unique/Primary 不同
replication_num §5.5：每个 Tablet 的副本数

11. 写入路径：从「业务行」到「Rowset / Segment」

一次导入（如 Stream Load）在存储侧可粗理解为：

  1. 解析行：校验类型、分区键是否命中已有分区（否则可能报「未知分区」）。
  2. 路由：对每一行计算 partition_id，再计算 bucket_id = hash(distributed_key) % num_buckets。
  3. 按 Tablet 分组：同一 Tablet 上的行进入同一批写入流水线。
  4. 生成 Rowset：该批次成功提交后，在对应 Tablet 上产生新的 Rowset（可能并行写入产生多个 Rowset，此处不展开）。
  5. 切 Segment：当内存中缓冲超过阈值，会刷盘为 Segment 文件；大导入可能对应「一个 Rowset 多个 Segment」。

用 §6.2 的两次导入对照：

  • 第一次导入后：分区 p20260115、p20260116 内的 Tablet 上各自出现属于 R1 的文件；T1 上有 10001、10003。
  • 第二次导入后：p20260117～p20260119 各 Tablet 出现 R2；T1 上再增加 10007。

导入与查询并发：导入不断追加 Rowset 会使 Tablet 上 Rowset 数量变多，读放大上升，因此依赖 Compaction 在后台把文件「收拾整齐」。

12. 读取路径：从 SQL 到 Segment 读哪些块

以查询为例：

SELECT user_id, SUM(amount)
FROM dwd_order_detail
WHERE order_date BETWEEN '2026-01-17' AND '2026-01-18'
GROUP BY user_id;

粗粒度读路径：

  1. 分区裁剪：只涉及 p20260117、p20260118 两个分区，其余三个分区整区跳过。
  2. Tablet 并行：每个分区内仍要扫 4 个 Tablet，但可并行调度到多个 BE。
  3. Rowset 枚举：每个 Tablet 上打开当前可见的 Rowset 列表（R1、R2、Compaction 后的 R3 等）。
  4. Segment 列裁剪：只读 user_id 与 amount 两列的列块，不读 city（若 SELECT 未引用）。
  5. 聚合下推（若优化器支持）：部分聚合可能在存储层或执行层完成，此处不展开版本差异。

与 §1 数据对照：该 SQL 实际会扫到的业务行是 order_id ∈ {10005,10006,10007,10008} 共 4 行；若你在执行计划中看 Scan 行数，应能与分区裁剪后的子集对应。

13. Compaction 再说明：解决什么问题、对示例数据意味着什么

13.1 解决的问题

  • 读放大：Tablet 上 Rowset 太多时，一次查询要打开很多小文件/元数据，延迟上升。
  • 空间与顺序：合并后更有利于顺序读与压缩效率。
  • 后台与前台争抢：Compaction 占用磁盘与 CPU，可能与导入/查询高峰叠加，因此生产上要关注 Compaction 指标与限流策略（以官方运维建议为准）。

13.2 对 Tablet T1 的直观含义

合并前：T1 上可能有 R1（10001、10003）与 R2（10007）两套物理文件。
合并后：可能变为 R3 一套文件，查询结果仍是三行，但扫描路径更短。

14. 表模型与存储行为差异（知识点 + 与示例的关系）

全文示例采用 Duplicate Key 明细模型，适合「订单明细一行一条、允许重复导入产生重复行」的教学场景。

模型 存储侧可建立的直觉
Duplicate 多次导入同一业务主键可能产生多行，需业务或下游去重。
Aggregate 按聚合键预聚合，列上常配合聚合函数语义，节省扫描量但限制灵活性。
Unique 唯一键语义下去重，存储与读路径需处理版本合并。
Primary Key 主键语义 + 更强更新能力，底层合并与删除标记机制更复杂（版本文档为准）。

若把 §1 表改成 Primary Key(order_id)，再重复导入 order_id=10003 且 amount 不同，则对外读到的可能是最新 amount（取决于导入版本与合并规则），这与 Duplicate 下「可能两行并存」完全不同——选型直接改变「同一主键在磁盘上如何收敛」。
