StarRocks 核心组件详解：FE 与 BE 的“大脑与四肢”分工

2026-04-17 19:59 19 min read 人阅读

StarRocks 极简架构的背后：FE 与 BE 的职责、角色与高可用设计

一、极简之美：两种组件支撑的企业级数据仓库

StarRocks 架构设计中最让人印象深刻的，不是它有多复杂，而是它有多“简单”——整个系统仅由两种核心组件组成：FE（Frontend，前端节点）和 BE（Backend，后端节点），不依赖任何外部组件（如 ZooKeeper、HDFS），部署和维护成本极低。

如果用一个比喻来理解，FE 是 StarRocks 的“大脑”，负责思考与指挥；BE 是“四肢”，负责干活和储存。 两者各司其职，通过极简架构支撑起了 PB 级数据的高性能分析。

对比维度 FE（Frontend） BE（Backend）
核心职责 元数据管理、SQL 解析、查询规划、查询调度、连接管理、身份认证 数据存储、SQL 执行、数据压缩、副本管理
比喻 大脑（思考与指挥） 四肢（干活与储存）
关键技术 Raft 协议（高可用）、CBO 优化器 列式存储、向量化执行、MPP 并行计算
节点数量建议 3 个及以上（奇数） 3 个及以上
硬件建议 ≥ 8 核 CPU，≥ 16GB 内存 ≥ 16 核 CPU，≥ 64GB 内存，SSD 磁盘

StarRocks 兼容 MySQL 协议，支持标准 SQL 语法，用户可通过各类 MySQL 客户端和常用 BI 工具无缝对接，极大降低了使用门槛。

二、FE：“智慧大脑”的元数据管理与查询协调

FE 节点是 StarRocks 集群的“总指挥部”，一条 SQL 从进入集群到变成执行计划，全程由 FE 主导。 其核心职责包括：

  • 元数据管理：管理所有数据库、表、分区、分桶、物化视图等元数据信息，每个 FE 节点在内存中存储和维护元数据的完整副本，确保所有 FE 之间服务一致。
  • SQL 解析与查询优化：接收客户端的 SQL 请求，进行词法分析、语法分析、语义分析，然后通过 CBO 优化器生成最优的分布式执行计划。
  • 查询调度：将优化后的执行计划拆分为多个 Fragment（执行片段），分发到存有相关数据的 BE 节点上执行。
  • 结果汇总：收集各 BE 节点返回的部分结果，合并后返回给客户端。

FE 的三种角色

为实现高可用，FE 节点内部通过类 Raft 协议（基于 Berkeley DB Java Edition）实现 Leader、Follower、Observer 三种角色。

角色 元数据读写权限 参与 Leader 选举 主要用途
Leader FE 读写（唯一写入口） 是（本身就是选举出的） 处理元数据变更、写入协调
Follower FE 只读 是 高可用容灾、参与选举
Observer FE 只读 否 扩展查询并发能力（不增加选主压力）

Leader 从 Follower 中自动选举产生，进行选主需要集群中有半数以上的 Follower 节点存活。 只有 Leader FE 提供元数据读写服务，Follower 和 Observer 只有读取权限，将元数据写入请求路由到 Leader 节点。Leader 更新元数据后，会通过 BDB JE 同步给 Follower 和 Observer，必须有半数以上的 Follower 节点同步成功才算作元数据写入成功。

Observer FE 主要用于扩展集群的查询并发能力，可选部署。它不参与选主，因此不会增加集群的选主压力。 在实际生产环境中，如果读并发较高，可以通过增加 Observer 节点来分担查询负载，而不影响集群的稳定性。

建议至少部署 3 个 FE 节点（如 1 个 Leader + 2 个 Follower），FE 节点个数必须为奇数（3 个或 5 个），以保证在出现网络分区时能够成功选举出 Leader。

三、BE：“强壮四肢”的数据存储与并行计算

如果说 FE 负责“出主意”，那 BE 就是负责“出力气”的那一个。BE 是 StarRocks 的计算与存储引擎，基于列式存储、向量化执行与 MPP 架构构建。

3.1 数据存储：列式存储 + 多副本机制

BE 采用列式存储方式组织数据，数据按列独立压缩与编码，查询时只读取涉及的列，大幅减少 I/O 量。支持 Zstandard、LZ4、Snappy 等多种压缩算法，结合字典编码、位图编码等技术，存储效率极高。

数据在 BE 上以 Tablet（数据分片） 为最小物理存储单元进行管理。每个 Tablet 默认以 3 副本的形式存储在不同的 BE 节点上，通过多副本机制保障数据可靠性。 当某个 BE 节点故障时，系统能够自动切换到其他副本继续提供服务。

3.2 SQL 执行：MPP 并行 + 向量化引擎

BE 节点在执行 SQL 时，充分利用了 StarRocks 的 MPP 架构优势：

  • MPP 并行计算：一条 SQL 语句首先被 FE 规划成逻辑执行单元，再按照数据的分布情况拆分成具体的物理执行单元。这些物理执行单元会在数据所在的 BE 节点上执行，实现“计算向数据移动”，避免数据跨节点传输和拷贝，从而获得极致的查询性能。
  • 向量化执行引擎：BE 在执行计算时，采用按列批量处理的方式，一次处理 1024 行数据（一个 Chunk），利用 CPU 的 SIMD 指令（如 AVX2、AVX512）实现单指令多数据并行计算，减少虚函数调用，提升计算效率。
  • 本地计算优先：查询时优先访问本地副本，计算任务调度到数据所在 BE 节点执行，最大化利用本地磁盘 I/O。

BE 节点数量直接影响集群的吞吐能力和数据容量，建议按每 TB 数据配置 1~2 个 BE 节点进行容量规划。 BE 节点数量最好为副本数的整数倍（如副本数为 3 时，BE 数量应为 3、6、9 等），以便副本均衡分布，避免数据倾斜。

四、从一条 SQL 看 FE 与 BE 的协作全流程

理解 FE 和 BE 的最佳方式，是跟踪一条 SQL 在 StarRocks 中的完整旅程：

阶段 1：请求接收与解析（FE）

  • 客户端通过 MySQL 协议提交 SQL
  • FE 接收请求，进行词法分析、语法分析、语义分析

阶段 2：查询优化（FE）

  • FE 的 CBO 优化器根据统计信息生成最优逻辑执行计划
  • RBO 规则重写阶段完成谓词下推、列裁剪等优化
  • 逻辑计划转换为分布式物理执行计划，拆分为多个 Fragment

阶段 3：任务分发（FE → BE）

  • FE 将 Fragment 分发到存有相关数据 Tablet 的 BE 节点
  • 同一个 Fragment 会在多个 BE 上启动多个实例并行执行

阶段 4：并行执行（BE）

  • 各 BE 节点独立执行分配给它的 Fragment
  • 利用向量化引擎批量处理数据，利用 SIMD 指令加速计算
  • 需要数据重组时，BE 节点间通过 GRPC 进行 Shuffle 数据传输

阶段 5：结果汇总（BE → FE）

  • 各 BE 将部分结果返回给 FE 的 Coordinator
  • FE 进行最终合并，将结果返回给客户端

整个过程充分体现了“大脑”（FE）做决策、“四肢”（BE）做执行的完美协作。

五、存算一体 vs 存算分离：BE 的两种形态

StarRocks 支持两种架构模式，BE 在这两种模式下的形态有所不同：

对比维度 存算一体架构 存算分离架构
数据存储位置 BE 本地磁盘 对象存储 / HDFS
计算节点 BE（计算 + 存储） CN（纯计算节点，取代 BE）
优势 极速查询性能，数据本地化计算，无网络传输开销 存储成本低（70-80% 降幅），计算资源弹性扩缩容
适用场景 追求最佳查询性能的场景 数据量大、追求成本优化的场景

在存算一体架构中，BE 同时承担数据存储和计算任务。将数据存储在 BE 本地使得数据可以在当前节点中计算，避免了数据传输和复制，从而提供极快的查询和分析性能。

在存算分离架构中，BE 被“计算节点（CN）”取代，CN 仅负责数据计算任务和缓存热数据，数据存储在低成本且可靠的远端存储系统中。 存算分离架构下，查询性能可提升 10 倍以上，存储成本降低 70-80%。

六、生产环境部署最佳实践

要让 FE 和 BE 稳定高效地运行，以下部署建议值得参考：

6.1 FE 部署建议

  • 节点数量：生产环境建议至少部署 3 个 FE 节点（1 Leader + 2 Follower），形成高可用集群。高并发场景下可增加至 5 个，包括增加 Observer 节点用于读扩展。
  • 硬件配置：FE 节点建议配置 ≥ 8 核 CPU，≥ 16GB 内存，数据盘 100GB 起步即可。
  • 分离部署：FE 与 BE 建议物理隔离部署，避免资源争抢。

6.2 BE 部署建议

  • 节点数量：建议至少 3 个 BE 节点，数量最好为副本数的整数倍（如副本数为 3 时，BE 数量应为 3、6、9 等），便于副本均衡分布。
  • 硬件配置：建议配置 ≥ 16 核 CPU，≥ 64GB 内存，使用 SSD 磁盘以最大化 I/O 性能。
  • 存储路径：每个 BE 节点可配置多个存储路径，分布在不同的物理磁盘上，提升 I/O 吞吐。
  • 网络要求：集群内部建议使用万兆网络（≥ 10Gbps），保证 Shuffle 过程的吞吐。

6.3 容量规划参考

数据规模 BE 节点数 FE 节点数 推荐硬件配置
< 10TB 3~5 3 16C/64GB/1TB SSD
10~50TB 8~16 3 32C/128GB/2TB SSD
> 50TB 20+ 5 64C/256GB/4TB SSD

注：BE 节点数量直接影响集群吞吐能力，建议按每 TB 数据配置 1~2 个 BE 节点进行容量规划。

七、总结

StarRocks 的 FE 和 BE 架构，用最简洁的两种组件，构建了一个高性能、高可用的分布式分析数据库。

核心要点 说明
FE 是大脑 负责元数据管理、SQL 解析优化、查询调度和结果汇总
BE 是四肢 负责数据存储（列式+多副本）和并行计算（MPP+向量化）
FE 高可用 Leader/Follower/Observer 三种角色 + Raft 协议，至少 3 节点（奇数）
BE 高性能 存算一体（极速）或存算分离（低成本），按需选择
极简架构 仅两种组件，无外部依赖，运维成本极低

如果说 CBO 优化器是 StarRocks 的“智慧”，向量化引擎是“肌肉”，MPP 架构是“骨骼”，那么 FE 和 BE 就是承载这一切的“大脑与四肢”——它们的分工协作，正是 StarRocks 能够实现极速全场景数据分析的根本原因。
