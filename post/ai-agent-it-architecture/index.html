AI Agent 时代的 IT 系统重构

2026-05-19 20:22 41 min read 人阅读

目前市面上所有的IT系统，包括但不限于财务系统、HR系统、OA系统、MES系统等，哪怕是个人APP，包括X、INS、微信、微博等全名应用，全部是以人为用户主体进行设计开发，但是随着AI的发展以及Agent的普及，未来的IT系统、APP等必定会重构。

背景：目前市面上所有的IT系统，包括但不限于财务系统、HR系统、OA系统、MES系统等，哪怕是个人APP，包括X、INS、微信、微博等全名应用，全部是以人为用户主体进行设计开发，但是随着AI的发展以及Agent的普及，未来的IT系统、APP等必定会重构。在此背景下，我想和你探讨几个问题：

1：未来的IT系统、APP是怎么样的，如何设计，用户完全是人肯定是过时的，那么用户是人和AI（Agent）还是只有Agent，人只是通过自然语言控制Agent来操作系统进行业务操作；

2：目前已经在运行的旧系统，是否可以改造以兼容Agent进行业务操作，比如开放API接口；

3：在新系统的架构下，OLAP数据库如何设计，和原来有什么区别；

4：由此进一步影响数据仓库，未来数据仓库是什么形态的；

5：你有其他什么想法或者观点，也可表达；

基于以上探讨，整理一个md格式的文件。

0. 背景

过去几十年的 IT 系统，几乎都是围绕“人”来设计的。

无论是财务系统、HR 系统、OA 系统、MES 系统，还是微信、微博、X、Instagram 等个人应用，本质上都默认：

  • 用户是人。
  • 人通过界面理解系统。
  • 人点击按钮、填写表单、查看报表。
  • 系统通过菜单、页面、流程、权限来约束人的操作。

但随着大模型、AI Agent、自然语言交互、工具调用、MCP、RPA、工作流编排等技术发展，未来的系统不会再只是“给人用的界面”，而会逐步变成“给人和 Agent 共同使用的能力平台”。

也就是说，未来系统的核心问题会从：

人如何使用系统？

转变为：

人和 Agent 如何共同完成业务目标？

甚至进一步转变为：

Agent 如何代表人、组织、岗位或流程，在可控边界内自动完成任务？

1. 未来 IT 系统与 APP 会是什么形态

1.1 用户主体会从“人”变成“人 + Agent”

未来很长一段时间内，系统用户不会直接变成“只有 Agent”，而更可能是：

人是责任主体，Agent 是执行主体或协作主体。

也就是说：

  • 人负责目标、授权、监督、确认和最终责任。
  • Agent 负责理解意图、拆解任务、调用工具、执行流程、汇总结果。
  • 系统负责提供可验证、可审计、可授权、可回滚的业务能力。

短期内，典型模式是：

人提出目标
  -> Agent 理解意图
    -> Agent 调用系统 API / 工具 / 工作流
      -> 系统执行受控业务操作
        -> Agent 汇总结果并请求确认
          -> 人确认或修正

长期看，部分高确定性、低风险、强规则的场景会逐步演进为：

人配置目标与边界
  -> Agent 持续监控
    -> Agent 自动执行
      -> 系统记录全过程
        -> 人只处理异常与关键审批

1.2 未来系统不是“页面优先”，而是“能力优先”

传统系统常见设计顺序是：

页面
  -> 表单
    -> 按钮
      -> 后端接口
        -> 数据库

Agent 时代更合理的设计顺序应该是：

业务能力
  -> 权限与约束
    -> API / Tool / Workflow
      -> 人机交互界面
      -> Agent 调用入口
      -> 数据与审计

这意味着未来系统要从“UI 驱动”转向“能力驱动”。

页面仍然存在，但页面不再是唯一入口。系统至少会有三类入口：

入口类型 使用者 典型形式
GUI 入口 人 页面、按钮、表单、报表
API / Tool 入口 Agent / 系统 REST、GraphQL、gRPC、MCP Tool、内部服务接口
自然语言入口 人 + Agent 对话框、语音、任务助手、企业 Copilot

1.3 APP 可能从“一个个独立应用”变成“能力节点”

现在的 APP 是一个独立入口，例如：

  • 打开财务系统做报销。
  • 打开 HR 系统查假期。
  • 打开 OA 系统提审批。
  • 打开 MES 系统查工单。

未来用户更可能直接对 Agent 说：

帮我把上周出差的发票整理好，生成报销单，如果金额没问题就提交审批。

Agent 背后可能同时调用：

  • 邮箱：找发票。
  • OCR：识别票据。
  • 财务系统：生成报销单。
  • OA 系统：发起审批。
  • 企业微信：通知上级。
  • 数据仓库：校验差旅政策。

此时用户感知到的不是“我打开了 5 个系统”，而是：

我把一个业务目标交给了 Agent。

因此，未来 APP 会越来越像“可被 Agent 调用的业务能力节点”，而不是孤立的人机界面。

1.4 未来系统设计的核心对象

传统系统核心对象通常是：

  • User
  • Role
  • Permission
  • Menu
  • Page
  • Form
  • Process

Agent 时代还需要新增：

  • Agent Identity：Agent 身份。
  • Delegation：人对 Agent 的授权。
  • Tool Permission：工具级权限。
  • Task Context：任务上下文。
  • Intent：意图。
  • Plan：执行计划。
  • Action：实际动作。
  • Observation：系统反馈。
  • Audit Trail：审计轨迹。
  • Human Approval：人工确认点。
  • Policy Guardrail：策略护栏。

可以把未来系统理解为：

传统业务系统
  + Agent 身份体系
  + 工具调用体系
  + 授权委托体系
  + 过程审计体系
  + 策略护栏体系
  + 自然语言交互体系

2. 旧系统是否可以改造以兼容 Agent

答案是：可以，但要分层改造，不能只“开放几个 API”就认为完成了 Agent 化。

2.1 旧系统改造的三个层级

第一层：可访问

目标是让 Agent 能够访问系统能力。

常见方式：

  • 开放 REST API。
  • 提供 GraphQL 查询接口。
  • 接入 MCP Server，把系统能力包装成 Tool。
  • 暂时通过 RPA 操作旧页面。
  • 对数据库只读开放查询视图。

这一层解决的是：

Agent 能不能用这个系统？

但这还不够。

第二层：可理解

目标是让 Agent 能够理解系统能力的语义。

旧系统的 API 往往是给程序员看的，例如：

POST /api/reim/create
GET /api/user/info
POST /api/workflow/start

Agent 更需要的是语义化描述：

工具名称：create_expense_claim
用途：根据发票、出差信息和费用明细创建报销单
输入：员工ID、发票列表、费用类别、项目编号、说明
输出：报销单ID、校验结果、下一审批人
限制：金额超过 5000 元必须人工确认

因此旧系统改造不只是开放 API，还要做：

  • API 语义描述。
  • 参数字段解释。
  • 枚举值说明。
  • 错误码解释。
  • 使用示例。
  • 业务约束说明。
  • 风险等级标识。

第三层：可治理

目标是让 Agent 的操作可控、可审计、可回滚。

旧系统原来只考虑“用户登录后点击按钮”，现在要考虑：

  • 这个 Agent 是谁创建的？
  • 它代表哪个人、哪个岗位、哪个组织？
  • 它能调用哪些工具？
  • 它能操作哪些数据范围？
  • 哪些动作需要人工确认？
  • 它的每一步操作是否可追踪？
  • 出错后如何撤销或补偿？

这一层解决的是：

Agent 用系统时，是否安全、可靠、可追责？

2.2 旧系统 Agent 化的推荐架构

不建议直接让 Agent 操作数据库，也不建议直接让 Agent 以管理员权限调用所有接口。更合理的是加一层 Agent Gateway。

Agent
  -> Agent Gateway
    -> Policy Engine
    -> Tool Registry
    -> Audit Log
    -> Legacy System API
      -> Old System

各组件职责：

组件 职责
Agent Gateway Agent 调用旧系统的统一入口
Tool Registry 注册系统可调用能力、参数、说明、风险等级
Policy Engine 判断 Agent 是否有权执行某动作
Audit Log 记录意图、参数、调用结果、人工确认
Legacy API Adapter 适配旧系统接口或页面能力

2.3 旧系统改造的优先级

不是所有功能都应该一次性开放给 Agent。

优先级建议：

  1. 只读查询类：风险低，最适合先开放。
  2. 低风险写入类：例如生成草稿、创建待确认单据。
  3. 中风险流程类：例如提交审批、发起工单，需要人工确认。
  4. 高风险操作类：付款、删除、批量变更、权限分配，必须强约束、强审计。

一个典型演进路径：

只读问答
  -> 草稿生成
    -> 人工确认后提交
      -> 低风险自动执行
        -> 高风险强审批执行

2.4 旧系统改造的关键原则

  • 不要让 Agent 绕过原有权限体系。
  • 不要让 Agent 直接操作核心数据库写入。
  • 所有写操作必须有业务语义，而不是裸 SQL。
  • 高风险动作必须有人类确认或策略确认。
  • 每次 Agent 操作都要记录“为什么做、做了什么、结果是什么”。
  • API 不只是技术接口，更是业务能力契约。

3. 新系统架构下 OLAP 数据库如何设计

Agent 时代的 OLAP 数据库不会被弱化，反而会更重要。因为 Agent 要做分析、解释、决策、推荐、监控，就需要更好的分析型数据底座。

但 OLAP 的设计重点会发生变化。

3.1 从“给报表查”变成“给人和 Agent 一起查”

传统 OLAP 主要服务：

  • BI 报表。
  • 管理驾驶舱。
  • 固定指标看板。
  • 分析师即席查询。

未来 OLAP 还要服务：

  • Agent 自动分析。
  • Agent 自动归因。
  • Agent 自动生成报告。
  • Agent 监控异常并触发动作。
  • Agent 基于数据自动推荐下一步。

因此 OLAP 不再只是“报表查询引擎”，还会成为 Agent 的“事实查询层”和“决策证据层”。

3.2 OLAP 表设计要更强调语义

传统 OLAP 表可能只强调：

  • 查询性能。
  • 分区分桶。
  • 聚合模型。
  • 指标口径。

Agent 时代还要强调：

  • 字段语义是否清楚。
  • 指标定义是否机器可读。
  • 维度枚举是否有解释。
  • 数据血缘是否可追溯。
  • 数据质量是否可被 Agent 感知。
  • 指标是否有业务解释和适用范围。

例如一个字段：

gmv_amount

仅有字段名是不够的，Agent 还需要知道：

字段名：gmv_amount
中文名：成交金额
口径：已支付订单金额，不扣除退款
单位：元
时间口径：按支付时间归属
适用范围：销售分析、渠道分析
不适用：财务收入确认
数据质量等级：A
更新频率：T+0 近实时

3.3 OLAP 需要支持“Agent 可解释查询”

未来 Agent 发起查询时，不能只返回结果，还需要返回：

  • 查询用到了哪些表。
  • 指标口径是什么。
  • 数据更新时间。
  • 是否命中缓存或物化视图。
  • 是否有数据质量告警。
  • 结果可信度如何。

一个理想返回不只是：

本月销售额：1200 万

而是：

本月销售额：1200 万
口径：已支付订单金额，不扣除退款
数据范围：2026-05-01 至 2026-05-19
数据更新时间：2026-05-19 15:30
来源表：dws_sales_order_day
质量状态：正常
注意：该指标不等同于财务确认收入

3.4 OLAP 设计会更关注三类表

第一类：业务事实表

记录真实业务事件。

示例：

  • 订单事实。
  • 审批事实。
  • 生产工单事实。
  • 员工考勤事实。
  • 财务凭证事实。

这些表仍然是数据分析的基础。

第二类：Agent 行为事实表

这是 Agent 时代新增的重要数据。

示例：

字段 含义
agent_id Agent 标识
human_user_id 代表的人
task_id 任务 ID
intent 用户意图
tool_name 调用的工具
action_type 查询/创建/提交/审批
risk_level 风险等级
approval_required 是否需要人工确认
action_result 成功/失败
latency_ms 调用耗时
created_at 动作时间

这类表用于：

  • 审计 Agent 行为。
  • 分析 Agent 效率。
  • 发现异常调用。
  • 评估 Agent 对业务流程的影响。

第三类：语义与指标元数据表

让 Agent 理解数据。

示例：

字段 含义
metric_name 指标名
metric_cn_name 中文名
business_definition 业务定义
formula 计算公式
source_table 来源表
owner 负责人
refresh_frequency 刷新频率
quality_level 质量等级
warning 使用注意事项

这类表会成为 Agent 查询数据前的重要上下文。

3.5 OLAP 权限会从“人查什么”扩展为“Agent 代表谁查什么”

传统权限：

user_id -> role -> table/row/column permission

Agent 时代需要扩展为：

agent_id
  -> delegated_user_id
  -> delegated_scope
  -> tool_permission
  -> data_permission
  -> action_risk_policy

同一个 Agent 在不同授权上下文中，能看到的数据可能不同。

例如：

  • 代表财务经理查询：可看部门费用。
  • 代表普通员工查询：只能看个人报销。
  • 代表审计 Agent 查询：可看审计范围内数据，但不能执行付款。

4. 未来数据仓库会是什么形态

未来数据仓库不会消失，而会从“报表数据仓库”升级为“语义化、可行动、可审计的数据智能底座”。

4.1 从 Data Warehouse 到 Data + Action Warehouse

传统数仓主要回答：

发生了什么？

未来数仓还要回答：

为什么发生？
接下来可能发生什么？
应该做什么？
谁或哪个 Agent 做了什么？
这个动作是否有效？

因此未来数仓不仅存业务结果，也会存：

  • 意图。
  • 动作。
  • 决策过程。
  • Agent 调用记录。
  • 人工确认记录。
  • 策略命中记录。
  • 反馈结果。

也就是从“数据仓库”演进为：

业务事实仓库
  + 指标语义仓库
  + Agent 行为仓库
  + 决策过程仓库
  + 反馈评估仓库

4.2 数仓分层会发生变化

传统分层：

ODS -> DWD -> DWS -> ADS

未来可能变成：

ODS：原始业务数据
DWD：明细事实与维度
DWS：主题汇总与指标
ADS：应用服务层
Semantic Layer：语义层 / 指标层 / 知识层
Agent Activity Layer：Agent 行为层
Decision Layer：决策过程与反馈层

其中最关键的新增层是：

Semantic Layer

负责让数据“可被人和 Agent 正确理解”。

包含：

  • 指标定义。
  • 业务术语。
  • 字段解释。
  • 维度层级。
  • 口径边界。
  • 数据血缘。
  • 数据质量。

Agent Activity Layer

负责记录 Agent 的行为。

包含：

  • Agent 接收了什么任务。
  • Agent 调用了哪些系统。
  • 调用是否成功。
  • 是否经过人工确认。
  • 是否发生异常。

Decision Layer

负责记录决策过程和结果反馈。

包含：

  • Agent 的分析过程摘要。
  • 推荐方案。
  • 人类选择。
  • 最终动作。
  • 后续业务结果。

4.3 数仓会更像“企业记忆系统”

未来数仓不只是存结构化业务数据，还会和知识库、向量库、文档库、日志系统融合。

一个企业级 Agent 需要的不只是：

  • 销售额是多少。

还需要：

  • 销售额这个指标是什么意思。
  • 这个指标和收入有什么区别。
  • 最近一次口径变更是什么时候。
  • 哪个部门负责这个指标。
  • 以前类似异常是怎么处理的。
  • 哪些制度文档约束这个动作。

因此未来数据底座可能是：

结构化数仓
  + 指标平台
  + 数据目录
  + 知识库
  + 向量数据库
  + 日志审计系统
  + 工作流记录

4.4 数仓质量会直接影响 Agent 的可靠性

过去数据质量差，主要影响报表。

未来数据质量差，会影响 Agent 的自动决策。

例如：

  • 员工权限数据错误，Agent 可能泄露敏感数据。
  • 财务指标口径错误，Agent 可能生成错误分析报告。
  • 库存数据延迟，Agent 可能错误触发采购。
  • 审批流数据不完整，Agent 可能误判流程状态。

因此未来数据治理会从“报表治理”升级为“AI 决策治理”。

数据治理的重要性会显著上升。

5. 其他观点与判断

5.1 未来系统的核心竞争力不是页面，而是可组合能力

很多系统今天的竞争力体现在：

  • 页面好不好用。
  • 流程全不全。
  • 报表多不多。

未来会越来越体现在：

  • 能力是否 API 化。
  • 语义是否清晰。
  • 权限是否可委托。
  • 操作是否可审计。
  • 是否容易被 Agent 编排。
  • 是否能和其他系统组合完成复杂任务。

5.2 Agent 不会消灭 UI，但会改变 UI 的地位

UI 不会消失。

但 UI 的地位会从“唯一入口”变成：

  • 人类确认界面。
  • 异常处理界面。
  • 过程可视化界面。
  • 权限与策略配置界面。
  • Agent 执行结果解释界面。

大量重复、规则清晰、跨系统的点击操作，会逐渐被 Agent 接管。

5.3 系统边界会从“应用边界”变成“任务边界”

传统 IT 系统按应用划分：

  • 财务系统。
  • HR 系统。
  • OA 系统。
  • MES 系统。

Agent 时代更关注任务：

  • 完成一次报销。
  • 完成一次入职。
  • 完成一次生产异常处理。
  • 完成一次销售分析。
  • 完成一次预算预警。

一个任务可能横跨多个系统，所以未来架构更需要：

  • 跨系统身份。
  • 跨系统权限。
  • 跨系统流程。
  • 跨系统数据语义。
  • 跨系统审计。

5.4 “可审计”会比“自动化”更重要

在企业系统中，Agent 最大阻力不是技术能力，而是信任。

企业会问：

  • Agent 为什么这么做？
  • 它依据了什么数据？
  • 它有没有越权？
  • 谁授权它这么做？
  • 出错后谁负责？
  • 能不能回滚？

因此，未来 Agent 系统设计的底线不是“能自动执行”，而是：

自动执行必须可解释、可审计、可约束、可回滚。

5.5 会出现新的系统角色：AgentOps

类似 DevOps、DataOps、MLOps，未来很可能出现 AgentOps。

AgentOps 负责：

  • 管理 Agent。
  • 注册工具。
  • 配置权限。
  • 监控调用。
  • 分析失败。
  • 管理提示词与策略。
  • 做安全审计。
  • 做效果评估。

Agent 不会是“装上就完事”的工具，而会成为企业 IT 治理对象。

5.6 未来企业 IT 的核心架构可能是“五层”

交互层：自然语言、GUI、语音、移动端
Agent 层：规划、记忆、工具调用、任务执行
能力层：API、Tool、Workflow、业务服务
数据层：OLTP、OLAP、数仓、向量库、知识库
治理层：权限、策略、审计、安全、质量、合规

其中治理层不是附属，而是贯穿所有层。

这个架构的核心不是让 Agent “随便操作系统”，而是让 Agent 在明确授权、语义清晰、过程可审计的框架内操作系统。

7. 总结

未来 IT 系统不会简单变成“没有人，只有 Agent”。

更合理的判断是：

人仍然是目标设定者、授权者、监督者和责任主体；Agent 会成为任务执行者、系统操作者、分析助手和流程协调者。

旧系统可以改造，但不能只做 API 开放，还需要语义化、权限委托、审计和治理。

新系统要从页面驱动转向能力驱动，从人类 UI 优先转向人和 Agent 双入口，从应用边界转向任务边界。

OLAP 和数据仓库不会消失，而会升级为 Agent 的事实依据、语义底座、决策记录和行为审计中心。

最终，企业 IT 的重点会从：

系统能不能让人点得动

转变为：

系统能不能让人和 Agent 在可信边界内共同完成业务目标

这可能是未来十年 IT 系统重构的核心方向。
