StarRocks Tablet 深度解析：从物理分片到集群均衡的存储基石

2026-04-15 20:31 29 min read 人阅读

Tablet：StarRocks 中最小的物理单元，最大的管理智慧

一、Tablet 是什么？StarRocks 的“原子存储单元”

在 StarRocks 的架构中，Tablet 是一个承上启下的核心概念。它不像“分区”那样经常被用户直接感知，也不像“列式存储”那样拥有响亮的名头，但它却是 StarRocks 存储引擎的“细胞”——所有的数据管理、副本同步、负载均衡、并行计算，最终都以 Tablet 为基本粒度展开。

1.1 定义：物理层面的水平分片

Tablet 是 StarRocks 中数据存储的基本物理单元，可以将其理解为一张表数据在物理层面的水平分片。作为 MPP 数据库，StarRocks 在建表时会按照既定规则将表数据划分为多个 Tablet，每个 Tablet 对应其中一部分数据。通过将这些 Tablet 尽可能均匀地分布在不同的 BE（或 CN）节点上，StarRocks 实现了数据的分布式存储与管理。

在 StarRocks 的存储引擎中，用户数据被水平划分为若干个数据分片（Tablet，也称作数据分桶）。每个 Tablet 包含若干数据行，各个 Tablet 之间的数据没有交集，并且在物理上是独立存储的。

1.2 Tablet 与分区、分桶的关系

理解 Tablet，必须理解它和分区（Partition）、分桶（Bucket）的关系：

  • 分区（Partition） ：将表数据按某个维度（通常是时间）进行逻辑划分，便于数据管理和查询裁剪。一个 Partition 可以视为是逻辑上最小的管理单元。
  • 分桶（Bucket） ：在每个分区内部，按指定的分桶键进行哈希分桶，将数据进一步切分为多个 Tablet。Tablet 的数量由建表时设置的分桶数决定，二者一一对应。
  • Tablet：一个 Tablet 只属于一个 Partition，而一个 Partition 包含若干个 Tablet。

建表示例：

CREATE TABLE orders (
    order_id BIGINT,
    user_id BIGINT,
    amount DECIMAL(10,2)
)
DISTRIBUTED BY HASH(user_id) BUCKETS 10;

这条建表语句会创建 10 个 Tablet，表中数据会按照 user_id 的哈希结果分布到这 10 个 Tablet 中。

总 Tablet 数 = 分区数 × 分桶数 × 副本数。 例如，一张按天分区（保留 30 天）、分桶数为 10、副本数为 3 的表，总 Tablet 数量为 30 × 10 × 3 = 900 个。

二、Tablet 的物理存储层级：从 Tablet 到 Rowset 到 Segment

Tablet 是逻辑分片概念，具体到物理文件存储，StarRocks 采用了多层级的存储架构：

Partition（分区）
    └── Tablet（分片，多个）
            └── Rowset（数据版本，多个）
                    └── Segment（物理文件，多个）

2.1 Tablet 的物理路径

对应到 BE 节点上物理文件的存储，Tablet 对应的存储路径在 be.conf 里的 storage_root_path 配置的目录下，相关的 Tablet 数据就存在 ${storage_root_path}/data 下。

具体文件目录层级如下：

shard_id/tablet_id/schema_hash/rowset_id + "_" + segment_id + ".dat"

2.2 Rowset：数据版本管理

每次数据导入都会在 Tablet 内部生成一个新的数据版本，保存在一个 Rowset 中。一个 Tablet 可能有 N（N>=0）个 Rowset，一个 Rowset 对应 M（M>=0）个实际数据文件。

Rowset 是不可变的——一旦写入完成，Rowset 的内容就不会再被修改。这种设计使得 MVCC（多版本并发控制）成为可能：写入操作生成新 Rowset，查询操作读取某个快照对应的 Rowset 集合，读写互不阻塞。

每次写入都会增加一个版本，无论是单条 INSERT 还是 Stream Load 导入几个 GB 的文件。

2.3 Segment：物理存储文件

如果一个 Rowset 数据量比较大，则拆分成多个 Segment 数据落盘。Segment 是 StarRocks 中最小的物理数据文件，以列式存储格式（Parquet 类似格式）组织，包含数据列和索引信息。

使用默认 write_buffer_size = 100MB，批量导入数据时，MemTable 写满或满足其他条件就会 Flush 到磁盘生成 Segment 文件。

2.4 元数据持久化：RocksDB

Tablet 的元数据信息（包括 Tablet ID、Schema、版本信息、Rowset 列表等）存储在 RocksDB 中，由 BE 节点的 TabletMetaManager 统一管理。

三、Tablet 副本机制：高可用的基石

3.1 副本配置与分布

StarRocks 采用多副本机制来保证数据的高可用性和一致性。每个 Tablet 默认有 3 个副本，这些副本存储在不同的 BE 节点上。

生产环境副本数建议设置为 3，BE 节点数量最好为副本数的整数倍（如 3、6、9），便于副本均衡分布。

3.2 副本同步与一致性

StarRocks 通过 Raft 一致性协议来管理副本间的数据同步，确保强一致性：

  • Leader 选举：每个 Tablet 的副本中有一个 Leader，负责处理所有读写请求
  • 日志复制：Leader 将操作日志复制到所有 Follower 副本
  • 提交确认：当多数副本确认接收后，数据才被提交
  • 状态机应用：所有副本按相同顺序应用日志，保证状态一致

3.3 副本修复机制

TabletChecker 作为常驻的后台进程，会定期检查所有 Tablet 的状态。对于非健康状态的 Tablet，将会交给 TabletScheduler 进行调度和修复。

副本修复的主要思想是先通过创建或补齐使得分片的副本数达到期望值，然后再删除多余的副本。 一个 Clone 任务就是完成从一个指定远端 BE 拷贝指定数据到指定目的端 BE 的过程。

修复场景分为两类：

  • 场景一：后端节点宕机，需要尽快修复受影响的 Tablet 副本，允许抢占式调度。
  • 场景二：新增后端节点，需要将副本迁移到新节点以平衡集群负载。

查看副本状态的常用命令：

-- 查看集群所有 Tablet 的状态
SHOW PROC '/statistic';

-- 查看表所有副本的状态
ADMIN SHOW REPLICA STATUS FROM table_xx;

-- 查看等待执行的调度任务
SHOW PROC '/cluster_balance/pending_tablets';

-- 查看正在运行的调度任务
SHOW PROC '/cluster_balance/running_tablets';

四、Compaction：Tablet 内部的“大扫除”

4.1 为什么需要 Compaction？

每次数据摄入都会在 Tablet 内部生成一个新的 Rowset。随着历史数据版本的累积，需要合并的文件数量增多，这将显著降低查询效率。

Compaction 是将不同版本的数据文件合并成大文件的过程，旨在减少系统中小文件数量、从而提升查询效率。

在合并过程中，旧版本的数据会被消除，例如 UPDATE 操作标记为删除的行在 Base Compaction 时会被物理删除，释放存储空间。

4.2 Compaction 类型

Compaction 分为两种类型：

类型 说明 特点
Cumulative Compaction 对增量 Rowset 进行合并 开销较小，频繁执行
Base Compaction 将增量数据合并到基线版本（start version = 0），同时物理删除标记为删除的行 开销较大，按需执行

4.3 Compaction Score 评分机制

StarRocks 通过 Compaction Score 来评估 Tablet 需要合并的紧急程度。Score 越高，表示需要合并的紧迫程度越高。

Score 的计算考虑了以下因素：

  • Tablet 模型：明细模型因为没有计算逻辑，Rowset 越大 Compaction 优先级越高
  • Rowsets 大小：小文件越多 Score 越高
  • Segment 数量：文件碎片化程度
  • Tablet Version 数量：版本累积越多 Score 越高
  • Rowsets Level：基于 Size-Tiered 策略的分层

4.4 Size-Tiered 策略

StarRocks 新版本中使用的 Size-Tiered 策略 将 Rowset 大小分层，相邻两层之间大小差距 5 倍，每一层都会保留 1 个 Rowset。默认配置下 7 个层级大小依次为 128K、640K、3M、15M、78M、390M、1.9G。

这种策略有效控制了 Compaction 的执行频率和合并粒度，避免单次 Compaction 耗时过长。

4.5 存算分离架构下的 Compaction

在存算分离架构下，Compaction 调度由 FE 发起，CN 节点执行。FE 按照 Partition 为单位来发起 Compaction 任务，Compaction 会生成一个新版本，走完整的写入、提交、发布流程。

Compaction 完成后，Compaction Service 将合并后的数据文件异步推送到业务所属计算组的节点上，避免因缓存失效（Cache Miss）导致的对象存储访问，保障查询性能不受影响。

五、Tablet 与负载均衡：数据分布的“自调节”

5.1 负载均衡的触发

StarRocks 在扩容或缩容以后，数据会自动进行均衡。BE 之间默认有 10% 的均衡差，超过这个阈值就会触发 Tablet 调度。

均衡的主要思想：对于某些分片，首先在低负载的节点上创建一个副本，然后再删除这些分片在高负载节点上的副本。

5.2 均衡调度参数

常用的均衡相关参数配置：

-- BE 之间均衡差阈值，默认 0.1（10%）
tablet_sched_balance_load_score_threshold = 0.1

-- 磁盘安全阈值，默认 0.8
tablet_sched_balance_load_disk_safe_threshold = 0.8

-- 最大同时调度的 Tablet 数量
tablet_sched_max_scheduling_tablets = 15000

-- 最大同时迁移的 Tablet 数量
tablet_sched_max_balancing_tablets = 6000

在业务低峰期，可以适当调大这些参数以加速均衡：

ADMIN SET FRONTEND CONFIG ("tablet_sched_max_scheduling_tablets"="15000");
ADMIN SET FRONTEND CONFIG ("tablet_sched_disable_balance"="false");
ADMIN SET FRONTEND CONFIG ("tablet_sched_slot_num_per_path" = "30");
ADMIN SET FRONTEND CONFIG ("tablet_sched_max_balancing_tablets" = "6000");

5.3 数据倾斜的预防

Tablet 分布不均会带来严重后果：Tablet 较少的节点 CPU 和内存的资源利用率都比其他节点更低，不能充分利用资源，扩容时效果不佳。

避免数据倾斜的核心在于分桶键的设计：

  • 选择高基数的列作为分桶键
  • 如果有唯一 ID，建议使用唯一 ID 分桶
  • 如果单列基数不够高，可以使用多列作为分桶键

六、Tablet 与查询执行：并行计算的发动机

6.1 Tablet 级别的并行扫描

在查询执行过程中，StarRocks 可以并行扫描多个 Tablet，以提升整体查询效率。以一张包含 10 个 Tablet 的表为例，查询引擎可以将扫描与计算任务拆分到多个并发执行单元中，分别处理不同 Tablet 上的数据，最终再对结果进行汇总。

基于 Tablet 的并行处理机制，StarRocks 能够更充分地利用多核 CPU 和分布式集群资源，从而提升查询性能。

6.2 Tablet 内并行 Scan

从 StarRocks 3.3.0 版本开始，引入了 Tablet 内并行 Scan 能力。它将原来大的串行 I/O 拆解为多个小 I/O，并利用线程池并行处理这些 I/O 任务，显著提升 I/O 效率。

这一优化带来的核心优势是：Tablet 可以管理的数据量变大了。 以前为了提升并行度，一般推荐单个 Tablet 1GB 大小较为合适，而有了 Tablet 内并行 Scan 技术，用户可以不再受这个限制（例如测试中单 Tablet 数据量就约为 5GB）。

更少的 Tablet 数量一方面可以降低 FE 上的内存使用，另一方面也可以显著降低写入时访问对象存储的 API 调用次数。

性能测试结果：

  • 对于 Cache Disabled 场景，开启了并行 Scan 等优化后，查询延迟显著下降 5-20 倍
  • Cache Disabled 时性能与存算一体的差距缩小到整体只有约 2 倍
  • Cache Enabled 场景下，与存算一体都开启并行 Scan 优化后的性能几乎一致

6.3 查询中 Tablet 的剪裁

当查询包含分桶键的过滤条件时，StarRocks 可以进行 Tablet 剪裁（Bucket Pruning） ，只扫描满足条件的 Tablet，大幅降低查询延迟和 CPU 开销。

例如，对 customer_id 进行哈希分桶的表，查询条件 WHERE customer_id = 123 可以直接定位到单个 Tablet，实现点查级别的响应速度。

七、Tablet 设计最佳实践

7.1 Tablet 大小规划

Tablet 的最佳大小可以参考以下评估：

数据类型 推荐 Tablet 大小
原始非压缩数据（CSV 等） 1 GB ~ 10 GB
Parquet/ORC 格式数据 1 GB 左右
压缩后数据 100 MB ~ 1 GB

每个 Tablet 的压缩后大小建议控制在 100MB-1GB 左右（2.4 及以后版本，建议控制在压缩前 10G 左右）。

7.2 分桶数计算

分桶数 = 预估数据总量 ÷ 期望的单个 Tablet 大小。

也可以参考公式：BE 数量 × CPU core / 2 来设置 Bucket 数量，以充分发挥集群的计算性能。

7.3 FE 元数据容量规划

FE 内存根据查询 QPS 与表数量、Tablet 数量来决定。一般 32GB 内存可支持 10 万以下 Tablet 数量。Tablet 数量在 10 万以上建议选用 64GB 内存规格。

7.4 分桶键选择原则

  • 选择高基数的列作为分桶键，避免 Bucket 之间数据倾斜
  • 基数的典型要求是分桶键的基数应为 BE 节点数量的 1000 倍以上
  • 如果有唯一 ID，则建议使用唯一 ID 分桶
  • 如果单列基数不够高，可以使用多列作为分桶键，但通常不要使用太多列

7.5 避免“Too Many Tablet Versions”错误

StarRocks 中每个 Tablet 有版本数限制（默认 1000 个）。当数据加载过于频繁或并发度过高时，会导致版本数快速增加，触发 too many tablet versions 错误。

应对策略：

  • 减少并发度：降低同时运行的导入任务数量
  • 适当调大批处理大小，减少写入频率
  • 关注 Compaction Score，确保 Compaction 跟得上写入速度

八、Tablet 运维常用命令

-- 查看集群所有 Tablet 的状态
SHOW PROC '/statistic';

-- 查看某个数据库下 Tablet 的状态
SHOW PROC '/statistic/DbId';

-- 查看某个表下的 Tablet
SHOW TABLET FROM table_xx LIMIT 10;

-- 查看某个 Tablet 的详细信息
SHOW TABLET 232081 \G

-- 查看表所有副本的状态
ADMIN SHOW REPLICA STATUS FROM table_xx;

-- 查看表所有副本的分布情况
ADMIN SHOW REPLICA DISTRIBUTION FROM table_xx;

-- 查看等待执行的调度任务
SHOW PROC '/cluster_balance/pending_tablets';

-- 查看正在运行的调度任务
SHOW PROC '/cluster_balance/running_tablets';

-- 查看已结束的调度任务
SHOW PROC '/cluster_balance/history_tablets';

九、总结

Tablet 是 StarRocks 存储引擎的“原子单元”，它串联起了从数据分布、版本管理、副本同步到查询执行的每一个环节：

  1. 物理分片：Tablet 是表数据在物理层面的水平分片，由分桶数决定，与分区共同构成数据的二维划分体系。
  2. 版本管理：每次写入生成新的 Rowset，多个 Rowset 构成 Tablet，通过 Compaction 合并小文件、消除冗余版本。
  3. 副本机制：每个 Tablet 默认 3 副本，通过 Raft 协议保证强一致性，通过 Clone 任务完成故障修复。
  4. 负载均衡：Tablet 是节点间数据迁移和调度的基本单位，通过 TabletScheduler 实现自动均衡。
  5. 查询执行：查询引擎并行扫描多个 Tablet，Tablet 内并行 Scan 进一步提升了 I/O 效率。
  6. 设计关键：合理的 Tablet 大小（1GB-10GB）和分桶键选择是性能优化的核心。

如果说分区是 StarRocks 的“逻辑管理单元”，那么 Tablet 就是 StarRocks 的“物理执行单元” ——理解 Tablet 的设计哲学和实践方法，是掌握 StarRocks 存储与查询性能的关键。
