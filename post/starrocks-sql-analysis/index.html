SQL 执行前搞这么多分析，StarRocks 不怕慢吗？

2026-04-18 21:09 13 min read 人阅读

如果你刚接触 StarRocks 这类 MPP 分析型数据库，可能会有一个很自然的疑问：“一条 SQL 提交后，要先做语法解析、语义分析、RBO 规则重写、CBO 成本优化……这么多步骤都是实时执行的，难道不会拖慢查询速度吗？为什么不能提前做掉一部分？”

这个疑问非常合理。毕竟在我们的直觉里，“少做点事”似乎应该更快。但 StarRocks 的设计哲学恰恰相反——它选择在每次查询执行前，都做一次精密而短暂的“思考”，而且这样做反而让整体性能实现了数量级的飞跃。
本文将为你揭示这背后的设计逻辑：StarRocks 如何在“实时决策”与“极致性能”之间找到平衡。

一、实时优化：用毫秒级“思考”换秒级“行动”

首先明确一点：解析、语义分析、RBO、CBO 等步骤，确实是在收到 SQL 查询后实时、动态执行的。

StarRocks 的 Frontend（FE）节点在收到查询请求后，会立即启动一个完整的优化流水线：

SQL 文本 → 词法/语法解析 → 语义分析 → RBO 规则重写 → CBO 成本优化 → 生成物理执行计划

这个过程通常耗时在 几十毫秒到几百毫秒 之间。

你可能会问：几百毫秒也是时间，积少成多不会影响高并发下的响应速度吗？

关键点在于 “思考”与“执行”的时间比例。

StarRocks 定位是分析型数据仓库（OLAP），处理的往往是 TB/PB 级别的复杂聚合查询。这类查询的原始执行时间可能在 数十秒到数分钟 甚至更长。而经过 CBO 优化后，执行时间可能被压缩到 几秒 以内。

用一个直观的对比来说明：

场景 优化时间 执行时间 总耗时
不进行优化（暴力执行） 0ms 120,000ms（2分钟） 120,000ms
进行完整优化 200ms 3,000ms（3秒） 3,200ms

用 200ms 的“思考”换取 117 秒的“行动”节省，这笔账怎么算都划算。

这就是 StarRocks 效率哲学的第一层：用微小的前置成本，撬动巨大的后端收益。

二、不止于“思考”：三大机制让优化器越用越快

如果 StarRocks 只是每次查询都从头优化一遍，确实会在高并发场景下产生累积开销。但它的设计远不止于此——通过以下三大机制，StarRocks 将“实时优化”的代价降到了最低。

2.1 计划缓存：为重复查询“抄近路”

在实际业务中，很多查询是高度重复的——比如报表系统每天执行的固定 SQL，或者应用程序中使用 Prepared Statement 的参数化查询。

对于这类查询，StarRocks 会启用 执行计划缓存：

  • 首次查询：完整走完优化流程，生成执行计划并缓存

  • 后续相同查询：直接命中缓存，跳过解析、分析、优化的全部步骤，立即分发执行

这就像学生考试：第一次遇到难题需要仔细思考解题步骤，但如果下次考到一模一样的题，就可以直接调用记忆中的解法。

计划缓存将高频查询的优化开销一次性摊销，使得重复查询的响应时间几乎等同于纯执行时间。

2.2 查询反馈：让 CBO “自我学习”

CBO 依赖统计信息来估算成本，但估算总有偏差。比如，它可能高估了某次 Join 的数据量，导致选了一个次优计划。

StarRocks 引入了 Query Feedback（查询反馈） 机制来解决这个问题：

  • 查询执行后，系统会收集实际运行数据（实际行数、实际字节数、实际执行时间等）

  • 这些反馈信息被记录并用于修正后续相似查询的成本估算

  • 随着运行时间增加，CBO 的成本模型会越来越贴近真实数据分布，估算越来越准

这相当于 CBO 拥有“自我进化”能力——它不是每次都从零开始“瞎猜”，而是从历史执行中学习经验。用得越久，决策越精准，犯错的概率越低。

2.3 物化视图：为复杂查询“预先算好”

对于特别复杂且频繁执行的查询（例如按天汇总的多表关联），实时计算无论如何都会有较高开销。StarRocks 提供了 物化视图 这一“作弊器”：

  • 你可以预先定义一个物化视图，本质上是将复杂查询的结果提前计算并物理存储

  • 当用户提交的查询能够被物化视图覆盖时，CBO 会自动改写查询，直接从物化视图中读取预计算结果

  • 原来需要扫描数十亿行原始数据的查询，变成了读取几百行预聚合数据

物化视图将“实时计算”转化为“实时读取”，相当于把最繁重的工作提前做完了。对于这类查询，优化器甚至不需要做复杂的成本估算——因为“直接读物化视图”的成本一定是最低的。

三、一个生动的类比：导航软件的路径规划

如果上述机制还不够直观，我们可以用一个生活中的例子来理解：

StarRocks 的优化器就像你手机里的导航软件。

类比项 导航软件 StarRocks 优化器
规划行为 输入目的地后，计算多条路线并推荐最快的一条 收到 SQL 后，枚举多种执行计划并选择成本最低的一条
耗时 规划需要几秒钟 优化需要几十到几百毫秒
收益 避开拥堵，节省数十分钟行车时间 选择最优 Join 顺序和分布策略，节省数十秒查询时间
缓存机制 常去地点一键导航，无需重复输入 计划缓存让重复 SQL 直接执行
学习机制 根据历史路况数据优化路线推荐 查询反馈让成本模型越来越准
预计算机制 实时路况数据的预先采集与处理 物化视图预先计算复杂查询结果

你会因为导航规划需要花几秒钟，就放弃使用导航、凭感觉开车吗？显然不会。因为你知道，这几秒钟的投入能让你避开拥堵，总到达时间反而更短。

StarRocks 的优化器正是这个道理——它不是在“浪费时间”，而是在为你的查询“规划最优路径”。

四、总耗时公式：理解 StarRocks 的性能逻辑

最后，我们用一个简单的公式来总结 StarRocks 查询的性能逻辑：

总耗时 = 规划时间（优化器耗时） + 执行时间（引擎计算耗时）

不同的优化策略，对这个公式的影响截然不同：

  • 不优化的查询：规划时间 ≈ 0，但执行时间可能极长（因选择了低效的执行路径）

  • 仅 RBO 优化：规划时间较短，执行时间有所缩短（但无法应对复杂场景）

  • RBO + CBO 完整优化：规划时间稍长（~100ms），但执行时间被压到最低

  • 命中计划缓存：规划时间 ≈ 0，执行时间同样最优（因为用的是之前选好的最优计划）

  • 命中物化视图：规划时间极短，执行时间也极短（直接读预计算结果）

StarRocks 的设计目标，就是让绝大多数查询落在后两种场景中——通过缓存和预计算，让“最优路径”成为默认路径。

五、总结

回到最初的问题：“StarRocks 在 SQL 执行时实时做语义分析、RBO、CBO，不会影响效率吗？”

答案是：不仅不会，反而这是 StarRocks 实现高性能的核心秘诀。

  • 短查询：依靠计划缓存，优化开销被一次性摊销，响应时间几乎等同于纯执行

  • 长查询：用毫秒级的优化投入，换取数十秒甚至分钟级的执行时间压缩，性价比极高

  • 重复查询：依靠物化视图和计划缓存，直接绕过复杂计算

StarRocks 的优化器不是一个“拖慢查询”的累赘，而是一个精明的投资决策者——它精准地知道什么时候该花时间“思考”，什么时候该直接“行动”，以及什么时候该调用“历史经验”。

这种设计哲学，正是现代分析型数据库能够在海量数据下依然保持高性能交互体验的根本原因。
