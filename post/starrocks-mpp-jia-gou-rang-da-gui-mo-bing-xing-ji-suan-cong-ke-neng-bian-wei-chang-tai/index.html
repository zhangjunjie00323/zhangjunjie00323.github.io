StarRocks MPP 架构：让大规模并行计算从“可能”变为“常态”

2026-04-16 22:01 26 min read -- 人阅读

MPP 架构详解（Massively Parallel Processing，大规模并行处理）

前言

想象一下，你面对的是一个拥有 100 亿条记录的订单表，需要统计每个区域每天的销售额。如果只用一台服务器，即使是最顶级的硬件，扫描这么多数据也可能需要几分钟甚至更长。

但如果你有 100 台服务器，每台只需要处理 1 亿条记录呢？理论上，时间可以缩短到原来的百分之一。

问题的关键就在于：如何让这 100 台服务器真正“一起干活”，而不是各自为战？

这就是 MPP（Massively Parallel Processing，大规模并行处理）架构要解决的核心问题。StarRocks 从设计之初就将 MPP 作为其查询执行的基础框架，让多节点并行计算成为查询处理的“默认模式”而非“特殊优化”。本文将深入解析 MPP 架构的核心理念、StarRocks 的具体实现，以及它如何让海量数据的交互式分析成为可能。

一、并行计算的理想与现实

在深入 MPP 之前，我们先理解一个更基础的问题：为什么让多台机器一起干活这件事，并不像听起来那么简单？

假设我们要计算 100 亿条销售记录的总额。用一台机器来做，就是老老实实地一条一条累加。用 100 台机器来做，流程大概是这样的：

  1. 把 100 亿条记录分给 100 台机器，每台 1 亿条
  2. 每台机器算出自己的“部分总和”
  3. 把 100 个部分总和收集起来，再加一次

看起来很简单对吧？但魔鬼藏在细节里：

  • 数据怎么分？ 如果 100 亿条记录分散在 100 台机器上，但某台机器上的数据特别多（数据倾斜），那它就会成为瓶颈——其他 99 台都算完了，还得等它一个。
  • 中间结果怎么传递？ 如果查询涉及 JOIN 或 GROUP BY，不同机器上的数据需要重新组织。比如按“地区”分组，就需要把所有“北京”的数据发到同一台机器上做最终聚合。这个数据“洗牌”（Shuffle）的过程，会产生大量的网络传输。
  • 谁来协调？ 谁负责把查询拆成 100 份？谁负责收集 100 份结果？如果这个“指挥官”自己成了瓶颈，那加再多“士兵”也没用。

传统的 Hadoop MapReduce 采用了一种“重容错”的思路：每个阶段（Map → Shuffle → Reduce）之间都会把中间结果写到磁盘上，虽然稳定可靠，但频繁的磁盘 I/O 和阶段间的等待使得延迟很难降到秒级以下——更适合小时级、天级的批量 ETL 任务，而非交互式分析。

MPP 架构选择了另一条路：用更紧密的协作和更少的“停顿”，换取极致的并行效率。

二、MPP 架构的核心理念：Share-Nothing + 完全并行

MPP 架构的核心思想可以概括为三个关键词：无共享（Share-Nothing）、分而治之、流水线并行。

2.1 无共享架构（Share-Nothing）

在 MPP 架构中，集群中的每个节点都有自己独立的 CPU、内存和本地磁盘，节点之间不共享任何计算或存储资源。这与传统的共享存储架构（如 Oracle RAC）形成鲜明对比：

架构类型 资源模型 优点 缺点
完全共享（SMP） 所有 CPU 共享内存和磁盘 单机性能强，编程简单 扩展性受限，硬件成本高
共享存储 多节点共享磁盘阵列 高可用性好 I/O 竞争激烈，扩展时性能下降
无共享（MPP） 每个节点拥有独立资源 水平扩展能力强，无资源竞争 网络通信成为关键瓶颈

在 Share-Nothing 架构下，StarRocks 集群天然支持水平扩展——增加节点数量，系统的计算能力和存储容量可以近线性增长。更重要的是，由于每个节点只处理自己本地存储的数据分片，大量计算可以在数据所在的节点上直接完成，避免了不必要的网络传输。

2.2 分而治之：查询的拆解艺术

MPP 架构处理查询的基本模式是：将一个大的查询任务分解为多个小任务，分发到各个节点并行执行，最后汇总结果。

这个过程就像建筑工地的包工头：接到一个“盖一栋楼”的任务，他不会自己一个人干，而是把它拆成“打地基”、“砌墙”、“装门窗”等子任务，分给不同的小组同时进行，最后各组汇报进度。

在 StarRocks 中，“包工头”是 FE（Frontend）节点，“工人”是 BE（Backend）节点。一个完整的查询处理流程如下：

text

客户端 → FE（接收 SQL）
          ↓
    【1. 解析与优化】
    词法分析 → 语义分析 → RBO 重写 → CBO 成本优化
          ↓
    【2. 生成分布式执行计划】
    将优化后的计划拆解为多个 Fragment（执行片段）
          ↓
    【3. 分发 Fragment】
    将 Fragment 调度到存有相关数据的 BE 节点上
          ↓
    【4. 并行执行】
    每个 BE 节点独立执行分配给它的 Fragment
    （同一个 Fragment 会在多个 BE 上启动多个实例并行执行）
          ↓
    【5. 数据 Shuffle】
    需要跨节点数据重组时，BE 节点间通过网络交换数据
          ↓
    【6. 结果汇总】
    最终结果汇总到 FE，返回客户端

2.3 流水线并行：消除“空等”

MPP 架构的另一个关键特性是流水线并行：数据在一个节点上处理完后，可以立即流式传输到下一个节点，而不是等整个阶段全部完成后再传递。

这与 MapReduce 的“阶段式”执行形成了鲜明对比。MapReduce 必须等所有 Map 任务完成后，才能开始 Shuffle 和 Reduce 阶段——就像工厂里必须等所有零件加工完毕，才能开始组装。而 MPP 的流水线模式更像是汽车生产线：零件一边加工一边进入下一道工序，整体完工时间大大缩短。

正是这种“流式”的设计，让 MPP 架构能够实现秒级甚至亚秒级的查询响应，满足交互式分析的需求。

三、StarRocks MPP 架构深度解析

理解了 MPP 的核心理念，我们来看看 StarRocks 是如何具体实现的。

3.1 架构组件：FE 与 BE 的分工

StarRocks 集群由两种核心节点组成：

组件 角色 核心职责
FE（Frontend） “大脑” 元数据管理、SQL 解析、查询优化、执行计划生成与调度、结果汇总
BE（Backend） “四肢” 数据存储、SQL 执行（计算任务）、数据导入

FE 节点内部还细分为三种角色：Leader（负责写入协调）、Follower（参与选举与容灾）、Observer（仅用于读扩展）。推荐在生产环境部署至少 3 个 FE 节点以实现高可用。

BE 节点是计算和存储的主体，支持多副本机制（默认三副本）确保数据可靠性。每个 BE 节点可以配置多个存储路径，建议使用 SSD 磁盘以最大化 I/O 性能。

值得注意的是，StarRocks 集群不依赖任何外部组件（如 ZooKeeper、HDFS 等），架构十分精简。FE 和 BE 之间通过高性能的 BRPC 框架进行通信，确保节点间数据传输的低延迟和高吞吐。

3.2 数据组织：从 Tablet 到分区

要理解 MPP 如何实现并行计算，必须先理解 StarRocks 的数据组织方式。

Tablet 是 StarRocks 中数据存储的基本物理单元，可以理解为一张表数据在物理层面的水平分片。StarRocks 在建表时会按照既定规则将表数据划分为多个 Tablet，每个 Tablet 对应其中一部分数据。通过将这些 Tablet 尽可能均匀地分布在不同的 BE 节点上，StarRocks 实现了数据的分布式存储与管理。

Tablet 的数量由建表时设置的分桶（Bucket）数决定。例如：

CREATE TABLE orders (
    order_id BIGINT,
    user_id BIGINT,
    amount DECIMAL(10,2)
) DISTRIBUTED BY HASH(user_id) BUCKETS 10;

这条建表语句会创建 10 个 Tablet，表中数据会按照 user_id 的哈希结果分布到这 10 个 Tablet 中。

Tablet 作为基础物理单元，在 StarRocks 中承担着多重角色：

  • 实现并行计算：查询引擎可以并行扫描多个 Tablet，每个 Tablet 由一个独立的执行单元处理
  • 作为数据管理的基本单位：Compaction、负载均衡等操作都以 Tablet 为粒度
  • 支持系统水平扩展：扩容时通过 Tablet 级别的调度，将数据迁移到新节点

合理设计分桶策略与分桶数量，对于系统性能和资源利用率都有重要影响。一般建议每个 BE 节点承载 10–20 个桶，分桶过少会导致并行度不足，过多则会增加元数据管理开销。

3.3 执行计划的拆解：Fragment 与 Pipeline

当一条查询到达 FE 后，优化器生成的分布式执行计划会被拆解为多个 Fragment（执行片段）。一个 Fragment 代表一个完整的单机执行片段，多个 Fragment 组合在一起，构成一个完整的分布式执行计划。

以一条简单的聚合查询为例：

SELECT region, SUM(amount) FROM sales GROUP BY region;

这条查询会被拆解为两阶段的 Fragment：

  1. 第一阶段 Fragment（本地聚合） ：在每个 BE 节点上并行执行，扫描本地的 sales 表数据分片，计算每个 region 的部分 SUM
  2. 第二阶段 Fragment（全局聚合） ：所有 BE 将本地聚合的结果按 region 通过网络 Shuffle 到指定节点，完成最终的合并

两个 Fragment 之间通过 Exchange 算子 连接，负责节点间的数据传输。同一个 Fragment 会在多个 BE 上启动多个实例并行执行，实例数量与数据分布的 Tablet 数量相关，从而实现“数据在哪里，计算就在哪里”。

在此基础上，StarRocks 还引入了 Pipeline 执行引擎，将 Fragment 进一步拆解为更细粒度的 Pipeline，每个 Pipeline 由多个算子（Operator）串联而成。这种设计可以更精细地利用多核 CPU，实现算子级别的并行执行。

3.4 数据交换：Shuffle 的代价与优化

在分布式计算中，数据不可能永远只在自己的节点上处理。当查询涉及 JOIN 或 GROUP BY 一个非分区键的字段时，不同节点上的数据需要重新组织——这就是 Shuffle 过程。

Shuffle 是 MPP 架构中最关键也最容易成为瓶颈的环节。StarRocks 通过以下策略来降低 Shuffle 的开销：

策略一：本地预聚合

在进行全局聚合之前，每个 BE 节点先对自己本地的数据进行一次预聚合，大幅减少需要通过网络传输的数据量。例如，一个 BE 节点上有 100 万条北京地区的订单，预聚合后只需要传输一条记录（北京，总额），数据量压缩了 100 万倍。

策略二：Colocate Join（协同分布 Join）

如果两张表在建表时就使用相同的分桶键和分桶数，那么相同 Join Key 的数据会天然地分布在同一个 BE 节点上。此时执行 JOIN 完全不需要 Shuffle，直接进行本地 Join，实现零网络开销。这正是我们在 CBO 文章中提到的 Colocate Join 策略的底层原理。

策略三：Broadcast Join

当一张表很小（例如维度表）而另一张表很大时，与其 Shuffle 大表，不如把小表广播到所有 BE 节点。每个 BE 节点拿到小表的完整副本后，直接与本地的数据分片进行 JOIN，避免了大规模的数据传输。

四、MPP 架构的优势与局限

4.1 核心优势

1. 高性能与低延迟

MPP 架构将查询任务并行分散到多个节点上，充分利用集群的整体计算资源。配合流水线式的数据传递，StarRocks 能够在秒级甚至亚秒级完成对 PB 级数据的复杂分析查询。

2. 水平扩展能力

Share-Nothing 架构天然支持水平扩展。增加节点数量，系统的计算能力和存储容量可以近线性增长，无需对应用层做任何修改。

3. 架构精简

StarRocks 的 MPP 框架不依赖任何外部组件，FE 和 BE 节点通过内置的 Raft 协议和 BRPC 框架实现高可用和高效通信，部署和运维成本极低。

4. 数据本地化计算

“计算向数据移动”是 MPP 架构的核心原则。查询任务被调度到数据所在的节点上执行，最大限度地减少了网络传输开销。

4.2 需要注意的局限

1. 集群规模调整复杂度

MPP 架构在增减节点时，数据的重分布可能需要一定时间。部分系统的扩缩容操作甚至需要停机，需要提前做好规划。

2. 数据倾斜的敏感性

如果数据分布不均匀（例如某个分桶键的值占比过高），会导致部分节点负载过高，拖慢整个查询的响应时间。合理选择分桶键和分桶数是避免数据倾斜的关键。

3. 不适合高并发事务型场景

MPP 架构主要针对分析型查询（OLAP）优化，对于大量短小的事务型操作（OLTP），传统的关系型数据库仍然是更合适的选择。

4. 网络依赖性强

由于节点间需要频繁进行数据交换，MPP 架构对网络带宽和延迟有较高要求。建议集群内部使用 10Gbps 以上的网络，以保证 Shuffle 过程不成为瓶颈。

五、MPP 架构 vs. 传统批处理引擎

为了更直观地理解 MPP 架构的特点，这里将其与大数据领域常见的批处理引擎（如 Hive on MR、Spark）做一个对比：

对比维度 MPP 架构（StarRocks） 传统批处理引擎（MapReduce/Spark）
执行模式 流水线并行，阶段间流式传输 阶段式执行，阶段间需等待全部完成
中间结果 内存/网络流式传递，避免磁盘 I/O 通常落盘（MapReduce 必须落盘，Spark 可选）
延迟 毫秒级 ~ 秒级 分钟级 ~ 小时级
容错机制 基于副本和 Raft，依赖重试 基于中间结果落盘，任务级重试
适用场景 交互式分析、实时报表、Ad-hoc 查询 大规模 ETL、批处理作业
典型代表 StarRocks、Doris、Greenplum、ClickHouse Hive on MR、Spark SQL

简而言之：MPP 追求“快”，批处理追求“稳”。 StarRocks 的定位就是前者——用极致的并行效率，让海量数据的交互式分析成为可能。

六、MPP 架构的最佳实践

要让 StarRocks 的 MPP 架构发挥最佳性能，以下几点值得关注：

1. 合理设计分桶策略

  • 分桶数应与 BE 节点数匹配，建议每个 BE 承载 10–20 个 Tablet
  • 选择区分度高的列作为分桶键，避免数据倾斜
  • 高频 JOIN 的表使用相同的分桶键和分桶数（Colocate 表）

2. 充分利用数据本地化

  • 尽量让查询的过滤条件和分桶键相关，减少需要 Shuffle 的数据量
  • 对于大表 JOIN 小表的场景，小表可以配置为 Broadcast Join

3. 关注网络配置

  • 集群内部建议使用万兆网络，保证 Shuffle 过程的吞吐
  • BE 节点尽量部署在同一机房或同一可用区，降低网络延迟

4. 监控并行度

  • 通过 EXPLAIN 查看执行计划中的 Fragment 数量和分布
  • 关注是否存在数据倾斜导致的“长尾”任务

七、总结

MPP 架构是 StarRocks 实现高性能查询的“肌肉系统”。它通过以下机制让大规模并行计算从“可能”变为“常态”：

  1. Share-Nothing 架构：每个节点独立管理资源，天然支持水平扩展
  2. 分而治之：查询被拆解为多个 Fragment，分发到数据所在节点并行执行
  3. 流水线并行：数据流式传输，消除阶段间的等待
  4. 智能数据分布：通过 Tablet 和分桶策略，实现“计算向数据移动”
  5. 多层次优化：Colocate Join、Broadcast Join、本地预聚合等策略降低 Shuffle 开销

如果说 CBO 是 StarRocks 的“智慧大脑”，智能物化视图是“缓存外挂”，那么 MPP 架构就是 StarRocks 的“筋与骨”——它让查询引擎真正具备了驾驭成百上千台服务器协同作战的能力。

在数据量爆炸式增长的今天，单机性能的提升速度已经远远跟不上数据的膨胀速度。MPP 架构代表的“横向扩展”思路，正是应对这一挑战的核心答案。
