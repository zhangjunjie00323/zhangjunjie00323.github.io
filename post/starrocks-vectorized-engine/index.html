StarRocks 向量化引擎：如何让 CPU 在数据分析中“火力全开”

2026-04-16 22:19 18 min read 人阅读

在数据仓库的性能优化中，有一个经常被忽视的“隐形战场”——CPU 执行效率。

前言

你可能会觉得，既然用了 MPP 架构、列式存储、CBO 优化器，查询应该已经够快了。但实际情况是：如果数据在 CPU 内部的处理方式不够高效，那么再好的架构、再优的执行计划，也会在“最后一公里”大打折扣。

传统数据库大多采用按行处理的模式，就像工厂里一次只加工一个零件，虽然逻辑简单，但 CPU 的大部分资源都在等待和调度中浪费掉了。StarRocks 的向量化引擎则采用了一种完全不同的思路——按列批量处理，一次处理一批数据，让现代 CPU 的流水线和 SIMD 指令集真正“吃饱”。

本文将带你深入 StarRocks 向量化执行引擎的内部，理解它的工作原理、性能优势，以及它如何与列式存储、MPP 架构形成合力，将查询性能推向极致。

一、从“按行”到“按列批”：计算模式的范式转移

1.1 传统按行处理的痛点

在传统的关系型数据库中，数据在内存中通常以行的形式组织，查询执行器也是一行一行地处理数据。例如，计算 SELECT a + b FROM table，执行流程大致是这样的：

for each row in table:
    result = row.a + row.b
    output(result)

这种方式在 OLTP 场景下没问题，但在分析型查询（OLAP）中会暴露出严重缺陷：

  • 函数调用开销巨大：每一行数据都要经过一次完整的表达式计算流程，虚函数调用、类型判断等开销累积惊人
  • CPU 流水线频繁中断：逐行处理意味着大量分支预测失败和指令流水线停顿
  • SIMD 指令无法利用：现代 CPU 支持单指令多数据（SIMD），可以一次对多个数据执行相同操作，但逐行模式天然无法利用这一能力

简单来说，按行处理让 CPU 大部分时间都在“等待”和“切换”，而不是在“计算”。

1.2 向量化执行的核心思想

向量化执行的核心思想可以概括为八个字：按列组织，批量处理。

它不是一次处理一行，而是一次处理一批数据（通常几千行，称为一个“向量”或“Chunk”）。同一批数据中，来自同一列的数据在内存中连续存放，计算时采用紧密循环对整批数据依次应用相同的运算。

以同样的 a + b 为例，向量化执行流程大致如下：

for each chunk of 4096 rows:
    column_a = chunk.get_column('a')
    column_b = chunk.get_column('b')
    result_column = column_a + column_b  // 批量加法
    output(result_column)

这样做带来几个直接好处：

  1. 函数调用次数减少数千倍：一次函数调用处理整个批次，而非每行一次
  2. 内存访问连续化：列式数据在内存中紧密排列，CPU 缓存命中率大幅提高
  3. SIMD 加速成为可能：编译器可以自动将循环优化为 SIMD 指令，一次处理 4/8/16 个数据

如果把按行处理比作“一件一件快递人工分拣”，向量化执行就是“把快递装进集装箱，用传送带整箱运输”——单位时间内的吞吐量有天壤之别。

二、StarRocks 向量化引擎的技术实现

StarRocks 从 2.0 版本开始全面启用向量化执行引擎，所有内置函数和表达式都经过向量化改造。其实现机制可以从以下几个层面来理解：

2.1 数据组织：列式存储是向量化的基础

向量化执行的第一步，是数据本身以列式形式组织。StarRocks 的存储层本身就是列式存储，数据从磁盘读取时就已经按列加载到内存。这为向量化计算提供了天然适配的数据格式：

  • 每一列的数据在内存中是连续数组，不需要行转列的开销
  • 相同列的数据类型一致，处理逻辑可以高度统一
  • 压缩后的列数据可以在内存中直接参与部分计算（如过滤）

当数据从存储层进入执行层时，StarRocks 使用 Chunk 作为数据传递的基本单元。一个 Chunk 包含一批行（默认 4096 行）的若干个列，每个列的数据存储为一个 Column 对象（实质是一个连续数组）。计算逻辑直接在 Column 上展开批量操作。

2.2 批量计算：紧密循环与编译器优化

向量化执行的关键在于消除循环内部的复杂逻辑。在 StarRocks 的表达式计算中，对于一个 a + b * c 这样的表达式，向量化版本会这样处理：

  1. 获取列 a、b、c 的数据指针
  2. 构造一个紧密循环，遍历 Chunk 中的每一行
  3. 循环体只包含最基础的算术运算，没有任何分支判断
// 伪代码示例：向量化加法
for (int i = 0; i < chunk_size; ++i) {
    result_data[i] = a_data[i] + b_data[i];
}

这种简洁的循环结构使得现代 C++ 编译器可以轻松进行自动向量化——将循环展开，映射到 AVX2、AVX512 等 SIMD 指令集。StarRocks 还内置了对特定 CPU 指令集的运行时探测和调优，优先使用当前 CPU 支持的最高效 SIMD 版本。

此外，StarRocks 还应用了 CodeGen 技术，在运行时将表达式编译为动态库并加载执行，进一步消除解释执行的开销，让向量化循环的性能逼近手写汇编。

2.3 算子向量化：从 Scan 到 Join 的全链路加速

向量化不仅仅是表达式计算的优化，而是贯穿整个查询执行链路：

算子类型 传统实现 向量化实现
Scan 逐行读取，行转列 直接读取列式数据块，零拷贝传递给上游
Filter 逐行判断，生成选中行集 批量比较，生成 Bitmap 或索引数组
Aggregate 逐行更新哈希表 批量计算哈希键，向量化聚合函数（如 SUM/AVG）
Join 逐行探测哈希表 批量探测，向量化哈希表查找
Sort 逐行比较 批量比较 + 列式排序算法

以 Filter 算子为例：对于 WHERE age > 30 这样的过滤条件，向量化执行会一次性对 Chunk 中的整列 age 数据进行比较，生成一个 uint8_t 的选择数组（Selection Vector），指示哪些行满足条件。后续算子只需要根据这个选择数组处理选中行即可，无需重复判断。

对于 Hash Join：向量化引擎会在探测阶段一次取出一批探测键，批量计算哈希值，然后批量在哈希表中查找匹配项。这减少了哈希表访问带来的随机内存访问开销，并使得探测阶段也能受益于 SIMD。

2.4 与列式存储的完美协同

向量化引擎与列式存储的结合，产生了“1+1>2”的效果：

  • 延迟物化：查询可能只涉及表的 3 列，但存储层有 50 列。列式存储天然支持只读需要的列，向量化引擎也只在需要的列上分配内存和计算，I/O 和内存开销最小化。
  • 压缩数据直接运算：对于使用字典编码或 RLE 编码的列，向量化引擎可以直接在压缩数据上进行部分运算（如等值比较），无需完全解压，大幅节省 CPU。
  • 谓词下推的向量化执行：谓词下推到存储层后，存储层可以利用向量化方式批量过滤，只返回符合条件的数据 Chunk。

三、性能收益：为什么向量化能带来数倍提升？

向量化执行带来的性能提升，来自于几个方面的叠加效应：

1. 减少指令开销

按行处理时，每行都要走一遍完整的表达式求值流程，涉及大量虚函数调用、类型分派等。向量化将这些开销分摊到数千行上，单行的平均指令数大幅下降。

2. 提高 CPU 缓存效率

列式数据在内存中连续存放，当 CPU 从内存读取一个缓存行（通常 64 字节）时，里面装的是同一列的多个数据，而不是一行中多个无关的列。这意味着每次缓存加载都是有效数据，缓存命中率显著提升。

3. 充分利用 SIMD 并行

现代 CPU 的 AVX2 指令一次可以处理 8 个 32 位整数或 4 个 64 位双精度浮点数。向量化循环使得编译器能够自动生成 SIMD 指令，理论峰值计算能力可提升 4-8 倍。

4. 减少分支预测失败

按行处理时，数据类型、空值判断等都会引入分支，CPU 分支预测失败会导致流水线停顿。向量化将分支移到循环外部（“循环外分支”），循环内部是无分支的密集计算，流水线可以高效运转。

根据 StarRocks 官方测试，在 TPC-H 标准测试集上，向量化引擎相较于非向量化模式有 3-10 倍的性能提升，尤其在计算密集型查询（如复杂表达式、多表 JOIN）中收益最为明显。

四、向量化执行 vs. 代码生成

在数据库领域，提升 CPU 执行效率的另一条技术路线是 代码生成（CodeGen） ，典型代表是 Apache Spark 的 Tungsten 引擎和 HyPer 数据库。它通过在运行时将 SQL 表达式编译成 JVM 字节码或机器码，消除解释执行的开销。

StarRocks 实际上采用了 向量化 + CodeGen 融合的策略：

  • 向量化负责数据层面的批量处理和 SIMD 加速
  • CodeGen 负责消除表达式计算中的虚函数调用和类型分派

两者并非替代关系，而是互补关系。向量化解决的是“数据流”的效率问题，CodeGen 解决的是“指令流”的效率问题。StarRocks 在默认启用向量化的同时，也对关键表达式路径应用了 CodeGen 优化，以追求极致性能。

五、最佳实践：让向量化发挥最大效力

要让 StarRocks 的向量化引擎充分施展拳脚，以下几点值得注意：

1. 升级到较新版本

StarRocks 从 2.0 开始默认开启向量化引擎，但后续版本持续对更多算子和函数进行了向量化改造。建议使用 3.0 及以上版本以获得最全面的向量化支持。

2. 避免使用未向量化的函数

极少数 UDF 或复杂表达式可能尚未向量化。可以通过 EXPLAIN 查看执行计划，若发现某些算子出现大量虚函数调用，可考虑改写 SQL 或联系社区确认支持情况。

3. 合理设置 Chunk 大小

StarRocks 默认的 Chunk 大小为 4096 行。对于宽表（列数多），可适当减小 Chunk 大小以降低内存占用；对于窄表（列数少），可适当增大以提升批量处理效率。一般默认值即可满足多数场景。

4. 善用列式存储特性

向量化引擎与列式存储深度绑定，设计表时应优先考虑列式存储模型（StarRocks 默认为列存）。避免使用行存表进行大规模分析查询，否则向量化收益会大打折扣。

六、总结

向量化执行引擎是 StarRocks 实现高性能数据分析的“动力心脏”。它通过以下核心机制让 CPU 在处理海量数据时不再“摸鱼”：

  1. 批量处理：一次处理一个 Chunk（数千行），分摊函数调用和类型分派开销
  2. 列式数据流：数据按列连续存储，CPU 缓存效率最大化
  3. SIMD 加速：利用现代 CPU 的向量指令，实现单指令多数据并行计算
  4. 全链路向量化：从 Scan、Filter 到 Join、Aggregate，所有算子都支持批量执行

如果说 MPP 架构是 StarRocks 的“骨骼”，CBO 优化器是“大脑”，智能物化视图是“缓存外挂”，那么向量化引擎就是 StarRocks 的“肌肉”——它让每一次计算都充满力量，将硬件性能发挥到极致。

在数据分析进入“秒级交互”时代的今天，向量化执行已经从“锦上添花的优化”变成了“不可或缺的基础能力”。理解它的原理，不仅能帮你更好地调优查询，也能让你在面对执行计划时，明白那些毫秒级的性能提升究竟从何而来。
