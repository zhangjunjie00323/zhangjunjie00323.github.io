哈希：从数学原理到工程应用完全指南

2026-05-25 22:22 51 min read 人阅读

哈希函数、哈希表、密码学哈希：三重视角下的全面解读

前言：一个简单问题引发的革命

假设你是一家图书馆的管理员，有 1000 万本书。每当读者还书时，你需要以最快速度找到这本书应该放回哪个书架。你会怎么做？

最笨的办法是扫遍所有书架，但这显然不可接受。更好的办法是：给每本书一个编号（哈希值），编号直接告诉你它在哪个书架。查找时间从 O(n) 降到 O(1)。

这就是哈希的核心思想——用一个函数将任意数据映射到一个固定大小的"摘要"或"地址"，使得后续查找、比较、索引变得极快。

从你输入密码登录网站的那一刻，到比特币矿工争相寻找一个满足条件的 nonce；从 Git 给每个 commit 生成的 SHA-1 指纹，到 Nginx 用一致性哈希做负载均衡——哈希无处不在，却又常常被当作"黑盒"对待。

本文的目标是拆开这个黑盒，让你不仅会用哈希，更理解它为何有效、何时失效、以及如何选择正确的哈希工具。

第一部分：基础篇 —— 理解哈希

第1章 什么是哈希函数？

1.1 核心定义

哈希函数（Hash Function）是一个数学函数，它将任意长度的输入映射为固定长度的输出：

h: {0,1}* → {0,1}^n

输入：任意长度的比特串（可以是一句话、一个文件、一部电影）
输出：固定长度的比特串（例如 256 位）

这个输出称为哈希值（Hash Value）、摘要（Digest）或指纹（Fingerprint）。

用一个简单的例子：

h("Hello World") = 7f83b1657ff1fc53b92dc18148a1d65dfc2d4b1fa3d677284addd200126d9069
h("Hello WorlD") = 7a4b8b5a2a6091e4303e87b0ccfb9b2b0dabc6e2ad7a75cb3b7d06a3cd8e9fde

仅仅改了一个字母的大小写，输出完全不同。这种"差之毫厘，谬以千里"的特性被称为雪崩效应（Avalanche Effect）。

1.2 哈希函数的三个基本要求

一个"好的"哈希函数（用于哈希表等数据结构）应满足：

（1）确定性：相同的输入永远产生相同的输出。这是最基本的要求——不能今天 h("cat")=42，明天 h("cat")=37。

（2）高效性：计算哈希值应该很快。如果算一次哈希需要 1 秒，那用它来加速查找就本末倒置了。

（3）均匀性：哈希值应该均匀分布在整个输出空间中。如果哈希函数把大多数 key 都映射到同一个值，哈希表就退化为链表。

1.3 哈希不是什么

初学者常有一个误解：哈希是加密。不，哈希不是加密。

加密 哈希
输入 明文（任意长度） 任意数据（任意长度）
输出 密文（同长度） 摘要（固定长度）
可逆性 可逆（有密钥就能解密） 不可逆（单向函数）
密钥 需要密钥 无需密钥
用途 保密通信 数据完整性、索引、签名

加密像是把信装进信封——有钥匙就能打开。哈希像是把信烧成灰——你得到了一堆灰烬，但不可能把灰烬还原成信。

第2章 哈希函数的设计原理

2.1 一个简单的哈希函数：模哈希

最简单的哈希函数：把输入看作整数，对某个数取模。

h(key) = key mod m

例如 m = 10：

h(42) = 42 mod 10 = 2
h(17) = 17 mod 10 = 7
h(100) = 100 mod 10 = 0

这个函数的优点是极快，缺点是如果 key 的分布有规律（比如都是 10 的倍数），所有哈希值都会是 0——灾难性的碰撞。

选择 m 的技巧：m 应该选素数，且不要选 2 的幂。因为当 m = 2^k 时，h(key) 只依赖于 key 的低 k 位二进制——信息利用率低，分布可能不均匀。

2.2 乘法哈希法

另一种经典方法：将 key 乘以一个常数 A（0 < A < 1），取小数部分，再乘以 m。

h(key) = floor( m × ( (key × A) mod 1 ) )

Donald Knuth 建议取 A ≈ (√5 - 1) / 2 ≈ 0.6180339887...（黄金比的倒数）。

乘法哈希的优点是 m 的选择不像除法哈希那样敏感，即使 m 是 2 的幂也能工作良好。

2.3 多项式哈希（用于字符串）

如何对字符串做哈希？把每个字符看作数字（如 ASCII 码），用多项式计算：

h(s) = (s[0] × a^{n-1} + s[1] × a^{n-2} + ... + s[n-1] × a⁰) mod m

这里 a 是一个精心选择的基数（通常取素数，如 31、131、137 等）。

以 a=31, m=10⁹+7 为例：
h("abc") = (97 × 31² + 98 × 31 + 99) mod (10⁹+7)
= (93217 + 3038 + 99) mod (10⁹+7)
= 96354

Java 的 String.hashCode() 使用的就是类似方法（a=31），而 Python 则使用 SipHash 算法。

为什么选 31？ 因为 31 = 2⁵ - 1，乘法可以用移位优化：31 × x = (x << 5) - x，比通用乘法快。

2.4 滚动哈希：字符串查找的利器

Rabin-Karp 算法使用了一种巧妙的"滚动"技巧。当你有一个字符串的哈希值，想要计算移除第一个字符、加入下一个字符后的哈希值时，不需要从头重算：

h(s[1..n]) = (h(s[0..n-1]) - s[0] × a^{n-1}) × a + s[n]  (mod m)

这使得你可以在 O(1) 时间内"滑动窗口"计算哈希值，将子串匹配从 O(mn) 降到平均 O(m+n)。这是滚动哈希在字符串算法中的力量。

第3章 哈希碰撞：不可避免的命运

3.1 抽屉原理决定了碰撞必然存在

输入空间是无限的，输出空间是有限的。 根据鸽巢原理（抽屉原理），必然存在两个不同的输入产生相同的哈希值。这就是哈希碰撞。

SHA-256 的输出是 256 位，共有 2²⁵⁶ 种可能的哈希值。这个数字大到不可思议（大约是宇宙中原子总数的 10²⁴ 倍），但输入的可能性是无限的，所以碰撞在数学上必然存在。

问题不是"会不会碰撞"，而是"碰撞的概率有多大"以及"碰撞了怎么办"。

3.2 生日悖论与生日攻击

一个反直觉的概率事实：在一个只有 23 人的房间里，两个人生日相同的概率超过 50%。

对于哈希函数，这个看似矛盾的现象有着直接的含义。如果你从一个有 N 种可能输出的哈希函数中随机选取输入，期望在选取大约 √N 个输入后就会发现第一次碰撞。

对于 64 位哈希（N=2⁶⁴），√N = 2³² ≈ 43 亿。也就是说，只需要约 43 亿次尝试，就有 50% 的概率找到碰撞——在今天的硬件上，这只需几秒钟。

启示：使用 64 位哈希来防碰撞是不安全的。至少需要 128 位（如 MD5）或更高（如 SHA-256）。

3.3 负载因子与碰撞概率

在哈希表中，定义负载因子 α = n/m（n 为已存储的元素数，m 为哈希表大小）。

当 α = 0.5 时（表半满），新插入一个元素时发生碰撞的概率约为 1 - e^{-α} ≈ 39%。当 α = 0.9 时，碰撞概率高达 59%。

这就是为什么大多数哈希表在负载因子超过一定阈值（通常是 0.75）时会自动扩容。

第二部分：哈希表 —— 计算机科学中最实用的数据结构

第4章 哈希表的结构与操作

4.1 哈希表的基本原理

哈希表（Hash Table / HashMap）将键（key）通过哈希函数映射到数组的索引（index），从而实现 O(1) 的平均查找、插入、删除。

          哈希函数          数组索引
"Alice"  ────→ h("Alice")=3  ──→ [0][1][2][3][4][5][6]
"Bob"    ────→ h("Bob")=5    ──→          ↑   ↑
"Carol"  ────→ h("Carol")=3  ──→  ← 碰撞！Carol 和 Alice 映射到同一位置

4.2 碰撞解决策略之一：链地址法

最直观的方法：每个数组槽位不是一个元素，而是一条链表（或现代实现中的红黑树）。

[0] → null
[1] → null
[2] → null
[3] → [Alice | value] → [Carol | value] → null   ← 链表
[4] → null
[5] → [Bob | value] → null
[6] → null

查找时，先算哈希找到槽位，再遍历该槽位的链表比较 key。

Java 的 HashMap 在 JDK 8+ 中使用链地址法，但当链表长度超过 8 时，将链表转换为红黑树，将最坏情况从 O(n) 优化到 O(log n)。这种混合策略非常精妙。

负载因子与扩容：Java HashMap 默认负载因子为 0.75。当元素数超过容量 × 0.75 时，自动扩容为原来的 2 倍，并重新哈希所有元素（rehashing）。一次 rehashing 的代价是 O(n)，但均摊到每次插入上只有 O(1)。

4.3 碰撞解决策略之二：开放地址法

不建链表，而是当碰撞发生时，按某种规则找到下一个空槽位。

（1）线性探测：依次检查 h(key) + 1，h(key) + 2，h(key) + 3...

插入 42，h(42)=3，位置 3 已有元素
检查位置 4 → 空，插入位置 4

线性探测的问题：一次聚集（Primary Clustering）。被占用的连续区域会像滚雪球一样越来越大，因为任何哈希到该区域的元素都必须探测到区域末尾。

（2）二次探测：依次检查 h(key) + 1²，h(key) + 2²，h(key) + 3²...

这减轻了一次聚集，但引入了二次聚集：哈希到同一位置的元素会有相同的探测序列。

（3）双重哈希：用第二个哈希函数决定步长。

探测序列：h(key) + i × h'(key)，i = 1, 2, 3...

这是开放地址法中效果最好的方法——不同的 key 即使哈希到同一位置，也有不同的探测序列。但需要精心设计 h' 使其永远不会返回 0。

删除的麻烦：在开放地址法中，不能简单地把删除的位置设为空，否则会"断开"探测链。标准做法是标记为"已删除"（tombstone），查找时遇到 tombstone 继续探测，插入时可以重用。

Redis 的选择：Redis 的字典使用渐进式 rehashing——在扩容时不一次性完成所有迁移，而是每次操作时逐步迁移一部分，从而避免了大规模 rehashing 造成的延迟抖动。这对于追求低延迟的 Redis 至关重要。

4.4 链地址法 vs 开放地址法

维度 链地址法 开放地址法
负载因子 可以 > 1 必须 < 1
缓存性能 较差（链表跳跃） 较好（连续内存）
删除操作 简单 复杂（需要 tombstone）
实现复杂度 较低 较高
代表 Java HashMap, Go map Python dict, C++ unordered_map 的部分实现

Python 的 dict 使用开放地址法（实际上是一种优化的双重哈希变体），因为在 CPython 的实现中，连续内存访问的缓存优势显著。

第5章 深入哈希表：从理论到工业实现

5.1 Java HashMap 源码级分析

Java 的 HashMap 是工业级哈希表实现的教科书。几个关键设计：

（1）容量始终是 2 的幂

为什么？因为 Java 不是用取模 h % n 来计算索引，而是用位运算：

index = h & (n - 1)   // 当 n 是 2 的幂时，这等价于 h % n，但快得多

但前面说过 m 选 2 的幂会导致低位依赖问题——Java 通过扰动函数来解决：

static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

将 hashCode 的高 16 位与低 16 位异或，让高位信息也参与到索引计算中。这个设计精巧地平衡了速度与均匀性。

（2）树化阈值：链表长度 ≥ 8 且总容量 ≥ 64 时，链表转为红黑树。

为什么是 8？在均匀分布且负载因子为 0.75 的前提下，单个槽位中元素数量服从 λ≈0.5 的泊松分布。P(元素数>8) ≈ 0.00000006——概率极低。如果真的发生了，很可能是有人故意构造碰撞攻击（哈希洪水攻击），此时树化是必要的防御。

（3）哈希洪水攻击：攻击者精心构造大量 key，使得它们全部哈希到同一个槽位，将 O(1) 的哈希表退化为 O(n) 的链表。Java 8 的树化是应对这种攻击的关键防御。

5.2 一致性哈希：分布式系统的基石

普通的哈希在扩容时需要重新分配几乎所有 key。在一个有 N 个节点的分布式缓存集群中，增加或移除一个节点会导致大约 N/(N+1) 比例的 key 被重新映射——这对缓存命中率是灾难性的。

一致性哈希（Consistent Hashing）解决了这个问题：

将哈希空间组织成一个环（0 到 2³²-1）。节点和 key 都映射到这个环上。每个 key 存储在沿环顺时针方向遇到的第一个节点上。

     [0]
      |
  [2³²-1]───[节点 B]───[节点 C]
      |                    |
  [节点 A]────────[key]───┘
  
  key 沿顺时针找到节点 C → 存储在 C 上

当新增节点 D 插入到 A 和 C 之间时，只有本应映射到 C 的部分 key（位于 D 和 C 之间的）需要迁移到 D，其余不受影响。

虚拟节点：为了防止节点数少时负载不均（节点在环上的位置是随机的，可能出现大的"空隙"），每个物理节点映射为多个虚拟节点，均匀分布在环上。

一致性哈希被广泛应用于 Memcached、Redis Cluster、Amazon DynamoDB、Apache Cassandra 等系统中。

第三部分：密码学哈希 —— 当安全成为首要考量

第6章 密码学哈希的特殊要求

6.1 三个核心安全性质

密码学哈希函数不仅需要一般哈希函数的性质，还需满足三个更强的要求：

（1）原像抵抗（Preimage Resistance）——单向性

给定哈希值 h，在计算上不可能找到任何输入 x 使得 H(x) = h。

如果你知道密码的哈希值是 ef92b77...，你无法反推出密码是什么。这就是为什么即使数据库泄漏了哈希值，攻击者也不能直接"解密"出密码。

（2）第二原像抵抗（Second Preimage Resistance）——弱碰撞抵抗

给定输入 x₁，在计算上不可能找到不同的 x₂ 使得 H(x₁) = H(x₂)。

这防止攻击者找到与合法文件具有相同哈希值的恶意文件。

（3）碰撞抵抗（Collision Resistance）——强碰撞抵抗

在计算上不可能找到任意两个不同的 x₁ 和 x₂ 使得 H(x₁) = H(x₂)。

这是三者中最强的要求。SHA-1 的碰撞抵抗已被打破（Google 在 2017 年发布了首个 SHA-1 碰撞），因此 SHA-1 不再被认为是安全的。

6.2 常用密码学哈希算法

算法 输出长度 状态 备注
MD5 128 位 已破解，不安全 切勿用于安全目的
SHA-1 160 位 已破解，不安全 2017 年公布首个碰撞
SHA-256 256 位 安全 比特币使用的算法
SHA-3 可变 安全 与 SHA-2 结构不同，2015 年标准
BLAKE2 可变 安全 比 MD5 快但比 SHA-3 安全
BLAKE3 可变 安全 极快，2020 年发布

6.3 生日攻击的数学

前面提到的生日悖论在密码学中有着致命的含义。

对于 n 位输出的哈希函数，暴力碰撞攻击的复杂度是 O(2^(n/2))，而不是直觉中的 O(2^n)。这意味着：

  • MD5（128 位）：碰撞需要约 2⁶⁴ ≈ 1.8×10¹⁹ 次计算——在过去可行
  • SHA-1（160 位）：碰撞需要约 2⁸⁰ 次——Google 的 SHAttered 攻击约需要 2⁶³ 次（利用弱点）
  • SHA-256（256 位）：碰撞需要约 2¹²⁸ 次——即使全宇宙的算力也无法在可预见的未来实现

这就是为什么 SHA-256 仍被认为是安全的。

第7章 密码存储：哈希最重要的安全应用

7.1 绝不要明文存储密码

这是安全的第一戒律。如果数据库以明文存储密码，一次数据泄露就等于所有用户的密码全部暴露。因为用户倾向于在不同网站使用相同密码，后果是灾难性的。

7.2 加盐哈希

最简单的做法是用 SHA-256 哈希后存储。但这面临彩虹表攻击——攻击者预先计算了常见密码（如 "password", "123456", "iloveyou"）的 SHA-256 值，只需查表就能还原。

盐（Salt） 的引入改变了这一切。盐是一个随机生成的字符串，与密码一起哈希：

存储格式：salt || hash(salt + password)

密码 "password123"，盐 "a3K9mW"
存储值："a3K9mW" + SHA-256("a3K9mWpassword123")

即使两个用户用了相同的密码，因为盐不同，存储的哈希值也完全不同。彩虹表也失效了——攻击者需要为每一个盐值单独建立彩虹表。

7.3 慢哈希：让暴力破解变得昂贵

SHA-256 被设计为快，这对于文件校验是优点，对于密码存储却是弱点——攻击者可以每秒尝试数十亿个密码。

bcrypt、scrypt、Argon2 被设计为故意慢。它们通过大量迭代或消耗大量内存，将单次哈希的时间控制在约 0.1 秒——对正常用户完全无感，但将攻击者的暴力破解速度降低了数个数量级。

Argon2 是 2015 年密码哈希竞赛的冠军，被广泛认为是当前最安全的密码哈希算法。它有三种模式：

  • Argon2d：抗 GPU 攻击
  • Argon2i：抗侧信道攻击
  • Argon2id：混合模式（推荐）

第四部分：应用全景 —— 哈希的广阔疆域

第8章 数据完整性与去重

8.1 文件校验

当你下载一个文件时，网站通常会提供一个哈希值（如 SHA-256）。你下载后计算本地文件的哈希值，与官方公布的值比对——如果不一致，说明文件在传输中被篡改或损坏。

$ sha256sum ubuntu.iso
a1b2c3d4... ubuntu.iso   ← 与官网公布的值比对

这不是为了防止恶意攻击（攻击者可以同时替换下载文件和公布的哈希值），而是为了检测传输错误和简单篡改。结合数字签名后，才能防御真正的攻击。

8.2 内容寻址：Git 与 IPFS

Git 的核心设计是一个内容可寻址的文件系统。每个文件、每个目录树、每个 commit 都由其内容的 SHA-1 哈希值来标识。

commit a1b2c3d4...  ← 这个 40 字符的十六进制字符串就是哈希值

这赋予了 Git 一个强大的特性：相同内容永远有相同的哈希值。如果两个不同的文件计算出了相同的 SHA-1（碰撞），Git 就会出现严重问题——这正是为什么 Git 正在从 SHA-1 迁移到 SHA-256。

IPFS（星际文件系统）将内容寻址推向极致。文件不是通过"在哪里"来定位（如 /home/user/photo.jpg），而是通过"是什么"来定位（如 /ipfs/QmXgZ...——这串字符就是文件内容的哈希值）。这颠覆了传统的基于位置的网络寻址方式。

8.3 去重系统

备份系统、Dropbox 等云存储服务大量使用哈希来做去重（deduplication）。上传文件时，先计算文件的哈希，在服务器上查找是否存在相同哈希的文件——如果存在，就不需要真上传，直接创建一个引用即可。这对存储成本的节省是巨大的。

第9章 区块链中的哈希

9.1 区块链接：不可篡改的链

区块链的"链"就是用哈希连接的。每个区块包含：

  • 本区块的数据（交易记录）
  • 上一个区块的哈希值
  • 一个随机数（nonce）
区块 #1                区块 #2                区块 #3
┌─────────────┐       ┌─────────────┐       ┌─────────────┐
│ 数据        │       │ 数据        │       │ 数据        │
│ prev: 0...  │  ←──  │ prev: H(#1) │  ←──  │ prev: H(#2) │
│ nonce: 123  │       │ nonce: 456  │       │ nonce: 789  │
└─────────────┘       └─────────────┘       └─────────────┘

如果有人想篡改区块 #1 的数据，它的哈希值会改变，导致区块 #2 的 "prev" 字段失效，进而改变区块 #2 的哈希值，以此类推——整条链全部失效。这就是区块链不可篡改的数学原理。

9.2 工作量证明与挖矿

比特币的挖矿过程本质上就是寻找哈希碰撞的特殊形式：

找到一个 nonce，使得 SHA-256(SHA-256(区块头)) < 目标值

目标值越小，挖矿难度越大。每个矿工都在暴力尝试不同的 nonce，希望自己的哈希值小于当前难度目标。第一个找到的矿工获得区块奖励。

这个看似"浪费"的计算过程恰恰是比特币安全的基石——要重写历史，你需要掌握 51% 以上的全网算力。

9.3 Merkle 树：高效验证

区块链不会把每笔交易的哈希单独存储。它使用 Merkle 树（Hash Tree）：

           Root Hash
          /         \
    Hash(A+B)     Hash(C+D)
     /    \        /    \
  Hash(A) Hash(B) Hash(C) Hash(D)
    |       |       |       |
  交易A   交易B   交易C   交易D

每层节点的哈希值由子节点的哈希值拼接后计算。只需 O(log n) 个哈希值就能证明某笔交易存在于区块中——这就是Merkle 证明，轻节点的核心技术。

第10章 布隆过滤器：概率的优雅

10.1 基本思想

布隆过滤器（Bloom Filter）是一个空间效率极高的概率数据结构，用于判断"某个元素是否可能在集合中"：

  • 如果布隆过滤器说"没有"——那一定是真没有
  • 如果布隆过滤器说"可能有"——那只有一定的概率对（存在误判）

这种单向确定性（one-sided error）使它非常有用。

10.2 工作原理

布隆过滤器使用 k 个独立的哈希函数和一个长度为 m 的位数组（初始全 0）。

插入元素 x：

对于每个哈希函数 h_i：
    bits[h_i(x) mod m] = 1

查询元素 y：

对于每个哈希函数 h_i：
    如果 bits[h_i(y) mod m] == 0：
        返回 "一定不存在"
返回 "可能存在"

为什么有误判？ 查询 y 时，y 的 k 个位置可能被其他元素设为了 1，导致我们错误地认为 y 在集合中。但这个错误只会发生在"存在"的方向——"不存在"的结论永远是正确的。

10.3 数学分析

对于 m 位的位数组、k 个哈希函数、n 个已插入元素：

误判率 p ≈ (1 - e{-kn/m})k

最优哈希函数数 k_opt = (m/n) × ln 2 ≈ 0.693 × m/n

此时误判率 p_opt ≈ (0.6185)^(m/n)

如果使用 10 位/元素（m/n=10），k=7，误判率约为 1%。仅用 1.25 字节/元素！

10.4 实际应用

  • Google Chrome 曾用布隆过滤器来检测恶意 URL
  • 数据库（Cassandra, HBase, PostgreSQL） 用布隆过滤器避免不必要的磁盘读取
  • 比特币 使用布隆过滤器实现 SPV（Simplified Payment Verification）轻钱包
  • Medium 用布隆过滤器来判断用户是否已经读过某篇文章

第11章 哈希的高级应用

11.1 哈希在机器学习中：特征哈希

当你的数据集有数百万个分类特征（如用户 ID、URL），独热编码会导致维度爆炸。特征哈希（Feature Hashing / Hashing Trick）将任意数量的分类特征映射到固定维度的向量中：

h("user_12345") mod d = 向量中的索引

碰撞会导致不同特征映射到同一维度，但在高维空间中（d 足够大时），碰撞的影响通常可以忽略。这在 NLP 和推荐系统中非常实用。

11.2 局部敏感哈希（LSH）：让相似的输入有相似的哈希

普通哈希函数要求相似输入产生完全不同的输出（雪崩效应）。但局部敏感哈希（Locality Sensitive Hashing）恰恰相反——它要让相似的输入产生相似（或相同）的哈希值。

这用于：

  • 图像相似度搜索（Google 图片搜索）
  • 音频指纹识别（Shazam 识别歌曲）
  • 近似最近邻搜索（高维向量检索）
  • 文档去重与抄袭检测

SimHash 是其中最著名的算法之一，Google 用它在数十亿网页中检测近似重复。

11.3 流式算法中的哈希

HyperLogLog：使用哈希来估算集合中去重元素的数量（基数估计），只需要约 1.5 KB 内存就能估计数十亿个 unique 元素的数量，误差约 2%。

Count-Min Sketch：使用哈希来估算数据流中每个元素的频率，用亚线性空间解决频率统计问题。

这两个算法的核心洞察都是：哈希将有序的信息打散，使我们可以用统计方法在极小的空间内进行近似推理。

第五部分：深入理解与实践

第12章 设计一个好的哈希函数

12.1 黄金法则

（1）使用成熟的、经过验证的算法。 不要自己设计密码学哈希函数。对于非密码学用途，xxHash、MurmurHash3、FarmHash 都是经过大量测试的优秀选择。

（2）均匀性是第一要务。 一个好的哈希函数应该使输出看起来"随机"——任何输入位的变化都应以约 50% 的概率翻转每个输出位。

（3）速度与安全的权衡。 SipHash 平衡了速度与针对哈希洪水攻击的安全性，被 Python、Ruby、Rust 等语言用作默认哈希算法。

12.2 评估哈希函数质量：雪崩测试

一个好的哈希函数应该满足严格雪崩准则：翻转输入中的任意一位，输出中每一位翻转的概率应该是 50%。

你可以写一个简单的测试：

取 10000 个随机输入
对每个输入：
    翻转其中一位
    比较原哈希值和翻转后的哈希值
    统计每一位的翻转频率
一个好的哈希函数中，每位翻转频率应该在 50% ± 1% 之间

12.3 常见的陷阱

陷阱 1：使用不安全或已被破解的算法。 MD5 和 SHA-1 不应用于任何安全场景。

陷阱 2：忘记加盐存储密码。 即使 SHA-256 是安全的，不加盐的密码哈希也容易受到彩虹表和字典攻击。

陷阱 3：基于指针的哈希。 在 Java 中，可变对象的默认 hashCode() 基于内存地址。如果你依赖它做去重，在不同 JVM 运行或 GC 后结果会不同——灾难性的 bug。

陷阱 4：用不可变的 key 做可变操作。 如果把可变对象放入 HashSet 后修改了它，哈希值改变后可能再也找不回这个元素了——它"丢失"在哈希表中了。

后记

从一道简单的模运算到保障全球金融系统的密码学哈希，从一个朴素的数据结构到支撑互联网基础设施的分布式哈希，哈希技术的故事是一部"简单的想法改变世界"的历史。

哈希的美妙之处在于它的简洁与深邃的统一：核心思想可以在五分钟内讲清楚，但它的分支和应用足够你研究一生。它用最少的资源（固定长度的比特串）解决了最多的难题（查找、验证、索引、共识）。

当你下一次在浏览器输入密码、在 GitHub 提交代码、或听到比特币挖矿的新闻时——你看到的不是魔法，而是哈希。一个灵巧而强大的数学发明，已经悄然成为数字世界的基础设施。
