StarRocks BE 存储引擎深度解析：从 Tablet 到列式存储的物理实现

2026-04-17 23:48 24 min read 人阅读

数据在 BE 上如何安家？——StarRocks 底层存储机制全揭秘

一、BE 在 StarRocks 中的角色：不止是“存储器”

在之前的文章中，我们深入探讨了 FE（Frontend）作为“大脑”的元数据管理与查询调度职责。那么，作为“四肢”的 BE（Backend） 节点，究竟是如何承担数据存储与 SQL 执行双重使命的？

BE 是 StarRocks 的后端节点，核心职责有二：

  1. 数据存储：负责将导入的数据按列式格式写入磁盘，并生成相关索引
  2. SQL 执行：执行 FE 分发下来的查询 Fragment，利用本地数据实现“计算向数据移动”

BE 节点之间完全对等，FE 按照既定规则将数据分配到各个 BE 节点。这种设计使得 StarRocks 能够实现水平无限扩展——增加 BE 节点数量，系统的存储容量和计算能力可近线性增长。

在生产环境中，BE 节点的硬件配置直接影响集群性能。建议每个 BE 节点配置 ≥ 16 核 CPU、≥ 64GB 内存，并使用 SSD 磁盘以最大化 I/O 性能。对于大规模集群，网络带宽建议 ≥ 10Gbps，以保证 Shuffle 过程的吞吐。

二、Tablet：BE 上最小的物理存储单元

2.1 什么是 Tablet？

Tablet 是 StarRocks 中最小的物理存储单元，也是数据复制、负载均衡、故障恢复的基本粒度。

StarRocks 采用分区（Partition）+ 分桶（Bucket） 的两级数据划分机制：

  • 分区：将表数据按某个维度（通常是时间）进行逻辑划分，便于数据管理和裁剪
  • 分桶：在每个分区内部，按指定的分桶键进行哈希分桶，将数据进一步切分为多个 Tablet

一张表的 Tablet 总数 = 分区数 × 分桶数 × 副本数。例如，一张按天分区（保留 30 天）、分桶数为 10、副本数为 3 的表，总 Tablet 数量为 30 × 10 × 3 = 900 个。

2.2 Tablet 的物理分布

每个 Tablet 会以多副本（Replica） 的形式存储在不同的 BE 节点上，默认副本数为 3。StarRocks 会自动管理每个 Tablet 副本的分布，确保：

  • 同一 Tablet 的多个副本分布在不同 BE 节点上，避免单点故障
  • 数据在集群内尽量均匀分布，避免数据倾斜

当 BE 节点规模发生变化（扩容或缩容）时，StarRocks 可以无需停止服务，直接完成节点的增减。节点变化会触发 Tablet 的自动迁移——新增节点时，部分 Tablet 会在后台自动均衡到新节点；节点下线时，其上的 Tablet 会自动迁移到其他节点，保证副本数不变。

2.3 Tablet 的设计意义

Tablet 作为基础物理单元，在 StarRocks 中承担着多重角色：

  • 实现并行计算：查询引擎可以并行扫描多个 Tablet，每个 Tablet 由一个独立的执行单元处理
  • 作为数据管理的基本单位：Compaction、负载均衡等操作都以 Tablet 为粒度
  • 支持系统水平扩展：扩容时通过 Tablet 级别的调度迁移数据
  • 保障高可用：多副本机制确保单节点故障不影响数据完整性

三、列式存储：BE 数据组织的核心哲学

3.1 列式存储的基本原理

StarRocks 的表数据按列存储。在物理层面，一列数据会经过分块、编码、压缩等操作后持久化到磁盘；在逻辑层面，一列数据可以看作由相同类型元素构成的一个数组。

这种设计的巧妙之处在于：一行数据的所有列值，在各列的数组中拥有相同的数组下标。这个下标是隐式的，不需要物理存储，从而节省了大量空间。

3.2 列式存储 vs 行式存储

对比维度 行式存储 列式存储
数据组织 按行连续存储 按列独立存储
压缩效率 较低，数据类型混合 高，同列数据类型一致
查询性能 适合点查（SELECT *） 适合分析查询（只读几列）
I/O 开销 需读取整行 只读取涉及的列
适用场景 OLTP 事务处理 OLAP 分析查询

在分析型查询（如聚合、过滤）中，列式存储可以直接跳过无关列的数据，大幅减少 I/O 开销。

3.3 StarRocks 列式存储的实现细节

StarRocks 采用的是显式列式存储方式——数据在物理存储上完全按列组织，每一列的数据独立存储在文件或块中。这种方式能够充分发挥列式存储的优势，尤其在大规模数据集和复杂查询场景下表现优异。

存储格式：StarRocks 的数据以列式格式存储，兼容 Parquet、CarbonData 等开放格式，支持高效的随机读取和压缩。

压缩算法：StarRocks 支持多种压缩算法，能根据列数据特性自动选择最优方式：

  • LZ4 / ZSTD：通用压缩算法，平衡压缩比与速度
  • 前缀编码（Prefix Encoding） ：适用于整数列，记录连续值的变化
  • 行程编码（RLE） ：适用于具有相同值的连续数据
  • 字典编码（Dictionary Encoding） ：适用于字符串列，将值映射为索引

这些压缩技术能将数据体积压缩 3-8 倍，显著降低存储成本和 I/O 开销。

四、索引体系：让列式存储“快上加快”

仅有列式存储还不够——在海量数据中快速定位目标行，需要高效的索引支持。StarRocks 在 BE 上构建了多层次的索引体系。

4.1 前缀索引（Prefix Index）

StarRocks 会自动为建表时指定的排序键（Sort Key）创建前缀索引。表中所有的行按照排序键做多重排序，排序后的位置就是该行的行号。

前缀索引是一种稀疏索引，每 1024 行数据构成一个逻辑数据块，索引记录的是每个数据块中排序键的最小值。查询时，系统通过二分查找快速定位目标行所在的数据块起始位置，从而在 Scan 阶段大面积减少无关数据扫描。

设计建议：前缀索引的长度限制为 36 字节，超过部分会被截断。因此应将经常作为查询条件、区分度高的列放在排序键的前面。

4.2 列级索引

除了前缀索引，StarRocks 还支持多种列级索引，进一步提升查询效率：

索引类型 原理 适用场景
Bloom Filter 概率性判断元素是否存在 高基数列的等值查询（如 user_id）
Bitmap 索引 用位图表示列值分布 低基数列的过滤（如 gender、status）
Zone Map 记录数据块中列的 min/max 范围查询的快速跳过

这些索引协同工作，使得 StarRocks 能够在 PB 级数据上实现秒级查询响应。

4.3 主键索引（Primary Key Index）

对于主键模型（Primary Key Model），StarRocks 还支持持久化主键索引。通过 enable_persistent_index = true 配置，可将主键索引持久化到磁盘，避免全内存索引带来的内存膨胀问题，支撑百亿级主键的高频更新场景。

五、BE 上的写入路径：数据如何落盘？

StarRocks 的写入路径经过精心设计，在保证数据可靠性的同时实现高吞吐。

5.1 写入流程概述

当数据导入时（无论是 Stream Load、Routine Load 还是 Broker Load），流程如下：

  1. FE 选定 Coordinator BE：FE 选择一个 BE 节点作为协调者，负责接收数据并分发
  2. 数据分片与分发：Coordinator BE 根据分桶规则，将数据拆分并发送到对应 Tablet 副本所在的 BE 节点
  3. 内存写入：每个 BE 节点先将数据写入 MemTable（内存缓冲区）
  4. 持久化：MemTable 写满或达到时间阈值后，Flush 为 Segment 文件写入磁盘
  5. 后台 Compaction：BE 后台定期将多个小 Segment 文件合并为大文件，减少文件数量、提升查询效率

5.2 Compaction 机制

Compaction 是 BE 存储引擎的关键后台任务。随着数据不断写入，磁盘上会产生大量小文件，影响查询性能。Compaction 负责将这些小文件合并：

  • Cumulative Compaction：将多个小文件合并为中等大小的文件
  • Base Compaction：将中等大小的文件进一步合并为大文件

合理的 Compaction 策略能显著提升查询性能，但也会消耗一定的 CPU 和 I/O 资源。建议通过 max_compaction_task_num_per_be 等参数控制 Compaction 并发度，避免在写入高峰期挤占查询资源。

六、存算一体 vs 存算分离：BE 的两种形态

StarRocks 支持两种架构模式，BE 在这两种模式下的形态有所不同：

对比维度 存算一体架构 存算分离架构
数据存储位置 BE 本地磁盘 对象存储（S3/OSS/HDFS）
计算节点 BE（计算 + 存储） CN（纯计算节点，取代 BE）
数据本地性 强，计算向数据移动 弱，数据需从远端读取
扩展方式 计算存储耦合扩展 计算存储独立扩展
存储成本 较高（本地 SSD） 低 70-80%（对象存储）

存算一体适合追求极致查询性能的场景——数据存储在本地，计算直接在数据所在节点执行，无网络传输开销。

存算分离适合数据量大、追求成本优化的场景——数据存储到低成本对象存储，计算节点根据业务负载灵活扩缩容。当缓存命中时，查询性能与存算一体架构相当。

七、BE 存储配置最佳实践

要让 BE 的存储引擎发挥最佳性能，以下几点配置建议值得关注：

7.1 存储路径配置

  • 每个 BE 节点可配置多个存储路径（storage_root_path），分布在不同物理磁盘上，提升 I/O 吞吐
  • 建议使用 NVMe SSD，避免机械盘成为 I/O 瓶颈
  • 多个存储路径之间以分号分隔，如：/data1/starrocks;/data2/starrocks

7.2 内存配置

  • BE 节点的 mem_limit 建议设置为物理内存的 60%~70%，预留空间给操作系统
  • 对于写入密集型场景，可适当调大 MemTable 内存限制
  • 开启 enable_memtable_spill，在内存不足时自动溢写磁盘，避免 OOM

7.3 副本配置

  • 生产环境必须设置 replication_num = 3，确保数据高可用
  • BE 节点数量最好为副本数的整数倍（如副本数为 3 时，BE 数量应为 3、6、9），便于副本均衡分布

7.4 Tablet 数量规划

  • 每个 BE 节点承载的 Tablet 数量建议在 10-20 个左右
  • 分桶数 = BE 节点数 × 2~4 倍，避免过少导致并行度不足，或过多导致元数据膨胀
  • 单个 Tablet 的压缩后大小建议控制在 500MB-1GB

八、总结

StarRocks BE 的存储引擎，是高性能数据分析的物理基石。它的设计哲学可以概括为：

  1. 分而治之：通过分区 + 分桶将大表切分为无数个 Tablet，实现并行处理
  2. 按列组织：采用列式存储，只读取查询需要的列，大幅减少 I/O
  3. 高效压缩：多种压缩算法将数据体积压缩 3-8 倍，降低存储成本
  4. 多层索引：前缀索引 + Bloom Filter + Bitmap + Zone Map，快速跳过无关数据
  5. 多副本保障：默认三副本机制，确保数据高可用和故障自愈
  6. 弹性伸缩：Tablet 自动迁移，节点增减无需停服

理解 BE 的存储原理，不仅能帮你更好地进行表设计（分区、分桶、排序键的选择），也能让你在遇到性能问题时，知道从哪个环节入手排查——是 Tablet 分布不均导致的数据倾斜？是 Compaction 不及时导致的小文件过多？还是索引设计不合理导致的扫描范围过大？

如果说 FE 是 StarRocks 的“智慧大脑”，那么 BE 就是承载一切的“强健体魄”——它以精妙的存储设计，支撑起了 PB 级数据的高性能分析。
