StarRocks 分区与分桶深度解析

一、开篇：理解 StarRocks 的数据组织哲学

StarRocks 是一款面向实时分析的 MPP（Massively Parallel Processing）数据库。要让一张几十亿甚至上百亿行的表在亚秒级返回查询结果，单靠"堆硬件"远远不够——关键在于数据如何被组织、切分和分布。

StarRocks 采用了一套独特的两级数据分布策略：分区（Partition） + 分桶（Bucket）。这两个概念经常被放在一起讨论，但它们处于完全不同的抽象层级，解决的是完全不同的问题。如果把一张表比作一座图书馆：

  • 分区像是按年份划分的书库楼层——你要找 2023 年的书，直接去三楼，不用在一楼和二楼浪费时间
  • 分桶像是每个楼层里的书架编号——到了三楼之后，书被均匀分布在 32 个书架上，你可以同时派 32 个人并行翻找

理解分区和分桶的区别与联系，是掌握 StarRocks 表设计的核心。本文将从原理到实践，从表层语法到底层存储引擎，全面剖析这两个概念。

二、数据结构全景图：从表到 Segment 文件

在深入分区和分桶之前，我们先建立 StarRocks 的数据存储层级全景图。这有助于理解分区和分桶在整体架构中的位置。

2.1 五层存储层级

Table（表）
  │
  ├── Partition 1（分区 1，对应 2025-01-01 这一天）
  │     ├── Tablet 1（分桶 1，在 BE-1、BE-3、BE-5 上各有一副本）
  │     ├── Tablet 2（分桶 2，在 BE-2、BE-4、BE-6 上各有一副本）
  │     └── Tablet N（分桶 N）
  │
  ├── Partition 2（分区 2，对应 2025-01-02 这一天）
  │     ├── Tablet 1
  │     ├── Tablet 2
  │     └── Tablet N
  │
  └── Partition M（分区 M）
        └── ...

深入每个 Tablet 内部：

Tablet（数据分片，最小分布与复制单元）
  │
  ├── Rowset 1（版本 [2-2]，某次导入产生）
  │     ├── Segment 1（.dat 列式数据文件，最大 512MB）
  │     └── Segment 2
  │
  ├── Rowset 2（版本 [3-3]，某次导入产生）
  │     └── Segment 3
  │
  └── Rowset 3（版本 [2-4]，Compaction 合并后产生）
        ├── Segment 4
        └── Segment 5

2.2 各层级的核心特性

层级 角色 数据量参考 管理方式
Table 逻辑表，对用户可见 无上限 CREATE TABLE
Partition 第一级物理分割，用于裁剪 单分区建议 < 100GB 表达式分区 / Range / List
Tablet 最小数据分布单元，多副本 100MB ~ 10GB（理想 100MB ~ 1GB） 由分桶数决定
Rowset 一次导入或 Compaction 产生的不可变数据集合 取决于导入批次 自动管理
Segment 物理列式数据文件 最大 512MB 自动管理

2.3 数据写入路由：从 SQL 到 Segment 的全路径

理解写入路由有助于理解分区和分桶如何协同工作：

1. 客户端发送 INSERT / Stream Load → FE（Frontend）
2. FE 解析 SQL，校验表结构、分区规则、权限
3. FE 根据分区规则，确定数据应落入哪个 Partition
   （例如：event_time='2025-03-15' → Partition p20250315）
4. FE 根据分桶键的哈希值，确定数据应落入该分区的哪个 Tablet
   （例如：HASH(user_id=12345) % 64 = 17 → Tablet 17）
5. FE 将数据路由到 Tablet 17 的 Leader Replica 所在的 BE 节点
6. BE 接收数据，写入 MemTable → Flush 为 Segment 文件 →
   生成 Rowset → 提交版本

关键理解：分区决定了"数据去哪一组 Tablet"，分桶决定了"数据去这一组里的哪一个 Tablet"。两者串行决策，缺一不可。

三、分区（Partition）—— 第一层数据切分

3.1 分区的本质

分区的本质是逻辑切分。它将表中的行按照分区列的值划分为不同的子集，每个子集就是一个分区。分区之间是互斥且完备的——每一行数据属于且仅属于一个分区。

分区的主要价值不在于"数据被分开了"，而在于分区裁剪（Partition Pruning）：当查询的 WHERE 条件中包含分区列时，StarRocks 的查询优化器可以直接跳过不相关的分区，从而大幅减少需要扫描的数据量。

-- 查询 2025 年 3 月 15 日的数据
SELECT * FROM orders WHERE order_date = '2025-03-15';

如果表按 order_date 做了按天分区：
  扫描 1 个分区（约 10GB）
如果表没有做分区：
  扫描整张表（约 365 × 10GB = 3.6TB）

如果不做分区，StarRocks 也会自动创建一个默认分区，所有数据都放在这一个分区里。这意味着没有任何分区裁剪的收益。

3.2 分区方式详解

StarRocks 在不同版本中支持多种分区方式。截至 2025 年中，推荐的使用策略如下：

3.2.1 表达式分区（Expression Partitioning）—— 推荐首选

表达式分区是 v3.1 引入、v3.4 起统一化的分区方式。建表时只需指定一个返回日期或整数的表达式，数据导入时自动根据表达式结果创建对应的分区，无需预先枚举分区列表。

-- 按天分区
PARTITION BY date_trunc('day', event_time)

-- 按月分区
PARTITION BY date_trunc('month', order_date)

-- 按小时分区（高频数据场景）
PARTITION BY date_trunc('hour', log_time)

-- 按整数范围分区（如按租户 ID 取模 100）
PARTITION BY (tenant_id % 100)

表达式分区的优势：

  • 零维护：无需手动 ADD PARTITION，导入数据时自动创建
  • 统一语法：一套语法覆盖 Range 和 List 的所有场景
  • 灵活：支持多种日期函数和整数运算

3.2.2 Range 分区

Range 分区按连续的值范围划分数据，适合按日期或数值区间查询的场景。

PARTITION BY RANGE(order_date) (
    PARTITION p202501 VALUES LESS THAN ("2025-02-01"),
    PARTITION p202502 VALUES LESS THAN ("2025-03-01"),
    PARTITION p202503 VALUES LESS THAN ("2025-04-01"),
    PARTITION p202504 VALUES LESS THAN ("2025-05-01")
)

Range 分区支持：

  • 多列分区键：PARTITION BY RANGE(dt, region_id)
  • MAXVALUE：PARTITION p_future VALUES LESS THAN (MAXVALUE) 作为兜底分区
  • 手动 ADD/DROP：ALTER TABLE t ADD PARTITION p202505 VALUES LESS THAN ("2025-06-01")
  • 动态分区：配合 dynamic_partition 属性自动管理

3.2.3 List 分区

List 分区按离散的枚举值划分数据，适合按地区、品类、状态等枚举列查询的场景。

PARTITION BY LIST(region) (
    PARTITION p_north VALUES IN ("北京", "天津", "河北"),
    PARTITION p_east  VALUES IN ("上海", "江苏", "浙江"),
    PARTITION p_south VALUES IN ("广东", "福建", "海南")
)

注意：自 v3.4 起，表达式分区已能够覆盖 List 分区的场景（例如 PARTITION BY region），List 分区仅作为兼容语法保留。官方建议新项目优先使用表达式分区。

3.3 分区键的选择原则

因素 建议 原因
列类型 仅支持 DATE、DATETIME、整数类型 协议限制，STRING 等类型不可作为分区键
首选时间列 如 order_date、event_time、dt 分析查询几乎都带时间范围
粒度选择 单分区原始数据 < 100GB 过大则裁剪效果差，过细则元数据膨胀
高频过滤列 查询中 80% 以上会作为过滤条件的列 最大化分区裁剪收益
管理需求 需要按天删除/归档 → 按天分区 直接 DROP PARTITION，秒级完成

3.4 分区粒度决策树

单分区数据量 < 1GB
  → 可以不分区，或按月分区以减少元数据开销
  → 如果查询总是精确到天，仍建议按天分区

单分区数据量 1GB ~ 100GB
  → 按天分区，这是最理想的状态

单分区数据量 > 100GB
  → 细化分区粒度（天 → 小时）
  → 或调整业务逻辑，减少单分区数据量

需要按小时级别的时效性
  → 按小时分区（适用于日志、监控类场景）
  → 注意：小时分区会导致分区数量 × 24 的膨胀

历史数据量极大，查询频率低
  → 使用冷热分层：热数据按天分区存 SSD，冷数据按月分区存 SATA
  → 低频率查询的历史数据用更大粒度的分区

3.5 动态分区 —— 自动化分区管理

手动管理分区的最大痛点是：忘记添加新分区会导致导入失败，忘记删除旧分区会导致存储膨胀。动态分区机制解决了这个问题。

PROPERTIES (
    -- 启用动态分区
    "dynamic_partition.enable" = "true",

    -- 分区粒度：DAY / WEEK / MONTH / HOUR
    "dynamic_partition.time_unit" = "DAY",

    -- 保留最近 N 天的分区（N 天之前的分区自动删除）
    "dynamic_partition.start" = "-30",

    -- 提前创建未来 N 天的分区（避免导入时无分区可用）
    "dynamic_partition.end" = "3",

    -- 分区名前缀
    "dynamic_partition.prefix" = "p",

    -- 时区设置
    "dynamic_partition.time_zone" = "Asia/Shanghai",

    -- 历史分区数量（用于冷热分层场景）
    "dynamic_partition.history_partition_num" = "0"
)

动态分区的工作机制：

  1. FE 每隔 dynamic_partition_check_interval_seconds（默认 600 秒）检查一次
  2. 根据当前时间和配置的 start / end 偏移量，计算出应存在的分区列表
  3. 自动创建缺失的未来分区，自动删除过期的历史分区
  4. 删除分区是物理删除，数据不可恢复（建议在业务低峰期操作）

四、分桶（Bucketing）—— 第二层数据切分

4.1 分桶的本质

分桶的本质是物理切分。在同一个分区内部，数据根据分桶键的哈希值被分配到不同的 Tablet 中。每个 Tablet 是最小的数据分布单元——它以多副本（默认 3 副本）的形式分布在不同的 BE 节点上。

分桶决定了两个关键指标：

  • 数据均匀性：分桶键选得好不好，决定了数据是否均匀分布
  • 查询并行度：分桶数量决定了单分区的最大并行扫描度

4.2 分桶方式

StarRocks 提供两种分桶方式：

4.2.1 哈希分桶（Hash Bucketing）—— 推荐

数据按分桶键的哈希值 CRC32(col) % BUCKETS 分配到各 Tablet。

-- 单列哈希分桶
DISTRIBUTED BY HASH(user_id) BUCKETS 32

-- 多列组合分桶（建议不超过 3 列）
DISTRIBUTED BY HASH(user_id, order_id) BUCKETS 64

-- 自动分桶（v2.5.7+，省略 BUCKETS 由系统自动计算）
DISTRIBUTED BY HASH(user_id)

哈希分桶的核心优势：

  1. 分桶裁剪（Bucket Pruning）：WHERE 条件中包含分桶键的等值条件时，StarRocks 可以直接定位到唯一的 Tablet，只需扫描 1/BUCKETS 的数据
  2. 本地聚合（Local Aggregation）：相同分桶键的数据在同一 Tablet，GROUP BY 分桶键时可以在本地完成聚合
  3. Bucket Shuffle Join：JOIN 键等于分桶键时，可以按桶做本地 JOIN
  4. Colocate Join：多表共置在同一 Colocation Group 中时，JOIN 完全无网络传输

4.2.2 随机分桶（Random Bucketing）

数据随机分布到各 Tablet，不基于任何键。

-- 随机分桶（仅 v3.1+，仅支持明细表）
DISTRIBUTED BY RANDOM
特性 哈希分桶 随机分桶
数据是否均匀 取决于分桶键选择 天然均匀
分桶裁剪 支持 不支持
Colocate Join 支持 不支持
Bucket Shuffle Join 支持 不支持
适用表类型 所有表类型 仅明细表（Duplicate Key）
设计负担 需要合理选择分桶键 零设计负担
适用场景 大多数场景 小表、临时表、查询模式不固定的表

使用建议：除非有明确的理由，否则始终使用哈希分桶。随机分桶仅适用于无 JOIN 需求、查询模式简单、数据量不大的明细表。

4.3 分桶键的选择 —— 最重要的设计决策

分桶键的选择直接影响查询性能和存储均衡。以下是决策框架：

第一原则：高基数

分桶键的基数（distinct values）决定了数据能否被均匀打散。

坏的选择：gender（基数 = 2~3）
   → 所有男性数据在一个 Tablet，所有女性在另一个 → 严重倾斜

好的选择：user_id（基数 = 亿级）
   → 数据均匀分布在所有 Tablet 上

折中选择：city（基数 = 几百~几千）
   → 可接受，但理想情况是基数 >> BUCKETS 数量

第二原则：高频查询列

分桶键应该同时是查询中频繁作为过滤条件的列。

如果你的 80% 查询都是：
  SELECT ... FROM orders WHERE user_id = ?

那么 user_id 是最理想的分桶键：
  → 每次查询只需扫描 1 个 Tablet（分桶裁剪）
  → 查询延迟从秒级降至毫秒级

第三原则：用于 JOIN / GROUP BY

如果表频繁参与 JOIN 或者有 GROUP BY 的需求，分桶键应尽量选 JOIN 键或 GROUP BY 键。

场景：orders 表和 users 表频繁按 user_id JOIN

  CREATE TABLE orders  (...) DISTRIBUTED BY HASH(user_id) BUCKETS 48;
  CREATE TABLE users   (...) DISTRIBUTED BY HASH(user_id) BUCKETS 48;

  配合 Colocate Join → JOIN 完全在本地完成，零网络传输

第四原则：组合分桶键

当单一列无法同时满足"高基数 + 高频查询 + JOIN 列"时，使用多列组合分桶键。

-- 两个维度组合：租户 + 订单
DISTRIBUTED BY HASH(tenant_id, order_id) BUCKETS 64

组合分桶键的规则：

  • 列的顺序有意义：HASH(A, B) 和 HASH(B, A) 不等价
  • 相同 A 值但不同 B 值的行会被分到不同 Tablet（进一步打散）
  • 建议不超过 3 列，过多会增加哈希计算开销
  • 多列组合分桶后，分桶裁剪条件需要包含全部分桶键或前缀

4.4 分桶数量的计算

分桶数量是最常见的配置失误点。过多则元数据爆炸（FE 内存不足），过少则并行度不足（查询性能差）。

公式一：按数据量估算（推荐）

分桶数 = 单分区预估数据量 / 目标 Tablet 大小

目标 Tablet 大小：100MB ~ 1GB（理想范围）
最大不超过 10GB

示例：
  单分区 100GB，目标 Tablet 1GB
  → BUCKETS = 100

公式二：按集群并行能力计算

分桶数 = BE节点数 × 单BE CPU核数

或保守公式：
分桶数 = BE节点数 × 单BE CPU核数 / 2

示例：
  6 个 BE 节点，每节点 16 核
  → BUCKETS = 6 × 16 = 96
  → 保守：96 / 2 = 48

公式三：同时满足两个条件

最终分桶数 = MAX(按数据量计算的值, 按并行能力计算的值)

但要确保：
  分桶数 / BE节点数 ≥ 4
  （保证负载均衡的最小倍数）

自动分桶（v2.5.7+）

如果省略 BUCKETS 参数，StarRocks 会自动计算。自动分桶的逻辑会随着版本更新不断优化，但对于大表或查询密集型场景，仍建议手动指定。

-- 自动分桶：省略 BUCKETS
DISTRIBUTED BY HASH(user_id)

-- 手动分桶：明确指定
DISTRIBUTED BY HASH(user_id) BUCKETS 128

五、分区与分桶的协同工作

5.1 核心区别

维度 分区（Partition） 分桶（Bucket）
层级 第一层（逻辑切分） 第二层（物理切分）
数据组织 将表切分为互斥的子集 将分区内的数据切分为并行的分片
主要目的 减少扫描范围（跳过无关分区） 并行计算 + 数据均匀分布
算法 范围判断（Range / 表达式） 哈希取模（HASH % BUCKETS）/ 随机
与查询的关系 WHERE 条件匹配 → 跳过整个分区 WHERE 条件匹配 → 定位到单个 Tablet
跨节点分布 不同分区可能集中在某些节点 同一分区的不同 Tablet 分布在所有节点
修改灵活性 可增删分区，不可改分区方式 v3.2+ 可修改分桶键和桶数
生命周期 有明确的起止时间（历史→过期→删除） 无生命周期概念

5.2 核心联系

分区和分桶不是独立工作的，它们构成了一个两级路由系统：

  1. 第一级路由（分区）：根据分区列的值确定数据落在哪个 Partition
  2. 第二级路由（分桶）：根据分桶键的哈希值确定数据落在 Partition 的哪个 Tablet

这种两级路由在查询时的表现：

查询: SELECT * FROM orders
       WHERE order_date = '2025-03-15'
         AND user_id = 12345

执行过程:
  1. 分区裁剪：order_date = '2025-03-15'
     → 只保留 Partition p20250315，其余 364 个分区被跳过

  2. 分桶裁剪：user_id = 12345
     → HASH(12345) % 64 = 17
     → 只扫描 Tablet 17，其余 63 个 Tablet 被跳过

  最终扫描数据量 = 全表数据量 / 365 / 64 ≈ 全表的 0.004%

5.3 分区和分桶的"排列组合"

不同的分区和分桶组合，对查询的影响不同：

场景 A：分区键命中 + 分桶键命中
  → 最优情况，扫描最少数据

场景 B：分区键命中 + 分桶键未命中
  → 扫描单个分区的所有 Tablet，仍是较好的情况

场景 C：分区键未命中 + 分桶键命中
  → 需要扫描每个分区中的特定 Tablet，分区数多时性能一般

场景 D：分区键未命中 + 分桶键未命中
  → 全表扫描，最差情况

核心设计原则：让最常见的查询同时命中分区键和分桶键。

六、分区、分桶与排序键（Sort Key）的关系

6.1 排序键是什么

排序键由建表时的 DUPLICATE KEY / AGGREGATE KEY / UNIQUE KEY / PRIMARY KEY 定义。它决定了数据在 Tablet 内部按什么顺序存储。

-- 明细表：排序键 = DUPLICATE KEY 列
CREATE TABLE events (
    event_time DATETIME,
    user_id    BIGINT,
    event_type VARCHAR(32)
) ENGINE=OLAP
DUPLICATE KEY(event_time, user_id)  -- 排序键
PARTITION BY date_trunc('day', event_time)
DISTRIBUTED BY HASH(user_id) BUCKETS 32;

6.2 排序键与分区键的关系

排序键的第一列通常就是分区键（尤其是时间列）。这不是强制要求，但是强烈推荐的做法：

推荐：排序键的第一列 = 分区键
  → 数据在 Tablet 内按时间顺序排列
  → 范围查询高效，ORDER BY 排序成本低

不推荐：排序键与分区键无关
  → 时间范围查询需要扫描更多 Segment
  → Compaction 时无法按时间段高效合并

6.3 排序键与分桶键的关系

排序键和分桶键是正交的，但它们的协调设计能产生 1+1 > 2 的效果：

CREATE TABLE orders (
    order_date  DATE,          -- 分区键
    user_id     BIGINT,        -- 分桶键 + 排序键第二列
    order_id    BIGINT,        -- 排序键第一列
    amount      DECIMAL(10,2)
) ENGINE=OLAP
DUPLICATE KEY(order_id, user_id)
PARTITION BY order_date
DISTRIBUTED BY HASH(user_id) BUCKETS 64;

在这个设计中：

  • order_date 负责分区裁剪（减少扫描范围）
  • user_id 负责分桶裁剪 + 数据打散 + Colocate Join
  • order_id 作为排序键第一列，加速按订单 ID 的精确查询

6.4 前缀索引（Prefix Index）

StarRocks 会自动基于排序键构建前缀索引。前缀索引的机制是：对排序键的前 N 列构建稀疏索引，查询时可以通过前缀索引快速跳过不匹配的数据块。

排序键: (A, B, C, D)

前缀索引可以加速的查询：
  WHERE A = ?                    ← 命中 1 列前缀
  WHERE A = ? AND B = ?          ← 命中 2 列前缀
  WHERE A = ? AND B = ? AND C = ?← 命中 3 列前缀

前缀索引无法加速的查询：
  WHERE B = ?                    ← 跳过了 A
  WHERE A = ? AND C = ?          ← 跳过了 B

排序键的设计原则：将高频、高选择性的过滤列放在排序键的前面，且不要跳过任何列。

七、Colocate Join：分区和分桶设计的"终极武器"

7.1 什么是 Colocate Join

Colocate Join（共置连接）是 StarRocks 的一项核心优化技术。它将一组表放入同一个 Colocation Group (CG) 中，确保这些表具有相同的分桶键、分桶数和副本分布。当 JOIN 列等于分桶键时，JOIN 操作完全在本地 BE 节点内完成，无需任何跨节点的数据传输。

7.2 工作原理

没有 Colocate Join 时（Broadcast / Shuffle Join）：
  BE-1: 持有 orders 的 Tablet 1~8
  BE-2: 持有 customers 的 Tablet 1~8
  → JOIN 时需要将 customers 的数据广播到所有 BE
  → 网络传输量大，延迟高

有 Colocate Join 时：
  BE-1: 持有 orders Tablet 1 和 customers Tablet 1（相同桶号）
  BE-2: 持有 orders Tablet 2 和 customers Tablet 2（相同桶号）
  ...
  → JOIN 时每个 BE 本地直接匹配，无需网络传输
  → 延迟降低 10~100 倍

7.3 实现约束

要将表放入同一个 Colocation Group，必须满足以下条件：

约束 说明
分桶键一致 类型、数量、顺序必须完全相同（列名可以不同）
分桶数一致 BUCKETS 数量必须相同
副本数一致 replication_num 必须相同
表在同一数据库 CG 作用于数据库级别

7.4 实战示例

-- 创建 Colocation Group 中的第一张表
CREATE TABLE orders (
    order_id    BIGINT,
    customer_id BIGINT,
    order_date  DATE,
    amount      DECIMAL(10,2)
) ENGINE=OLAP
DUPLICATE KEY(order_id)
PARTITION BY order_date
DISTRIBUTED BY HASH(customer_id) BUCKETS 48
PROPERTIES (
    "colocate_with" = "customer_group",
    "replication_num" = "3"
);

-- 创建同一 Colocation Group 中的第二张表
CREATE TABLE customers (
    customer_id BIGINT,
    name        VARCHAR(100),
    city        VARCHAR(50)
) ENGINE=OLAP
PRIMARY KEY(customer_id)
DISTRIBUTED BY HASH(customer_id) BUCKETS 48
PROPERTIES (
    "colocate_with" = "customer_group",
    "replication_num" = "3"
);

-- 查询：自动使用 Colocate Join
SELECT o.*, c.name, c.city
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.order_date = '2025-03-15';

7.5 Colocate Join 的局限和降级

  • CG 状态为 Unstable 时：Colocate Join 会降级为普通的 Shuffle Join，不影响正确性，但性能下降
  • 表结构变更后：如 ALTER TABLE 修改了分桶键或桶数，该表会退出 CG
  • BE 节点变更后：CG 需要重新均衡，期间可能处于 Unstable 状态
  • 查看 CG 状态：SHOW PROC '/colocation_group'

八、底层存储引擎与分区/分桶的交互

8.1 Tablet 的内部结构

每个 Tablet 由多组 Rowset 组成，每个 Rowset 包含若干 Segment 文件。这一结构对分区和分桶设计有以下影响：

分区和分桶影响 Tablet 的"粒度"：
  - 分区过多 → Tablet 总数 = 分区数 × 分桶数 → 元数据膨胀
  - 分桶过多 → 同样导致 Tablet 总数爆炸

每个 Tablet 在 FE 内存中约占 5KB：
  - 1000 个 Partition × 64 个 Bucket = 64,000 个 Tablet
  - 64,000 × 5KB ≈ 320MB FE 内存 → 可接受

  - 3650 个 Partition × 256 个 Bucket = 934,400 个 Tablet
  - 934,400 × 5KB ≈ 4.6GB FE 内存 → 压力较大

8.2 Compaction 与分区/分桶的关系

Compaction（合并）是 StarRocks 后台持续运行的维护任务，它将多个小 Rowset 合并为一个大 Rowset。分区和分桶设计会影响 Compaction 的效率：

  • 分区粒度过细 + 高并发小批量导入 → 每个 Tablet 内积压大量小 Rowset → Compaction 压力大
  • 分区合理 + 批量导入 → 每个 Tablet 内 Rowset 数量可控 → Compaction 压力小
  • 分桶数量影响 Compaction 并发度：更多 Tablet 意味着更多可并行的 Compaction 任务
-- 查看 Compaction 状态
SHOW PROC '/statistic';
SHOW PROC '/tasks';

8.3 主键表（Primary Key）的特殊考量

主键表的分区和分桶设计有一个额外的重要约束：主键表必须设置分区。

原因：主键表维护了一个主键到 Row ID 的映射索引（Persistent Index）。如果不做分区，全表的索引会一直驻留在内存中，导致：

  • 大表 → 索引过大 → BE 内存 OOM
  • 即使能承受，全表索引的维护成本也极高
推荐做法：
  主键表 + 按天分区 + 动态分区（保留 30 天）
  → 只有最近 30 天的数据索引在内存中
  → 历史分区可设置为冷数据，释放内存

九、实战：建表全流程设计

9.1 建表决策框架

以下是一个系统化的建表决策流程：

第一步：确定业务查询模式
  ├── 最常见的 WHERE 条件是什么？
  ├── 是否频繁 JOIN？JOIN 键是什么？
  ├── 是否有 GROUP BY？GROUP BY 的列是什么？
  └── 查询的时间范围通常多大？

第二步：选择分区方案
  ├── 分区键 = 最常用的时间过滤列
  ├── 分区粒度 = 基于单分区数据量选择（< 100GB）
  ├── 分区方式 = 优先表达式分区
  └── 启用动态分区自动管理

第三步：选择分桶方案
  ├── 分桶键 = 高基数 + 高频过滤 + JOIN 键 的列
  ├── 分桶数 = MAX(数据量/1GB, BE数×CPU核数/2)
  └── 考虑是否需要 Colocate Join

第四步：设计排序键
  ├── 排序键第一列 ≈ 分区键（通常是时间列）
  ├── 后面紧跟高频过滤列
  └── 注意前缀索引的匹配规则

第五步：验证与调优
  ├── 检查数据分布均匀性
  ├── 检查查询计划是否使用分区裁剪和分桶裁剪
  └── 监控 Tablet 大小和 Compaction 状态

9.2 完整建表示例

示例一：用户行为日志表（明细模型）

CREATE TABLE user_behavior_log (
    event_id      BIGINT       NOT NULL COMMENT "事件ID",
    user_id       BIGINT       NOT NULL COMMENT "用户ID",
    event_type    VARCHAR(32)  NOT NULL COMMENT "事件类型",
    event_time    DATETIME     NOT NULL COMMENT "事件时间",
    event_date    DATE         NOT NULL COMMENT "事件日期(冗余)",
    page_url      VARCHAR(1024)         COMMENT "页面URL",
    duration_ms   INT                   COMMENT "停留时长(ms)",
    device_id     VARCHAR(128)          COMMENT "设备ID"
) ENGINE = OLAP
DUPLICATE KEY(event_id, event_time)
COMMENT "用户行为日志表"
PARTITION BY date_trunc('day', event_time)        -- 按天表达式分区
DISTRIBUTED BY HASH(user_id, device_id) BUCKETS 96 -- 组合哈希分桶
PROPERTIES (
    "replication_num" = "3",
    "storage_medium" = "SSD",
    -- 动态分区：保留最近 15 天 + 提前创建 3 天
    "dynamic_partition.enable" = "true",
    "dynamic_partition.time_unit" = "DAY",
    "dynamic_partition.start" = "-15",
    "dynamic_partition.end" = "3",
    "dynamic_partition.prefix" = "p",
    "dynamic_partition.time_zone" = "Asia/Shanghai",
    -- Bloom Filter 加速低基数列的等值查询
    "bloom_filter_columns" = "event_type,device_id"
);

设计说明：

  • 分区键 event_time：日志查询几乎都带时间范围，按天分区实现最大裁剪
  • 分桶键 user_id, device_id：两个高基数列组合，充分打散数据，同时支持按用户或设备的分桶裁剪
  • 排序键 event_id, event_time：event_id 保证精确查询高效，event_time 保证范围扫描有序
  • 动态分区保留 15 天：控制存储成本

示例二：电商订单表（主键模型）

CREATE TABLE orders (
    order_id      BIGINT        NOT NULL COMMENT "订单ID",
    user_id       BIGINT        NOT NULL COMMENT "用户ID",
    order_status  TINYINT       NOT NULL COMMENT "订单状态",
    total_amount  DECIMAL(18,2) NOT NULL COMMENT "订单金额",
    pay_type      TINYINT                COMMENT "支付方式",
    create_time   DATETIME      NOT NULL COMMENT "创建时间",
    update_time   DATETIME      NOT NULL COMMENT "更新时间"
) ENGINE = OLAP
PRIMARY KEY(order_id)                              -- 主键模型
PARTITION BY date_trunc('day', create_time)         -- 按天表达式分区
DISTRIBUTED BY HASH(user_id) BUCKETS 64             -- 按用户ID哈希分桶
PROPERTIES (
    "replication_num" = "3",
    "dynamic_partition.enable" = "true",
    "dynamic_partition.time_unit" = "DAY",
    "dynamic_partition.start" = "-90",              -- 保留 90 天订单
    "dynamic_partition.end" = "3",
    "dynamic_partition.prefix" = "p",
    "dynamic_partition.time_zone" = "Asia/Shanghai",
    "bloom_filter_columns" = "order_status,pay_type"
);

设计说明：

  • 主键模型 + 按天分区：避免全表索引常驻内存
  • order_id 作为主键：支持 Upsert（订单状态更新）
  • user_id 作为分桶键：用户查看"我的订单"高效（分桶裁剪）
  • 保留 90 天：平衡查询需求与存储成本

示例三：Colocate Join 场景 —— 实时数仓星型模型

-- 事实表：订单明细
CREATE TABLE fact_order_detail (
    order_id      BIGINT   NOT NULL,
    product_id    BIGINT   NOT NULL,
    customer_id   BIGINT   NOT NULL,
    order_date    DATE     NOT NULL,
    quantity      INT      NOT NULL,
    unit_price    DECIMAL(10,2) NOT NULL
) ENGINE = OLAP
DUPLICATE KEY(order_id)
PARTITION BY order_date
DISTRIBUTED BY HASH(customer_id) BUCKETS 48
PROPERTIES (
    "colocate_with" = "sales_star",
    "replication_num" = "3"
);

-- 维度表：客户
CREATE TABLE dim_customer (
    customer_id   BIGINT       NOT NULL,
    customer_name VARCHAR(100),
    city          VARCHAR(50),
    segment       VARCHAR(20)
) ENGINE = OLAP
PRIMARY KEY(customer_id)
DISTRIBUTED BY HASH(customer_id) BUCKETS 48
PROPERTIES (
    "colocate_with" = "sales_star",
    "replication_num" = "3"
);

-- 维度表：产品
CREATE TABLE dim_product (
    product_id    BIGINT       NOT NULL,
    product_name  VARCHAR(200),
    category      VARCHAR(50),
    brand         VARCHAR(50)
) ENGINE = OLAP
PRIMARY KEY(product_id)
DISTRIBUTED BY HASH(product_id) BUCKETS 48
PROPERTIES (
    "colocate_with" = "sales_star",
    "replication_num" = "3"
);

设计说明：

  • 三张表都在同一个 Colocation Group sales_star 中
  • fact_order_detail 和 dim_customer 按 customer_id 分桶 → 它们的 JOIN 是 Colocate Join
  • dim_product 按 product_id 分桶 → 与事实表的 product_id JOIN 需要 Shuffle，但维度表很小，代价可控
  • 如果事实表与产品表的 JOIN 也很频繁，可以考虑将产品维度表也改为按 customer_id 分桶，或用其他优化策略

十、常见问题与避坑指南

10.1 数据倾斜

问题：某个 Tablet 的数据量远大于其他 Tablet，导致该 BE 节点成为查询瓶颈。

原因诊断：
  1. 分桶键基数太低（如只用 gender 或 status）
  2. 业务数据本身有热点（如少数大客户占了 80% 数据）
  3. 分桶键与业务增长不匹配（如按城市分桶，一线城市数据远超其他）

解决方案：
  方案一：使用多列组合分桶加基数
    DISTRIBUTED BY HASH(user_id, order_id) -- 比单独 user_id 更均匀

  方案二：热点值加盐（Salt）
    DISTRIBUTED BY HASH(
      CASE
        WHEN user_id IN (1001, 1002, 1003)  -- 热点用户
        THEN CONCAT(user_id, '_', order_id % 10)  -- 加随机后缀
        ELSE CAST(user_id AS STRING)
      END
    )

  方案三：业务层预处理
    将大客户的数据独立成单独的表或分区

检查数据分布：
  SHOW DATA FROM database.table;   -- 查看各 Tablet 的数据量
  SHOW TABLET FROM table_name;     -- 查看 Tablet 详细信息

10.2 Tablet 数量过多

问题：FE 内存持续增长，查询变慢，SHOW PROC '/statistic' 显示 Tablet 数量很大。

原因诊断：
  1. 分区粒度过细（如按小时分区 + 保留 365 天 = 8760 个分区）
  2. 分桶数过大（如给 1GB 的表设置了 256 个桶）
  3. 两者叠加：8760 分区 × 256 桶 = 2,242,560 个 Tablet

解决方案：
  1. 检查分区粒度是否合理（单分区是否真的需要 < 1GB）
  2. 历史分区合并：将 30 天前的天分区合并为月分区
  3. 减少分桶数：小表用 8~16 个桶足够

计算公式：
  Tablet 总数 = 分区数 × 分桶数 × 副本数
  FE 内存占用（估算）= Tablet 总数 × 5KB

  建议 Tablet 总数控制在 10 万以内

10.3 主键表导入 OOM

问题：主键表在大批量导入时 BE 节点 OOM。

原因：主键表维护 Persistent Index，导入时需要加载索引页到内存。
      如果不分区或分区不合理，索引量过大导致 OOM。

解决方案：
  1. 必须设置分区：主键表不分区 = 全表索引常驻内存
  2. 合理设置分区粒度：按天或按小时分区，控制单次导入索引量
  3. 调整参数：
     - persistent_index_major_compaction_limit_per_disk：触发索引 Compaction
     - 增大 BE 内存或降低并发导入数

10.4 分桶键与更新操作的冲突

问题：需要 UPDATE 分桶键列的值，但 StarRocks 不支持。

分桶键列的值不支持 UPDATE。

解决方式：
  方案一：选择不会被更新的列作为分桶键（如 order_id、user_id）
  方案二：如果必须更新，通过 DELETE + INSERT 实现
  方案三：建表时就避免将可能变更的列作为分桶键

10.5 分区裁剪不生效

问题：查询计划显示扫描了全部分区，而非预期的少数分区。

原因诊断：
  1. WHERE 条件中对分区列使用了函数
     WHERE DATE_FORMAT(order_date, '%Y-%m') = '2025-03'  -- 裁剪失效
     WHERE order_date >= '2025-03-01' AND order_date < '2025-04-01' -- 裁剪生效

  2. WHERE 条件中分区列在 OR 的另一侧
     WHERE order_date = '2025-03-15' OR user_id = 12345  -- 裁剪可能失效

  3. 使用了 NOT IN 或 !=
     WHERE order_date != '2025-03-15'  -- 裁剪失效

解决方案：
  1. 对分区列不做函数包装，用范围条件替代
  2. 用 UNION ALL 替代复杂的 OR 条件
  3. 检查 EXPLAIN 输出确认裁剪效果

EXPLAIN SELECT ...;  -- 查看是否显示 "partitions=X/Y"

11.1 关键监控指标

-- 查看表的数据分布

-- 查看所有 Tablet 的详细信息

-- 查看 Compaction 状态

-- 查看 Colocation Group 状态
SHOW PROC '/colocation_group';

-- 查看分区信息
SHOW PARTITIONS FROM table_name;

-- 查看查询计划（确认分区裁剪和分桶裁剪）
EXPLAIN SELECT ...;

-- 查看查询 Profile（实际执行情况）
EXPLAIN ANALYZE SELECT ...;

11.2 查询计划解读

一个良好的查询计划应该包含：

  OlapScanNode
    TABLE: orders
    PARTITION: p20250315          ← 分区裁剪生效
    BUCKETS: 1/64                 ← 分桶裁剪生效（只扫描 1/64 的桶）
    PREAGGREGATION: ON            ← 本地预聚合生效
    ...

对比一个较差的查询计划：

  OlapScanNode
    TABLE: orders
    PARTITION: p20250101..p20251231  ← 全部分区
    BUCKETS: 64/64                   ← 全部桶
    PREAGGREGATION: OFF
    ...

检查项 目标 操作
分区键选择 90% 查询命中分区裁剪 调整为更常用的时间列
分区粒度 单分区 1GB~100GB 调整 time_unit
分桶键选择 数据均匀 + 查询裁剪 换用更高基数列

12.1 关键版本变化

12.2 未来展望：动态 Tablet 拆分与合并

StarRocks 社区正在开发**动态 Tablet 拆分与合并（Dynamic Tablet Splitting & Merging）**功能（见 GitHub Issue #59134）。这一功能将彻底改变分桶设计的方式：

  • 建表时不再需要精确估算分桶数：系统会根据实际数据量自动拆分或合并 Tablet
  • Tablet 过大自动拆分：当 Tablet 超过阈值（如 10GB），自动拆分为多个小 Tablet
  • Tablet 过小自动合并：当多个 Tablet 都很小时，自动合并为一个
  • 优先支持存算分离模式：Shared-Data 集群将率先受益

这一特性落地后，"分桶数设为多少"这个最令 DBA 头疼的问题将迎刃而解。

十三、总结

一张图说清楚分区和分桶

                                  Table
                                    │
                    ┌───────────────┼───────────────┐
                    │               │               │
              Partition 1     Partition 2     Partition N
              (2025-03-14)   (2025-03-15)   (2025-03-16)
                    │               │               │
              ┌─────┼─────┐   ┌─────┼─────┐   ┌─────┼─────┐
              │     │     │   │     │     │   │     │     │
           T-1   T-2   T-M  T-1   T-2   T-M  T-1   T-2   T-M
           (在BE-1, (在BE-2,        ...
            BE-3,    BE-4,
            BE-5)    BE-6)

  分区 = "横向切"：把表按时间/范围切成多层
  分桶 = "纵向切"：把每一层按哈希切成多个并行的分片

核心设计原则

  1. 分区键选时间列，粒度控制在单分区 1GB~100GB
  2. 分桶键选高基数 + 高频查询列，理想情况是同时作为 JOIN 键
  3. 分桶数量 = MAX(数据量 ÷ 1GB, BE数 × CPU核数 ÷ 2)
  4. 排序键第一列尽量等于分区键，后面紧跟高频过滤列
  5. 主键表必须分区，利用冷热分离释放内存
  6. 考虑 Colocate Join，将频繁 JOIN 的表放入同一个 Colocation Group
  7. 优先使用表达式分区（v3.4+），用动态分区自动管理生命周期
  8. 监控 + 验证：建表后用 EXPLAIN 和 SHOW DATA 验证设计效果
