一、环境与建表

1.1 创建练习库

-- 建议单独建库，避免污染生产库；全文示例均在此库执行
CREATE DATABASE IF NOT EXISTS mv_lab;
USE mv_lab;

1.1.1 零基础先看：常见 SQL 关键字白话解释

这一节建议先读一遍，再往下执行建表 SQL。

  • CREATE TABLE：创建一张“实体表”，数据会真正存储在这张表里。
  • IF NOT EXISTS：如果表已经存在，就不报错，直接跳过创建。
  • NOT NULL：这一列不允许为空（必须有值）。
  • COMMENT '...'：给表/字段加说明，方便后续维护。
  • DUPLICATE KEY(col1, col2)：明细模型。同一个 key 可以出现多行，不会自动去重。
    • 适合订单明细、行为日志等“保留原始记录”的场景。
    • 例如 DUPLICATE KEY(dt, order_id) 不是主键约束，只是声明数据组织键。
  • PARTITION BY RANGE(dt)：按时间把大表切成多个“分区小块”。
    • 好处：查询和刷新可只扫描相关分区，速度快、资源省。
    • 例如查 2024-01-03 时，只读对应分区，不读全部历史数据。
  • PARTITION p20240101 VALUES [('2024-01-01'), ('2024-01-02'))：定义分区范围，通常是左闭右开。
  • DISTRIBUTED BY HASH(order_id)：把数据按 order_id 哈希打散到多个 Tablet（可理解为分片）。
    • 目标：并行计算更充分，避免单机热点。
    • 经验：常选高基数、查询常用过滤/关联列。
  • PROPERTIES ("replication_num" = "1")：副本数。
    • 学习环境用 1 省资源；生产通常 3 提高可用性。
  • DECIMAL(18, 4)：定点小数，总位数 18、小数位 4，适合金额字段避免浮点误差。
  • TINYINT：很小的整型，常用来表示状态码（如 0/1/2）。

1.2 订单事实表 TestMV_orders（分区表，核心基表）

-- ============================================================
-- 订单明细表：DUPLICATE KEY + RANGE 分区
-- 生产中最常见的异步 MV 基表形态
-- ============================================================
CREATE TABLE IF NOT EXISTS TestMV_orders (
    dt              DATE            NOT NULL COMMENT '订单日期，分区键',
    order_id        BIGINT          NOT NULL COMMENT '订单 ID',
    user_id         INT             NOT NULL COMMENT '用户 ID',
    product_type    VARCHAR(32)     NOT NULL COMMENT '商品品类：Electronics/Clothing/Food',
    sales_amount    DECIMAL(18, 4)  NOT NULL COMMENT '销售额',
    order_status    TINYINT         NOT NULL COMMENT '0=取消 1=完成 2=退款'
)
DUPLICATE KEY(dt, order_id)            -- 明细模型：允许重复 key，不会自动去重
COMMENT '订单事实表'                     -- 表注释，便于元数据管理
PARTITION BY RANGE(dt) (
    -- 分区命名可自定义，这里按日期命名，便于识别
    -- 每个分区是一个时间范围，查询可按分区裁剪
    PARTITION p20240101 VALUES [('2024-01-01'), ('2024-01-02')),
    PARTITION p20240102 VALUES [('2024-01-02'), ('2024-01-03')),
    PARTITION p20240103 VALUES [('2024-01-03'), ('2024-01-04')),
    PARTITION p20240104 VALUES [('2024-01-04'), ('2024-01-05')),
    PARTITION p20240105 VALUES [('2024-01-05'), ('2024-01-06'))
)
DISTRIBUTED BY HASH(order_id)           -- 按 order_id 打散分布，提升并行度
PROPERTIES ("replication_num" = "1");   -- 副本数：学习环境 1；生产建议 3

三、同步物化视图（Rollup）

同步 MV = 基表上的加速索引，导入时同步刷新，仅 单表 + Default Catalog。

3.1 语法模板

CREATE MATERIALIZED VIEW [IF NOT EXISTS] mv_name
[COMMENT "说明"]
AS
SELECT ... FROM single_table [WHERE ...] [GROUP BY ...] [ORDER BY ...];

3.2 列裁剪

CREATE MATERIALIZED VIEW TestMV_mv_k1_k2
AS SELECT k1, k2 FROM TestMV_duplicate_table;

3.3 排序键

CREATE MATERIALIZED VIEW TestMV_mv_k2_order
AS SELECT k2, k1 FROM TestMV_duplicate_table ORDER BY k2;

3.4 聚合预计算

CREATE MATERIALIZED VIEW TestMV_mv_k1_k2_sumk3
AS SELECT k1, k2, SUM(k3) AS sum_k3 FROM TestMV_duplicate_table GROUP BY k1, k2;

3.5 去重

CREATE MATERIALIZED VIEW TestMV_mv_deduplicate
AS SELECT k1, k2, k3, k4 FROM TestMV_duplicate_table GROUP BY k1, k2, k3, k4;

3.6 验证同步 MV 构建与改写

-- 查看 Rollup 构建进度，等待 State = FINISHED
SHOW ALTER MATERIALIZED VIEW;

-- hint 直接扫 MV
SELECT * FROM TestMV_mv_k1_k2_sumk3 [_SYNC_MV_];

-- 业务 SQL 自动改写（无需 hint）
EXPLAIN
SELECT k1, k2, SUM(k3) FROM TestMV_duplicate_table GROUP BY k1, k2;
-- 成功标志：计划中出现 TestMV_mv_k1_k2_sumk3

3.6.1 [_SYNC_MV_] 是什么？为什么要写在表名后

  • [_SYNC_MV_] 是 StarRocks 的 hint（提示），用于“直接读取同步物化视图（Rollup）”。
  • 同步 MV 本质是基表上的加速索引，不是普通独立表；因此直接 SELECT * FROM 同步MV名 可能报 Unknown table。
  • 正确写法是把 hint 放在同步 MV 名后：
SELECT * FROM TestMV_mv_k1_k2_sumk3 [_SYNC_MV_];
  • 这属于固定写法（针对“直接查同步 MV”这个场景）。
  • 如果你是查基表（不是查同步 MV 名），通常不需要写 [_SYNC_MV_]，优化器会自动改写。

3.6.2 hint 是什么意思（零基础版）

  • hint 可以理解为“你给优化器的执行建议”。
  • 不写 hint 时：优化器自动选执行计划。
  • 写 hint 时：你告诉优化器“优先按这个路径执行”。
  • 在本手册里，[_SYNC_MV_] 就是一个典型 hint：提示引擎按同步 MV 方式读取。

3.7 同步 MV 数据变更测试

-- ① 插入新数据到基表
INSERT INTO TestMV_duplicate_table VALUES (1, 10, 500, 5000);

-- ② 同步 MV 随导入自动更新，直接查聚合结果
SELECT k1, k2, SUM(k3) AS sum_k3
FROM TestMV_duplicate_table
WHERE k1 = 1 AND k2 = 10
GROUP BY k1, k2;
-- 期望 sum_k3 = 100+200+500 = 800

-- ③ 通过 hint 看 MV 物理数据
SELECT * FROM TestMV_mv_k1_k2_sumk3 [_SYNC_MV_] WHERE k1 = 1 AND k2 = 10;

DROP MATERIALIZED VIEW IF EXISTS TestMV_mv_k1_k2;
DROP MATERIALIZED VIEW IF EXISTS TestMV_mv_k2_order;

四、异步物化视图 CREATE

4.1 完整语法

CREATE MATERIALIZED VIEW [IF NOT EXISTS] [db.]mv_name
[COMMENT "说明"]
[DISTRIBUTED BY HASH(col) [BUCKETS n]]     -- 与 REFRESH 至少指定其一
[REFRESH [IMMEDIATE|DEFERRED] [ASYNC | SCHEDULE [START(...)] EVERY(...)|MANUAL]]
[PARTITION BY col | date_trunc(...) | str2date(...)]
[ORDER BY (sort_col)]                       -- MV 排序键，非 query 内 ORDER BY
[PROPERTIES ("key"="value", ...)]
AS SELECT ...;

4.2 【Lab 主 MV】分区 + 手动刷新（推荐先练这个）

-- 生产最常用模式：分区对齐 + MANUAL 精确控制 + 分批刷分区
CREATE MATERIALIZED VIEW TestMV_mv_order_daily_sales
COMMENT '按天按品类预聚合销售额（Lab 主练习 MV）'
PARTITION BY dt                              -- 与 TestMV_orders.dt 一一对齐
DISTRIBUTED BY HASH(product_type)
REFRESH MANUAL                               -- 手动刷新，Lab 可控
PROPERTIES (
    "replication_num" = "1",
    "partition_refresh_number" = "1"          -- 每次最多刷 1 个分区
)
AS
SELECT
    dt,
    product_type,
    SUM(sales_amount)  AS total_sales,       -- 预聚合：销售额
    COUNT(*)           AS order_cnt,         -- 预聚合：订单数
    SUM(CASE WHEN order_status = 1 THEN sales_amount ELSE 0 END) AS valid_sales
FROM TestMV_orders
GROUP BY dt, product_type;
-- 创建后立即首次刷新（MANUAL 模式需手动触发）

5.2.1 WITH SYNC MODE 是什么意思？（零基础）

以这条语句为例：

REFRESH MATERIALIZED VIEW TestMV_mv_order_daily_sales WITH SYNC MODE;

可以拆成两部分理解：

部分 含义
REFRESH MATERIALIZED VIEW TestMV_mv_order_daily_sales 手动刷新这个异步物化视图，把基表最新数据重新计算并写入 MV
WITH SYNC MODE 同步调用：当前 SQL 会一直等到刷新任务成功或失败后才返回结果

SYNC 与 ASYNC 对比

-- 写法 1：默认异步（多数客户端等价于 WITH ASYNC MODE）
REFRESH MATERIALIZED VIEW TestMV_mv_order_daily_sales;
-- 任务提交后很快返回，刷新在后台执行；是否完成需查 task_runs 或 SHOW MATERIALIZED VIEWS

-- 写法 2：同步等待
REFRESH MATERIALIZED VIEW TestMV_mv_order_daily_sales WITH SYNC MODE;
-- 阻塞当前会话，直到刷新结束才返回；适合刷完立刻做 SELECT 对账
模式 返回时机 典型用途
ASYNC（默认） 提交刷新任务后即返回 生产调度、大批量刷新、不阻塞连接
SYNC 刷新成功或失败后才返回 练习 Lab、脚本对账、刷完马上验证数据

如何确认 SYNC 刷新是否成功

-- 1) 语句无报错且正常结束，通常表示刷新成功

-- 2) 查看 MV 最近刷新状态
SHOW MATERIALIZED VIEWS WHERE NAME = 'TestMV_mv_order_daily_sales';
-- 关注：last_refresh_state（SUCCESS / FAILED / RUNNING）

-- 3) 查看刷新任务明细
SELECT TASK_NAME, CREATE_TIME, FINISH_TIME, STATE, ERROR_MESSAGE
FROM information_schema.task_runs
ORDER BY CREATE_TIME DESC
LIMIT 5;

常见误区

  • REFRESH ... WITH SYNC MODE 只适用于异步物化视图（建表时写了 REFRESH MANUAL / ASYNC / SCHEDULE 等）。
  • **同步物化视图（Rollup）**没有 REFRESH 语句，数据随基表导入自动更新，不需要也不能写 WITH SYNC MODE。
  • WITH SYNC MODE 不是「同步物化视图」的意思，而是「同步等待刷新任务完成」。

一句话记忆：前面是「刷新这个 MV」；WITH SYNC MODE 是「我等它刷完再继续」。

-- ============================================================
-- 通用：对比任意 MV 与基表聚合是否一致（替换 MV 名和 GROUP BY 列）

SELECT
    COALESCE(b.dt, m.dt) AS dt,
    COALESCE(b.product_type, m.product_type) AS product_type,
    b.base_total,
    m.mv_total,
    IFNULL(b.base_total, 0) - IFNULL(m.mv_total, 0) AS diff
FROM (
    SELECT dt, product_type, SUM(sales_amount) AS base_total
    FROM TestMV_orders GROUP BY dt, product_type
) b
FULL OUTER JOIN (
    SELECT dt, product_type, total_sales AS mv_total
    FROM TestMV_mv_order_daily_sales
) m ON b.dt = m.dt AND b.product_type = m.product_type
WHERE ABS(IFNULL(b.base_total, 0) - IFNULL(m.mv_total, 0)) > 0.001
   OR b.dt IS NULL OR m.dt IS NULL;
