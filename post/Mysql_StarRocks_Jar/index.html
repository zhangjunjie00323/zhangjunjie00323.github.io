StarRocks 连接驱动对比：MySQL JDBC vs StarRocks JDBC

2026-06-23 21:56 64 min read 人阅读

在同一台电脑、同一个客户端（DataGrip）、同一个 StarRocks 集群上，对比 MySQL JDBC 与 StarRocks JDBC 两种连接方式的差异——包括连接配置、执行耗时、结果拉取、IDE 体验，以及如何做公平对比。

实验环境：StarRocks 4.0.9-ee · 示例表 pms.worktime（约 496 万行）
核心结论（先读）：查询快慢主要由 StarRocks 服务端决定；换驱动对「分析型 SQL 执行」影响通常很小，对「大结果集拉回客户端」和「IDE 语法识别」影响更明显。

一、为什么 StarRocks 能用两种驱动？

StarRocks FE 对外提供 MySQL 协议兼容 的查询端口（默认 9030）。因此：

连接方式 本质
MySQL JDBC 把 StarRocks 当作「类 MySQL」数据库，走 MySQL wire protocol
StarRocks JDBC 官方驱动，同样连 9030，针对 StarRocks 类型/行为做了适配

两者发出去的 SQL 文本相同 时，服务端执行计划应一致；差异主要在 客户端 JDBC 层 和 IDE 集成。

┌─────────────┐     MySQL JDBC      ┌──────────────┐
│  DataGrip   │ ──────────────────► │              │
│  (本机)     │     StarRocks JDBC  │  StarRocks   │
└─────────────┘ ──────────────────► │  FE :9030    │
                                    └──────┬───────┘
                                           │ 执行计划 / 聚合 / Scan
                                           ▼
                                    ┌──────────────┐
                                    │  BE 集群      │
                                    └──────────────┘

二、两种驱动怎么配（DataGrip）

在 DataGrip 里建 两个数据源，除驱动外其余配置尽量相同，便于对比。

2.1 公共配置（两个数据源保持一致）

配置项 建议值 说明
Host FE 节点 IP 不要一个写 IP、一个写域名（除非刻意测试）
Port 9030 StarRocks 查询端口，不是 MySQL 3306
User / Password 同一账号 注意用户名不要有空格（root ≠ root ）
Database pms 或留空 一致即可
URL 模板 用变量 {host} 不要把 IP 写死在驱动 Class 里

2.2 数据源 A：MySQL 驱动

项 值
驱动 MySQL（8.x）
Driver Class com.mysql.cj.jdbc.Driver
URL 模板 jdbc:mysql://{host}:{port}/{database}
依赖 DataGrip 自带或 Maven mysql:mysql-connector-java

可选 URL 参数（减少元数据干扰）：

jdbc:mysql://{host}:9030/pms?useSSL=false&serverTimezone=Asia/Shanghai

2.3 数据源 B：StarRocks 官方驱动

项 值
驱动 自定义 / StarRocks
Driver Class com.starrocks.jdbc.Driver
URL 模板 jdbc:starrocks://{host}:{port}/{database}
Maven 坐标 com.starrocks:starrocks-connector-j:1.1.1（以 Maven Central 为准）

若列表里没有 StarRocks，在 Driver 设置里 + 添加，从 Maven 下载 jar。

2.4 连接成功标志

Test Connection 应显示类似：

DBMS: StarRocks 4.0.9-ee   （StarRocks 驱动，最理想）
DBMS: MySQL 8.x            （MySQL 驱动，也常见）

若显示 MySQL (无版本) 且伴随 JVM errno=1455，多半是 本机 DataGrip 内存/页面文件不足，与驱动无关——先把 -Xmx4096m 改为 -Xmx2048m 并重启 DataGrip。

三、对比什么？不对比什么？

3.1 适合对比的维度

维度 MySQL 驱动 StarRocks 驱动 预期
分析 SQL 执行耗时（服务端为主） 客户端计时 同左 几乎相同
EXPLAIN ANALYZE 耗时 与驱动无关 同左 完全相同
大结果集拉回（无 LIMIT / 导出行多） 受 fetchSize 影响 类型映射更贴合 StarRocks 驱动可能略好
元数据浏览（表结构、列类型） 按 MySQL 解析，偶有偏差 更贴近 StarRocks StarRocks 驱动更准
SQL 编辑器标红/补全 DUPLICATE KEY 等常误报 误报较少 体验差异，不影响已执行 SQL 的速度
StarRocks 特有语法 可能不识别 更好 属兼容性，非性能

3.2 不要用来对比的

  • IDE 语法检查是否标红 → 不是执行性能
  • 第一次冷启动连接 → 受 JVM 预热、DNS 影响，波动大
  • 不同 Host/Port/账号 → 不是驱动差异

四、公平对比实验设计（推荐流程）

4.1 实验原则

  1. 控制变量：同机、同 DataGrip 版本、同 FE、同 SQL、同账号、同时间段（低峰）
  2. 预热：每条 SQL 第 1 遍丢弃，第 2 遍记时（避免 Page Cache 干扰）
  3. 拆清两段耗时：
    • 服务端：EXPLAIN ANALYZE 的 TotalTime
    • 端到端：DataGrip 执行面板显示的时间
  4. 分场景：小结果集 / 中结果集 / 大结果集，驱动差异在不同场景表现不同

PS：execution 和 execution 分别是什么

阶段 你的值 含义
execution 29 ms SQL 在 StarRocks 服务端执行 的时间：解析、优化、Scan、聚合、排序等，直到结果开始/完成在服务端产出
fetching 326 ms 把结果从 StarRocks 拉回 DataGrip 的时间：网络传输 + JDBC 读结果集 + 客户端缓冲
合计 355 ms DataGrip 显示的端到端时间（你表格里的「第 2 次 (s)」≈ 0.547 s）

可以简单理解成：

DataGrip 总时间 ≈ 服务端算（execution）+ 结果传回客户端（fetching）

Q2 · 复杂分析（主实验，约 500 万行扫描，最终 LIMIT 5000）

用于对比「重查询 + 中等结果集」——最能代表日常看板 SQL。

WITH base AS (
    SELECT
        user_id, project_id, dept_id, time, time_class, item_id,
        normal_hour,
        COALESCE(adjust_hour, 0) AS adjust_hour,
        COALESCE(added_hour, 0) AS added_hour,
        normal_hour + COALESCE(adjust_hour, 0) + COALESCE(added_hour, 0) AS total_hour
    FROM worktime
),
daily AS (
    SELECT
        user_id, project_id, dept_id, time, time_class,
        COUNT(*) AS line_cnt,
        COUNT(DISTINCT item_id) AS item_cnt,
        SUM(total_hour) AS sum_total
    FROM base
    GROUP BY user_id, project_id, dept_id, time, time_class
),
monthly AS (
    SELECT
        user_id,
        DATE_TRUNC('month', time) AS month_key,
        COUNT(DISTINCT project_id) AS project_cnt,
        SUM(line_cnt) AS record_cnt,
        SUM(sum_total) AS month_total_hours,
        MAX(dept_id) AS dept_id
    FROM daily
    GROUP BY user_id, DATE_TRUNC('month', time)
),
ranked AS (
    SELECT
        *,
        ROW_NUMBER() OVER (PARTITION BY month_key ORDER BY month_total_hours DESC) AS hrs_rank,
        LAG(month_total_hours) OVER (PARTITION BY user_id ORDER BY month_key) AS prev_hrs
    FROM monthly
)
SELECT
    user_id, month_key, dept_id, project_cnt, record_cnt,
    ROUND(month_total_hours, 2) AS month_total_hours,
    hrs_rank,
    ROUND(month_total_hours - prev_hrs, 2) AS mom_delta
FROM ranked
WHERE hrs_rank <= 200
ORDER BY month_key DESC, month_total_hours DESC
LIMIT 5000;

六、如何读数：别把两件事混在一起

计时方式 量的是什么 驱动是否影响
DataGrip 右下角耗时 提交 SQL → 结果展示完毕（端到端） 会（尤其大结果集）
EXPLAIN ANALYZE StarRocks 集群内执行 不会
EXPLAIN（无 ANALYZE） 只看计划，几乎不跑全量数据 不会

写文章时的推荐表述：

在相同 SQL、相同集群下，MySQL JDBC 与 StarRocks JDBC 的 EXPLAIN ANALYZE 耗时一致，说明性能瓶颈在 StarRocks 服务端；客户端显示的差异主要来自 结果集传输与 JDBC 反序列化，在返回行数较少时可以忽略。

项目 推荐

7.4 选型建议（一句话）

  • 比执行速度：优先优化 SQL 和表设计；驱动选型影响次要
  • 比开发体验：StarRocks 驱动更省心
  • 比兼容成本：MySQL 驱动零改造成本最高

八、常见问题（写进文章 FAQ）

Q1：两种驱动查同一条 SQL，速度会一样吗？

大多数情况：一样或几乎一样。
只有「返回行数巨大」「频繁拉元数据」时，端到端时间可能有可见差别。

Q2：为什么 MySQL 驱动下 DUPLICATE KEY 标红？

DataGrip 用 MySQL 语法检查器解析 StarRocks DDL，标红不等于不能执行。这是 IDE 体验问题，不是 StarRocks 报错。

Q3：生产环境用哪个驱动？

  • 应用代码：StarRocks 官方 JDBC
  • DBA 桌面工具：StarRocks 驱动更佳；MySQL 驱动可继续用
