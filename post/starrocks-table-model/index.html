StarRocks 表模型深度解析：明细、聚合、更新与主键的选择之道

2026-04-17 23:10 27 min read 人阅读

从日志分析到实时更新：一文读懂 StarRocks 四种表模型的适用场景

一、数据模型的根基：排序键与数据组织

在深入四种表模型之前，必须先理解 StarRocks 最底层的设计——排序键（Sort Key）。

StarRocks 在建表时，会将数据按照指定的一列或多列排序后存储，这部分用于排序的列就称为排序键。排序键通常为查询时过滤条件频繁使用的列，用以加速查询 。

数据导入至 StarRocks 时，引擎会根据排序键对数据进行排序，并以此为基础构建前缀索引——一种稀疏索引，能在查询时快速定位数据所在的磁盘位置，在 Scan 阶段大面积减少无关数据扫描 。

排序键的设计原则：

  • 将经常作为查询条件的列指定为排序键
  • 当涉及多个列时，将区分度高、经常查询的列放在前面
  • 排序键不宜包含过多列（建议 3-4 列），过多的排序列会增加导入时的排序开销
  • 前缀索引长度限制为 36 字节，超过部分会被截断

自 3.0 版本起，主键表支持使用 ORDER BY 显式定义排序键；自 3.3 版本起，明细表、聚合表和更新表也支持此功能，使得排序键的定义更加灵活 。

二、明细模型（Duplicate Key Model）：原始数据的忠实记录者

2.1 模型概述

明细模型是 StarRocks 的默认建表模型。如果在建表时未指定任何 KEY 关键字，默认创建的就是明细表。在该模型下，即使导入两条完全相同的数据，StarRocks 也会原封不动地保存 。

明细表的数据不具有任何约束，相同的数据行可以重复存在。它适用于存储不需要约束和预聚合的原始数据 。

2.2 适用场景

  • 日志数据分析：如服务器访问日志、应用埋点日志等，数据只追加不更新
  • 交易流水记录：需要保留每一条明细，不可丢失或合并
  • 查询方式灵活的分析：不需要局限于预聚合的分析方式，希望随时访问最原始的数据
  • 不需要更新的历史数据：如历史订单快照、审计日志

2.3 建表示例

CREATE TABLE access_log (
    event_time DATETIME NOT NULL COMMENT "事件时间",
    event_type INT NOT NULL COMMENT "事件类型",
    user_id INT COMMENT "用户ID",
    device_code INT COMMENT "设备编码",
    channel INT COMMENT "渠道"
)
DUPLICATE KEY(event_time, event_type)
DISTRIBUTED BY HASH(user_id) BUCKETS 10
PROPERTIES ("replication_num" = "3");

建表说明：

  • DUPLICATE KEY 显式指定表类型为明细模型，括号中的列为排序键
  • 如果省略 DUPLICATE KEY，StarRocks 默认选择表的前三列作为排序键
  • 排序键必须定义在其他列之前
  • 必须使用 DISTRIBUTED BY HASH 指定分桶键

三、聚合模型（Aggregate Key Model）：预聚合的性能加速器

3.1 模型概述

聚合模型是 StarRocks 中用于数据统计和汇总的模型。建表时定义排序键（维度列）和指标列，并为指标列指定聚合函数（SUM、MIN、MAX、REPLACE 等）。当多条数据具有相同的排序键时，指标列会根据聚合函数进行合并，最终表中只保留聚合后的数据 。

3.2 适用场景

  • 报表统计：如按天、按地域统计 PV/UV、销售额等
  • 多维分析：不需要查询原始明细，只需要聚合结果的场景
  • 指标预计算：将高频聚合查询的结果提前物化，降低查询开销

3.3 聚合原理：读时合并

聚合模型在 StarRocks 中有三个聚合阶段 ：

  1. ETL 阶段：每个批次的导入数据内部，相同排序键的数据进行聚合
  2. Compaction 阶段：BE 后台定期将多个版本的文件合并为大版本
  3. 查询阶段：对于查询涉及的数据，所有版本的同一排序键再次聚合后返回

3.4 建表示例

-- 分析各城市用户访问不同网页的总次数
CREATE TABLE page_visits (
    site_id LARGEINT NOT NULL COMMENT "站点ID",
    date DATE NOT NULL COMMENT "日期",
    city_code VARCHAR(20) COMMENT "城市编码",
    pv BIGINT SUM DEFAULT "0" COMMENT "访问次数"
)
AGGREGATE KEY(site_id, date, city_code)
DISTRIBUTED BY HASH(site_id) BUCKETS 10
PROPERTIES ("replication_num" = "3");

建表说明：

  • AGGREGATE KEY 指定表类型为聚合模型，同时定义排序键
  • 排序键需要满足唯一性约束，必须包含全部维度列
  • pv BIGINT SUM 表示 pv 列是采用 SUM 聚合的指标列
  • 若未显式定义 AGGREGATE KEY，默认除指标列外的所有列均为排序键

四、更新模型（Unique Key Model）：读时合并的覆盖写

4.1 模型概述

更新模型允许用户对表中的数据进行更新操作。每条记录都有一个唯一键（Unique Key），当新导入的数据与表中已有数据的主键相同时，新数据会覆盖旧数据 。

更新模型可以看作聚合模型的一种特殊情况——相当于为所有指标列指定了 REPLACE 聚合函数，返回主键相同的一组数据中的最新数据 。

4.2 读时合并 vs 写时合并

更新模型采用读时合并（Merge on Read） 策略：导入时不会立即合并数据，而是在查询时动态合并多版本数据。这种方式写入性能较好，但查询时存在合并开销，谓词和索引也无法充分下推，可能影响查询性能 。

正是由于这一限制，更新模型正在逐渐被主键模型取代。主键模型采用写时合并（Delete+Insert），查询性能更优 。

4.3 建表示例

CREATE TABLE orders_unique (
    order_id BIGINT NOT NULL COMMENT "订单ID",
    order_state INT COMMENT "订单状态",
    total_price BIGINT COMMENT "订单金额",
    create_time DATE NOT NULL COMMENT "创建时间"
)
UNIQUE KEY(order_id)
DISTRIBUTED BY HASH(order_id) BUCKETS 8
PROPERTIES ("replication_num" = "3");

五、主键模型（Primary Key Model）：实时更新的终极方案

5.1 模型概述

主键模型是 StarRocks 从 1.19 版本开始引入的新一代表模型，专为实时和频繁更新场景设计。它具有唯一性非空约束，能够在支撑实时更新的同时保证高效查询性能 。

与更新模型相比，主键模型的核心差异在于更新原理：

对比维度 更新模型（Unique Key） 主键模型（Primary Key）
更新原理 Merge on Read（读时合并） Delete + Insert（写时替换）
查询性能 合并开销大，谓词无法下推 无合并开销，索引正常使用
适用场景 低频批量更新 高频实时更新（百万级 TPS）
部分列更新 不支持 支持
主键索引持久化 不支持 支持（enable_persistent_index）

5.2 核心特性

  • 主键唯一非空约束：主键列的值不可重复、不可为 NULL
  • 写时替换：新数据直接标记旧数据为删除，同时插入新数据，查询时无需合并
  • 部分列更新：可以只更新指定的几列，未指定的列保持原值
  • 持久化主键索引：通过 enable_persistent_index 将主键索引持久化到磁盘，避免内存膨胀

5.3 适用场景

  • MySQL CDC 实时同步：业务数据库变更秒级同步到 StarRocks
  • 高频 Upsert 场景：如实时订单表、用户状态表的持续更新
  • 实时数仓 ODS/DWD 层：需要同时支持高频写入和高效查询
  • 部分列更新：如只更新订单状态，不更新其他字段

5.4 建表示例

CREATE TABLE orders_pk (
    order_id BIGINT NOT NULL COMMENT "订单ID",
    dt DATE NOT NULL COMMENT "分区日期",
    user_id INT NOT NULL COMMENT "用户ID",
    amount DECIMAL(10,2) COMMENT "订单金额",
    status TINYINT COMMENT "订单状态"
)
PRIMARY KEY (order_id, dt)          -- 主键必须包含分区列和分桶列
PARTITION BY RANGE(dt) ()
DISTRIBUTED BY HASH(user_id) BUCKETS 10
ORDER BY (dt, order_id)             -- 排序键可与主键不同
PROPERTIES (
    "replication_num" = "3",
    "enable_persistent_index" = "true"
);

建表关键约束：

  • 主键列必须在其他列之前定义
  • 主键必须包含分区列和分桶列——这是主键表最重要的设计约束
  • 主键列的值不可更新，避免破坏数据一致性
  • 主键支持的数据类型：数值（整型、布尔）、日期和字符串

六、四种模型全景对比与选型指南

6.1 核心差异一览

对比维度 明细模型 聚合模型 更新模型 主键模型
英文名称 Duplicate Key Aggregate Key Unique Key Primary Key
数据唯一性 无约束，可重复 排序键唯一 唯一键唯一 主键唯一且非空
重复数据处理 全部保留 按聚合函数合并 新数据覆盖旧数据 新数据替换旧数据
更新能力 不支持 不支持 支持（低频） 支持（高频，推荐）
部分列更新 不支持 不支持 不支持 支持
典型场景 日志、流水 报表、汇总 维表更新（旧方案） 实时更新、CDC 同步
推荐程度 无更新需求首选 预聚合场景 逐渐淘汰 有更新需求首选

6.2 选型决策树

text

需要更新数据？
    ├── 否 → 需要预聚合？
    │       ├── 否 → 明细模型（Duplicate Key）
    │       └── 是 → 聚合模型（Aggregate Key）
    └── 是 → 高频更新或部分列更新？
            ├── 是 → 主键模型（Primary Key）★ 推荐
            └── 否 → 更新模型（Unique Key，但建议升级为主键模型）

选型建议总结 ：

  • 有明确主键且需要更新历史数据 → 主键模型
  • 保留明细数据且历史数据不更新 → 明细模型
  • 只需保留聚合数据 → 聚合模型

七、表模型与数据分布的协同设计

表模型的选择只是建模的第一步，分区和分桶的设计同样关键。

7.1 分区设计要点

  • 值变化不频繁的时间列常用于 WHERE 过滤，建议作为分区列
  • 有数据淘汰需求的场景，选择使用动态分区自动管理
  • 单个分区数据量建议不超过 100GB
  • 数据量超过 50GB 或 5000 万行的表建议创建分区

7.2 分桶设计要点

  • 分桶数：每个桶预估大小 500MB-1GB（StarRocks 压缩比约 7:1 至 10:1）
  • 分桶键选择：选择 WHERE 子句中频繁使用且区分度高的列
  • 避免数据倾斜：如果单列区分度低（如 city_id 只有几十个取值），考虑使用联合分桶
  • 主键表特别约束：分桶键必须是主键的子集

7.3 排序键与索引的配合

  • 经常作为查询条件的列建议选为排序列
  • 排序列建议 3-4 列，不超过 4 列
  • 低基数列（如性别、城市）不适合建 Bitmap 索引，StarRocks 会自动创建低基数字典加速查询
  • 高基数、低重复度的列适合创建 Bloom Filter 索引

八、总结

StarRocks 的四种表模型构成了数据建模的基石，理解它们的差异是高效使用 StarRocks 的前提：

  1. 明细模型：最纯粹的记录者，适合日志、流水等只追加不更新的场景
  2. 聚合模型：预聚合的加速器，适合报表统计和多维分析
  3. 更新模型：早期更新方案，采用读时合并，正在被主键模型取代
  4. 主键模型：实时更新的终极方案，采用写时替换，完美支持 CDC 同步和高频 Upsert

一句话口诀：

  • 不更新不求聚合 → 明细
  • 不更新只求汇总 → 聚合
  • 要更新要性能 → 主键

选择合适的表模型，配合合理的分区、分桶和排序键设计，才能让 StarRocks 的性能优势发挥到极致。
