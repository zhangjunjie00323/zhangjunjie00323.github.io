StarRocks 架构设计哲学：为什么它只需要两种节点就能支撑 PB 级数据分析？

2026-04-12 19:54 32 min read 人阅读

从存算一体到存算分离：StarRocks 架构演进背后的技术思考与最佳实践

一、架构总览：极简设计背后的哲学

StarRocks 的架构设计有一个让人印象深刻的特点：它不是有多复杂，而是有多简单。整个系统仅由两种核心组件组成——FE 和 BE，不依赖任何外部组件（如 ZooKeeper、HDFS），部署和维护成本极低。

1.1 极简 MPP 框架

StarRocks 采用 MPP（Massively Parallel Processing，大规模并行处理） 框架，这是一种“无共享”架构——集群中的每个节点都有自己独立的 CPU、内存和本地磁盘，节点之间不共享任何计算或存储资源。相比传统的完全共享（SMP）或共享存储架构，MPP 天然支持水平扩展，增加节点数量即可近线性提升系统的计算能力和存储容量。

在 StarRocks 的 MPP 框架中，查询会被拆解为多个物理算子（Scan、Aggregation、Join 等），分发到各个 BE 节点上并行执行。这种“分而治之”的模式，让一个原本需要单机处理数小时的大查询，被分解为上百个小任务并行执行，最终在秒级返回结果。

1.2 三大核心引擎的协同

StarRocks 的高性能不仅仅依赖于 MPP 框架，而是由三大核心引擎协同驱动：

架构层次 核心组件 主要职责
协调与优化层 FE（Frontend） 元数据管理、SQL 解析、CBO 查询优化、执行计划生成与调度
计算与存储层 BE（Backend）/ CN（Compute Node） 数据存储、SQL 执行、列式存储、向量化计算
存储抽象层 本地磁盘 / 对象存储 / HDFS 存算一体模式下使用 BE 本地磁盘；存算分离模式下使用远程对象存储

这三个层次紧密协作，形成了一条高效的查询处理流水线：FE 负责“思考”，BE/CN 负责“执行”，存储层负责“持久化”。

二、FE：元数据大脑与查询协调的艺术

FE 是 StarRocks 的“总指挥部”，一条 SQL 从进入集群到变成执行计划，全程由 FE 主导。

2.1 FE 的四大核心职责

职责领域 具体工作
元数据管理 管理所有 Catalog、数据库、表、分区、分桶、物化视图、用户权限等元数据信息，每个 FE 节点在内存中存储和维护元数据的完整副本。
客户端连接管理 兼容 MySQL 协议，接收和处理客户端 SQL 请求，支持标准 SQL 语法。
查询规划与优化 SQL 解析 → 语义分析 → RBO 规则重写 → CBO 成本优化 → 生成分布式执行计划。FE 采用 CBO，基于精准统计信息（表/列/分区分桶）生成最优执行计划，支持复杂谓词下推、Join 重排序、物化视图改写。
查询调度 将优化后的执行计划拆分为多个 Fragment，分发到 BE 节点执行，并汇总结果返回客户端。

2.2 FE 的三种角色与高可用机制

为实现高可用，FE 节点内部通过类 Raft 协议（基于 BDBJE） 实现 Leader、Follower、Observer 三种角色的分工协作：

角色 元数据写入 元数据读取 参与选举 主要用途
Leader FE 是（唯一入口） 是 是（本身是选出的） 元数据变更、写入协调
Follower FE 否 是 是 高可用容灾、参与选举
Observer FE 否 是 否 扩展查询并发

Leader 从 Follower 中自动选举产生，只有 Leader FE 提供元数据读写服务。Leader 更新元数据后，会通过 BDBJE 协议将变更日志同步给所有 Follower 和 Observer，必须有半数以上的 Follower 节点同步成功才算写入成功。Observer 主要用于扩展集群的查询并发能力，不参与选主，不会增加集群的选主压力。

生产环境建议至少部署 3 个 FE 节点（1 Leader + 2 Follower），FE 节点个数必须为奇数（3 个或 5 个），以保证在网络分区时能够成功选举出 Leader。

2.3 EditLog 与 Checkpoint：元数据的持久化机制

FE 将元数据主要存储在内存中，同时采用 EditLog + Checkpoint 机制保证持久化和多节点同步：

  • EditLog：每一次元数据变更操作（如 CREATE TABLE、ALTER TABLE）都被记录为一条 EditLog 日志条目，存储在 BDBJE 数据库中。Leader 负责写入，Follower 和 Observer 通过 BDBJE 协议同步。
  • Checkpoint：当 EditLog 数量达到阈值（默认 50,000 条）时，FE 会将当前内存中的完整元数据序列化为一个 Image 文件持久化到磁盘，历史的 EditLog 和旧 Image 可以被清理。

FE 重启时，先加载最新的 Image 文件恢复基础元数据，再从 BDBJE 中读取 Image 之后的所有 EditLog 逐条回放，将元数据更新到最新状态。这套机制既保证了元数据的持久可靠，又兼顾了启动恢复的效率。

三、BE：列式存储与向量化执行的“强壮四肢”

如果说 FE 负责“出主意”，那 BE 就是负责“出力气”的那一个。BE 是 StarRocks 的计算与存储引擎，基于列式存储、向量化执行与 MPP 架构构建。

3.1 数据存储：列式存储 + 多副本机制

BE 采用列式存储方式组织数据——数据按列独立压缩与编码，查询时只读取涉及的列，大幅减少 I/O 量。支持 Zstandard、LZ4、Snappy 等多种压缩算法，结合字典编码、位图编码等技术，能将数据体积压缩 3-8 倍。

数据在 BE 上以 Tablet（数据分片） 为最小物理存储单元进行管理。StarRocks 采用分区 + 分桶的两级数据划分机制：分区将表数据按某个维度（通常是时间）进行逻辑划分，便于数据管理和裁剪；分桶在每个分区内部按指定的分桶键进行哈希分桶，将数据进一步切分为多个 Tablet。每个 Tablet 默认以 3 副本的形式存储在不同的 BE 节点上，通过多副本机制保障数据可靠性。

3.2 索引体系：让列式存储“快上加快”

仅有列式存储还不够——在海量数据中快速定位目标行，需要高效的索引支持。StarRocks 在 BE 上构建了多层次的索引体系：

索引类型 原理 适用场景
前缀索引 每 1024 行构成一个逻辑数据块，索引记录每个数据块中排序键的最小值，查询时通过二分查找快速定位。 排序键上的等值和范围查询
Bloom Filter 概率性判断元素是否存在。 高基数列的等值查询（如 user_id）
Bitmap 索引 用位图表示列值分布。 低基数列的过滤（如 gender、status）
Zone Map 记录数据块中列的 min/max。 范围查询的快速跳过

前缀索引的长度限制为 36 字节，超过部分会被截断。因此应将经常作为查询条件、区分度高的列放在排序键的前面。

3.3 向量化执行引擎：让 CPU “火力全开”

向量化执行是 StarRocks 高性能的“动力心脏”。与传统的逐行处理不同，向量化引擎一次处理一批数据（默认 1024 行，称为一个“Chunk”），充分利用现代 CPU 的 SIMD 指令集（如 AVX2、AVX512）实现单指令多数据并行计算。

对比维度 行式处理 向量化处理
数据处理单元 单行数据 数据块（1024 行/块）
CPU 利用率 低（频繁分支预测失败） 高（顺序处理 + SIMD 指令优化）
内存访问模式 随机访问（高 Cache Miss） 顺序访问（高 Cache 命中率）
函数调用开销 每行触发函数调用 整批数据单次函数调用

这种设计使得 StarRocks 的计算层全面受益于 SIMD 指令集加速，性能较传统引擎提升 5-10 倍。

四、数据组织：分区、分桶与四种表模型

4.1 分区与分桶

分区和分桶是 StarRocks 数据组织的两大核心概念：

  • 分区（Partition） ：将表数据按某个维度（通常是时间）进行逻辑划分。优势在于可以按分区删除或归档历史数据，查询时可以裁剪分区，只扫描相关分区的数据。
  • 分桶（Bucket / Tablet） ：在每个分区内部，按指定的分桶键进行哈希分桶，将数据进一步切分为多个 Tablet。Tablet 是 StarRocks 中最小的物理存储单元，也是数据复制、负载均衡、故障恢复的基本粒度。

一张表的 Tablet 总数 = 分区数 × 分桶数 × 副本数。每个 BE 节点承载的 Tablet 数量建议在 10-20 个左右，单个 Tablet 的压缩后大小建议控制在 500MB-1GB。

4.2 四种表模型

StarRocks 支持四种表模型，适用于不同的业务场景：

模型 核心特点 更新能力 适用场景
明细模型（Duplicate Key） 保留所有明细数据，无唯一约束 不支持 日志存储、交易流水
聚合模型（Aggregate Key） 写入时自动预聚合，排序键唯一 不支持 报表统计、多维分析
更新模型（Unique Key） 读时合并，新数据覆盖旧数据 支持（低频） 维表更新（旧方案）
主键模型（Primary Key） 写时替换（Delete+Insert），支持部分列更新 支持（高频，推荐） 实时更新、CDC 同步

主键模型是 StarRocks 专为实时和频繁更新场景设计的表模型。它采用 Delete+Insert 策略，借助主键索引配合 DelVector 的方式，保证在查询时只需要读取具有相同主键值的数据中的最新数据。主键模型支持持久化主键索引，通过 enable_persistent_index = true 配置，可将主键索引持久化到磁盘，避免全内存索引带来的内存膨胀问题，支撑百亿级主键的高频更新场景。

五、存算一体 vs 存算分离：两种架构的选择

StarRocks 从 3.0 版本开始支持存算分离架构，将计算与存储解耦，使二者可以独立扩展。这是 StarRocks 架构演进中最重要的一次升级。

5.1 两种架构的对比

对比维度 存算一体架构 存算分离架构
数据存储位置 BE 本地磁盘 对象存储（S3/OSS）或 HDFS
计算节点 BE（计算 + 存储） CN（纯计算节点）
数据本地性 强，计算向数据移动 弱，数据需从远端读取，但本地缓存热数据
扩展方式 计算存储耦合扩展 计算存储独立扩展
存储成本 较高（本地 SSD） 低 70-80%（对象存储）
适用场景 追求极致查询性能 数据量大、追求成本优化

在存算分离架构中，数据持久化存储在更为可靠和廉价的远程对象存储（如 S3）或 HDFS 上，CN 本地磁盘只用于缓存热数据来加速查询。在本地缓存命中的情况下，查询性能与存算一体架构持平；冷查询时性能约为存算一体的 1/3。

降本增效的显著成果：与传统架构相比，StarRocks 存算分离架构查询性能提升 10 倍以上，存储成本降低 70-80%。通过将数据存储在低成本的对象存储中，借助云对象存储实现 11 个 9 的高可靠性，还能秒级增减计算节点。

5.2 多计算组（Multi-Warehouse）

存算分离架构的一个重要衍生能力是 Multi-Warehouse（多计算组）。它基于存算分离架构，实现了数据共享与计算隔离：多个 Warehouse 共享同一存储层（如 OSS），实现数据高效共用；计算资源通过硬隔离机制独立分配。

Multi-Warehouse 针对大型 StarRocks 集群的资源隔离难题提供了有效解决方案——导入任务过大影响查询性能、大型 SQL 操作阻塞其他业务查询等问题可以通过将不同业务分配到不同 Warehouse 来解决。在使用存算分离架构的客户中，约半数已启用 Multi-Warehouse，该功能与弹性伸缩已成为企业用户广泛组合使用的两大核心功能。

六、Lakehouse 架构：数据湖与数据仓库的统一

StarRocks 不仅是高性能分析型数据仓库，更是一个完整的 Lakehouse 引擎——它能够直接在数据湖格式上进行高性能查询，实现“One data, all analytics”。

6.1 统一 Catalog 与多源联邦

StarRocks 通过 External Catalog 机制，提供了开箱即用的数据湖查询功能，无需将数据导入即可直接查询 Hive、Iceberg、Hudi、Paimon 等数据湖格式中的数据。通过统一的 Catalog 对元数据进行统一管理，简化数据访问与权限控制，实现数据集中治理。

数据源类型 支持格式 核心能力
Hive ORC、Parquet、Avro 元数据同步、分区裁剪
Iceberg Parquet、Avro 快照查询、时间旅行、Schema 演化
Hudi Parquet Copy-on-Write / Merge-on-Read
Paimon Parquet、ORC 流批统一、主键表支持
JDBC MySQL、PostgreSQL 等 联邦查询

6.2 Iceberg 深度集成

StarRocks 4.0 版本对 Apache Iceberg 的支持进行了系统化的深度集成，带来了多项关键升级：

  • 原生写入支持：可直接通过 StarRocks 对 Iceberg 表执行 DELETE 操作
  • 隐藏分区读写：自动处理隐藏分区，元数据解析速度大幅提升
  • Compaction API：优化小文件管理
  • 性能大幅提升：StarRocks 查询 Iceberg 的性能达到 Trino 的 3-6 倍，查询 Delta Lake 的性能是 Databricks Photon 的 2 倍

在真实企业实践中，腾讯微信将数据写入 Iceberg 作为统一存储，借助 StarRocks 实现准实时分析，数据新鲜度大幅提升，查询性能提高 3-6 倍。携程旅行把数据统一存储在 Hive，利用 StarRocks 直接服务 BI 报表和交互式分析，并按需创建物化视图，使查询性能提升 10 倍以上。

6.3 Paimon 生态与流批一体

StarRocks 同样深度支持 Apache Paimon。Paimon 是一种流批统一的湖存储格式，前身为 Flink Table Store。StarRocks 与 Paimon 的结合，可以实现 Flink + Paimon + StarRocks 的端到端流批一体 Lakehouse 方案。

在 Lakehouse 典型场景中，StarRocks + Paimon 组合的 TPC-H 1T 性能测试，相比上一个版本性能提升 100%，相比 Trino + Paimon 实测性能提升高达 12 倍。

七、性能表现：基准测试与企业实践

StarRocks 的性能优势不仅体现在理论设计上，更在多个权威基准测试和企业实践中得到验证。

7.1 TPC-H 基准测试

在 TPC-H 10TB 基准测试中，阿里云 EMR Serverless StarRocks（Stella 内核）以 QphH 超 754 万分的性能结果斩获全球冠军，领先第二名 111%。相比上一版本性能提升超过 120%，充分展现了云原生架构的技术优势。

7.2 TPC-DS 基准测试

StarRocks 4.0 在 TPC-DS 1TB 基准测试上，JOIN、聚合、去重、溢出处理等核心算子经过深度调优，查询性能较上一年提升了约 60%。

7.3 竞品对比

相比 ClickHouse，StarRocks 在 TPC-H 多表 JOIN 查询中领先 3-5 倍。原因在于 ClickHouse 的查询规划器在 JOIN 排序和节点间数据移动方面存在不足。相比 Apache Doris，StarRocks 在 Doris 基础上重写了执行引擎，采用全向量化执行模型，并引入 CBO 优化器，显著提升复杂查询性能。

7.4 企业实践

  • Cisco Webex：从 Pinot 迁移至 StarRocks，统一了实时与历史数据分析架构。相比 Trino 平均查询延迟改善了约 50%。
  • 携程 UBT 系统：日均新增数据量约数十 TB，通过 StarRocks 存算分离架构支撑排障统计和监控场景。
  • Haezoom：迁移至 StarRocks 后实现 1.74 倍吞吐量提升、44% 更快查询，基础设施成本降低 30%。

八、版本演进与未来展望

8.1 StarRocks 版本演进路线

版本 核心突破 关键能力
1.x 极速查询性能 解决 BI 报表、数据探寻慢的问题
2.x 实时分析 可变数据引入实时分析

8.2 未来展望

StarRocks 将继续深耕 Lakehouse，进一步提升对 Iceberg、Paimon 等数据湖格式的支持，同时向 AI Native 时代迈进。2026 年 Roadmap 聚焦三大方向：

  • 云原生成熟：全面拥抱存算分离，强化弹性扩缩与多租户/多计算仓隔离能力
  • 统一流式湖仓：围绕 Paimon/Fluss 推进湖流一体能力
  • 多模态数据湖：与 Paimon/Lance 深度集成，面向湖上多模态分析与检索场景

九、总结

StarRocks 的架构设计哲学可以概括为“极简、极致、极智”：

  1. 极简架构：仅 FE 和 BE 两种组件，无外部依赖，运维成本极低。FE 采用 Leader/Follower/Observer 三种角色 + BDBJE 协议实现高可用，BE 采用 MPP + 列式存储 + 向量化引擎实现高性能计算。
  2. 极致性能：向量化引擎 + SIMD 指令优化 + CBO 智能优化 + 多层索引体系，查询性能较传统数据库提升 3-10 倍。在 TPC-H 和 TPC-DS 等权威基准测试中持续领先。
  3. 极智优化：CBO 优化器 + 智能物化视图 + 透明查询改写，自动选择最优执行计划。主键模型支撑百万级 TPS 的实时更新，完美适配 CDC 同步场景。
  4. 极广场景：从存算一体的极致性能到存算分离的成本优化，从内部表的实时数仓到数据湖的 Lakehouse 分析，StarRocks 一个引擎覆盖全部分析需求。
  5. 开放生态：深度集成 Iceberg、Hudi、Paimon 等主流数据湖格式，支持 MySQL 协议兼容，与 Flink CDC、Kafka 等生态工具无缝对接。

如果说 CBO 是 StarRocks 的“智慧大脑”，向量化引擎是“动力心脏”，MPP 架构是“骨骼”，那么 FE 与 BE 就是承载这一切的“大脑与四肢”——它们的分工协作，正是 StarRocks 能够实现极速全场景数据分析的根本原因。
