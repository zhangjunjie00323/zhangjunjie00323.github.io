图床完全指南：从入门到自建

2026-05-05 21:24 24 min read 人阅读

一份覆盖图床选型、部署、集成与避坑的完整手册

一、什么是图床，为什么你需要一个

图床（Image Hosting）本质上是一台专门托管图片的服务器。你在 Markdown 里写下 ![](https://img.example.com/photo.png) 时，那个 img.example.com 所指向的服务，就是图床。

和图床相对的概念是"本地引用"——图片和 Markdown 文件放在同一个文件夹里，用相对路径引用。这在小规模写作中完全够用，但一旦你开始跨平台发布、多设备同步、或者文章里图片超过十张，本地引用很快就会暴露出三个致命问题：

跨平台迁移的噩梦。你把文章从本地发到博客平台（如 Hexo、Hugo、WordPress），或者分享给同事，图片路径全部失效。你不得不逐张重新上传，或者手动替换所有路径。

版本管理的膨胀。Git 仓库里每张图片都是二进制文件，一次 git clone 可能拉下来几百 MB 的历史图片。Git LFS 能缓解，但不是根本方案。

协作的断裂。多人协作文档时，每个人的本地路径不同，图片引用变成乱码。即使你用云盘同步，路径的绝对性依然是个坑。

图床解决的就是这个核心问题：把图片变成互联网上的独立资源，通过 URL 访问，不依赖任何特定的文件系统路径。你的 Markdown 文件因此变得纯粹——只有文字和一个 URL，可以自由地在任何平台、任何设备之间流动。

二、图床方案全景对比

市面上的图床方案可以分成四大类，每一类有自己的适用场景。

2.1 公共免费图床

图床 特点 风险
SM.MS 开源、API 友好、支持 5MB 服务器在海外，国内访问慢；免费版每日 20 张限额
imgur 全球最大、API 完善 2023 年开始清理 NSFW 和长期未访问图片，大陆访问不稳定
路过图床 国内访问快、免注册 无 SLA 保障、随时可能关停
聚合图床 多接口聚合，支持微博/哔哩哔哩等 依赖于第三方服务稳定性

适用场景：偶尔贴几张图到论坛或临时分享，对稳定性和持久性没有要求。

不适用场景：博客配图、技术文档、任何需要"写完文章三年后图片还在"的情况。

一句话总结：免费图床可以用于测试，但不要用于正文。这不是偏见，是血泪教训——你永远不知道哪一天你的文章会变成满屏的裂图。

2.2 云厂商对象存储

这是目前最主流、最可靠的选择。

厂商 产品 免费额度 每 GB 存储 每 GB 流量 CDN
阿里云 OSS 无 ¥0.12/GB/月 ¥0.25-0.50/GB 自带 CDN（另付费）
腾讯云 COS 50GB 免费 6 个月 ¥0.099/GB/月 ¥0.21-0.50/GB CDN 另计
七牛云 Kodo 10GB 永久免费 超出 ¥0.099/GB/月 0.24-0.28/GB 自带融合 CDN
又拍云 USS 加入联盟每月 10GB 免费 ¥0.0043/GB/天 ¥0.33-0.50/GB 自带 CDN
AWS S3 5GB 免费 12 个月 $0.023/GB/月 $0.09/GB CloudFront 另计
Cloudflare R2 10GB 永久免费 $0.015/GB/月 零流量费 天然走 Cloudflare CDN

Cloudflare R2 是一个值得单独拿出来说的选项。它最大的优势是免流量费——对象存储最贵的部分不是存储，而是下行流量。如果你博客日访问量上千，S3 或 OSS 的流量账单会让你肉疼，R2 直接把这个成本抹掉了。唯一的代价是数据必须走 Cloudflare 网络，但考虑到大多数个人博客本来就在用 Cloudflare 做 DNS 和 CDN，这几乎不算代价。

适用场景：个人博客、技术文档站、所有需要长期可靠性的场景。

2.3 自建图床

适合有 NAS、云服务器或者对数据主权有执念的用户。

方案 架构 优势 劣势
Chevereto PHP + MySQL 功能完整（相册、API、多用户）、社区版免费 需要 VPS/NAS、维护成本高
Lsky Pro（兰空图床） PHP + MySQL/PostgreSQL 国产开源、界面美观、支持多种存储驱动（本地/S3/OSS/COS） 需要 PHP 环境
PicList + S3 兼容存储 Electron 桌面端 + S3 API 无需服务端、直接上传到对象存储 功能不如完整图床丰富
MinIO + Nginx Go 编写的对象存储 完全兼容 S3 API、轻量、可靠 需要自己处理 HTTPS 和外网访问

自建图床的一个新思路：用 Lsky Pro 作为管理后台，后端挂在 Cloudflare R2 上。这样做到了管理便利性和零流量成本的兼得，是目前个人自建的最优解之一。

2.4 图床即服务

一些写作和笔记工具本身内置了图床能力：

工具 图床方式 特点
Typora + PicGo/PicList 上传时自动替换为远程 URL 最流畅的 Markdown 写作体验，支持 SMMS/GitHub/OSS/S3 等
Obsidian + Image Uploader 插件 粘贴时自动上传 社区插件丰富，但稳定性依赖插件维护者
Notion 自动上传到 Notion 服务器 完全无感，但不能控制 URL，迁移困难
语雀 内置图床 同上，导出时图片变为本地文件
MWeb 内置图床 + 多种存储后端 macOS/iOS 生态优秀，跨平台弱

三、方案推荐：按场景选择

情景一：刚开始写博客，不想折腾

推荐：PicGo/PicList（桌面端）+ GitHub 仓库作为图床。

把图片推到 username/assets 仓库里，通过 raw.githubusercontent.com 或 JSDelivr CDN 加速访问。这方案的唯一缺点是 GitHub 仓库有大小限制（推荐保持在 1GB 以下），但对个人博客来说完全够用。国内访问需要借助 JSDelivr 或自备 CDN。

JSDelivr 加速后的图片链接格式：

https://cdn.jsdelivr.net/gh/用户名/仓库名@版本号/图片路径

情景二：有一定流量，追求专业

推荐：Cloudflare R2 + PicList 上传。

零流量费的诱惑太大。配合 PicList（PicGo 的增强 fork），配置好 S3 兼容 API 后粘贴图片自动上传、自动替换 URL，写文章时完全无感。

情景三：国内用户，流量较大

推荐：阿里云 OSS 或腾讯云 COS + CDN + PicGo。

国内对象存储在 CDN 上对大陆访问的优化远好于 Cloudflare。如果你读者主要在国内，多花一些流量费换来更快的加载速度是值得的。

情景四：对数据有完全控制欲

推荐：MinIO + Nginx 反代 + Lsky Pro 管理面板。

MinIO 部署在你的服务器或 NAS 上，Nginx 处理 HTTPS 和外部访问，Lsky Pro 提供上传和管理界面。这是最重但最自主的方案。

四、实战：搭建 Cloudflare R2 图床 + PicList 上传

这是目前个人用户性价比最高的方案，完整步骤不超过 15 分钟。

4.1 创建 R2 存储桶

  1. 登录 Cloudflare Dashboard，进入 R2 页面
  2. 点击「Create bucket」，命名如 my-images
  3. 创建完成后，进入桶设置 → 「Manage R2 API Tokens」
  4. 创建一个 API Token，权限选择「Object Read & Write」
  5. 记录下来三组信息：
    • Account ID（在 R2 页面右上角）
    • Access Key ID 和 Secret Access Key（创建 Token 后显示）
    • Endpoint：格式为 https://<account_id>.r2.cloudflarestorage.com
    • 桶名：你创建的桶名（如 my-images）

4.2 设置公开访问

R2 默认桶是私有的。在桶设置中，将桶的公开访问与一个自定义域名绑定：

  1. 在 Cloudflare 中添加一个 DNS 记录（如 img.yourdomain.com），指向任意 IP，开启橙色云朵代理
  2. 进入 R2 桶的「Settings」→「Public Access」→「Custom Domains」
  3. 绑定刚才的 img.yourdomain.com
  4. 此后桶内文件可以通过 https://img.yourdomain.com/photo.png 公开访问

4.3 配置 PicList

PicList 是 PicGo 的增强版（GitHub 搜索 Kuingsmile/PicList），对 S3 兼容协议支持更好：

  1. 下载 PicList 桌面客户端
  2. 图床设置 → 找到「Amazon S3」或「S3 兼容」插件
  3. 填入配置：
    • AccessKey ID：R2 的 Access Key ID
    • Secret AccessKey：R2 的 Secret Key
    • 自定义节点：https://<account_id>.r2.cloudflarestorage.com
    • 桶名：my-images
    • 自定义域名：https://img.yourdomain.com
    • 路径：{year}/{month}/{filename}{extname}（按年月归档）
  4. 测试上传，确认成功

4.4 与 Typora 集成

Typora → 偏好设置 → 图像 → 上传服务 → PicGo（或自定义命令）。

配置好后，你在 Typora 中粘贴截图或拖入图片时，Typora 会自动调用 PicList 上传到 R2，然后把本地的 ![](image.png) 替换为 ![](https://img.yourdomain.com/2026/05/screenshot.png)。

整个过程对写作完全无感，这是目前 Markdown 写作体验最顺滑的图床集成方案。

五、进阶技巧

5.1 图片压缩前置

在上传前自动压缩图片，既能节省存储和流量，也能大幅提升网页加载速度。PicList 内置了压缩插件，支持 TinyPNG API 和本地压缩。建议设置：

  • JPEG 质量 80%，PNG 转为 JPEG（非透明图）
  • 最大宽度 1920px（4K 屏幕也不模糊）
  • 超过 2MB 的图片强制压缩

5.2 CDN 与缓存策略

如果使用自己的域名挂在 Cloudflare 上，可以利用其缓存规则：

# Cloudflare Cache Rule
URL: img.yourdomain.com/*
Cache Level: Cache Everything
Edge TTL: 30 days
Browser TTL: 7 days

图片是静态资源，URL 不变内容就不变，缓存时间设长一点完全没问题，还能大幅降低回源次数。

5.3 多份备份策略

图床虽然可靠，但单点故障永远存在。建议叠加备份：

  1. R2 作为主存储（和 CDN 一体）
  2. GitHub 仓库作为备份：用 GitHub Actions 定时从 R2 同步图片到 assets 仓库
  3. 本地保留原始截图：PicList 设置中勾选「保留本地副本」

这样即使某一天 R2 出了极端情况（概率极低，但不是零），你还有 GitHub 和本地的完整备份。

5.4 防盗链

如果你的图床流量莫名其妙飙升，大概率是被第三方网站直接引用了图片链接。在 Cloudflare 中可以通过 Transform Rules 屏蔽非授权域名的 Referer：

http.referer contains "evil-site.com" → block

对于严格防盗，可以设置白名单：只允许自己的博客域名和本地开发环境访问。

六、避坑清单

在折腾图床这件事上，前人已经踩过足够多的坑。以下是经过验证的教训：

不要用微博/哔哩哔哩等社交平台做图床。虽然有人通过这些平台的图片上传接口间接实现免费图床，但这些接口不对第三方开放，随时可能被风控封禁。历史上有过多起大规模封号导致图片全部丢失的事件。

不要在对象存储上裸奔（无 CDN）。对象存储的默认外网下行流量费很高。即使你用 Cloudflare R2 免流量费，也建议套一层 CDN 来提升全球加载速度。

不要用太短的随机文件名。建议文件名格式：{year}/{month}/{timestamp}_{random6}.{ext}。这样既保证了唯一性，又方便按时间归档和排查。

不要忘记设置 CORS。如果你用图床的图片在前端做 Canvas 操作，需要在存储桶上配置 CORS 头，否则会报跨域错误。R2 中在桶设置的 CORS 部分添加允许的 Origin 即可。

不要忽略 HTTPS。所有现代浏览器对混合内容（HTTPS 页面引用 HTTP 图片）都有不同程度的拦截。确保你的图床域名全程 HTTPS，Cloudflare 默认提供免费 SSL 证书，不需要额外配置。

七、未来趋势

图床这个概念本身正在被更上层的方案所吸收：

AI 原生写作工具的图床集成。新一代 AI 写作工具（如 Notion AI、Cursor 等）在生成内容的同时，AI 生成的配图也自动托管在平台内置的图床上。用户不再需要手动管理图片存储——这可能是图床的终极形态：它变得如此透明，以至于你不再意识到它的存在。

边缘存储的普及。Cloudflare R2、Vercel Blob、Netlify Large Media 等产品正在把存储推到边缘节点，让图片离用户更近。未来的图床不会是一个中心化的服务器，而是一个分布在全球数百个节点的存储网络。

图片格式的进化。WebP 和 AVIF 正在取代 JPEG 和 PNG。下一代图床将自动在上传时转换为最优格式，并根据请求端的支持能力动态返回不同格式。类似 imgix 和 Cloudinary 这样的图像 CDN 服务就是把"存储 + 格式转换 + 尺寸裁剪 + CDN"做成了一站式方案，价格不菲但对于流量巨大的产品来说物有所值。
